# gRPC Cert / Domain Auto-Align — Design

> Status: Approved (all decision points decided by the implementer per
> explicit user goal "所有的决策点你来决定").
> Builds on: `2026-05-18-caddy-https-design.md` (Caddy HTTPS for the
> server install). Camp-A approach: keep direct `:7443` + bundle pin;
> Caddy never fronts gRPC.

## Problem

When an operator enables Caddy HTTPS with `--domain`, only the Web UI
gets the domain. The gRPC control plane (`:7443`) still advertises (and
its self-signed cert only covers) the machine hostname + loopback. Two
consequences:

1. The advertised endpoint baked into client bundles is not the domain,
   so the operator must understand and set `--advertised-endpoint`
   separately.
2. Even the existing default (public IP) is **not in the gRPC cert
   SAN** — a pre-existing latent gap: a remote client doing TLS
   verification against `<ip>:7443` would fail name validation
   (`crates/portunus-client/src/control.rs:71-77` sets the verification
   name to the advertised host; the auto cert SAN in
   `crates/portunus-server/src/tls.rs:111-121` is only
   `hostname`/`localhost`/`127.0.0.1`/`::1`).

Goal: the operator enters **one** value (the domain). Everything else —
advertised endpoint and a SAN-correct gRPC cert — is derived
automatically, with no new concepts the operator must learn, and without
changing the trust model.

## Core rule (server-side)

**The self-signed gRPC cert's SAN MUST cover the effective
advertised-endpoint host.** On startup:

- Compute `desired_host` = host part of the effective advertised
  endpoint (the explicit `--advertised-endpoint` value when present;
  otherwise unchanged from today — loopback/grpc-addr, already covered
  by the existing localhost/loopback SANs, so `desired_host = None`).
- If `desired_host` is `Some(h)`:
  - **No cert files present** → generate self-signed including `h` in
    SAN (`IpAddress` SAN if `h` parses as an `IpAddr`, else `DnsName`),
    in addition to the always-present `hostname` + `localhost` +
    `127.0.0.1` + `::1`. Write the autogen marker.
  - **Cert files present AND Portunus-autogenerated** (marker present):
    parse the leaf SAN; if it already covers `h` → use as-is; if not →
    back up the old pair, regenerate including `h`, log
    `tls.cert_regenerated` at WARN with re-issue guidance.
  - **Cert files present AND operator-supplied** (no marker): never
    touch. If the SAN does not cover `h`, log `tls.cert_san_mismatch`
    at WARN (informational) and continue — the operator owns this cert.

The rule is idempotent and self-healing: a server whose advertised host
changes (install → `domain` verb → upgrade) re-aligns on next start
without operator action. It also closes the pre-existing public-IP gap
(public-IP advertised now lands in the SAN as an `IpAddress` SAN).

Trust model is unchanged: still self-signed + bundle SHA-256 pin (Camp
A). gRPC stays on `:7443`; Caddy never fronts it.

### Autogen marker

`<data_dir>/.portunus-autocert` — a small file written **only** when the
server itself generates the pair. Content: the SAN host list it baked in
(one per line) for debuggability. Semantics:

- marker present ⇒ the cert pair is ours ⇒ eligible for SAN-driven
  regeneration.
- marker absent but `server.crt`/`server.key` present ⇒ operator-
  supplied ⇒ never regenerate (warn-only on SAN mismatch).

This cleanly separates "ours, safe to manage" from "operator's, hands
off" without sniffing issuer/CN heuristics.

### Cert regeneration & the pin

`leaf_fingerprint_hex` is recomputed from the (re)generated cert at load
time (`tls.rs` `load_or_generate`) and flows into `AppState`
(`state.rs:28-33`: `server_endpoint` / `server_cert_sha256` /
`server_cert_pem`), so **newly issued bundles automatically carry the
new endpoint, cert and pin**. Bundles already handed to clients are
stale by definition — this is the unavoidable, intentional re-pin when
the operator changes the advertised identity. Greenfield installs have
no clients yet (no breakage). For an existing install the WARN log +
installer note tell the operator to re-issue client bundles.

`fingerprint_paths()` (`tls.rs:175`) is unrelated to the daemon path
(used by a CLI helper) and is left untouched.

## Effective advertised endpoint (precedence)

Decided precedence (most → least): **explicit beats derived.**

1. Explicit `--advertised-endpoint <h:p>` (CLI) — wins outright.
2. `--domain <fqdn>` with no explicit advertised → advertised =
   `<fqdn>:7443`.
3. Neither → unchanged legacy behavior (runtime auto:
   loopback/grpc-addr).

`7443` is the project-wide control-plane port constant; the design uses
it as the derived port (matches `--advertised-endpoint host:7443`
guidance already in the installer help).

## Threading the value to the running server (necessary sub-part)

`--advertised-endpoint` is a clap arg with **no `env` binding**
(`main.rs:35-36`), and the installer currently writes it only as an
inert `Environment=PORTUNUS_ADVERTISED_ENDPOINT=` line that the server
ignores (confirmed during the Caddy smoke). For this feature to work the
value must reach the server **as a CLI argument**. Minimal, tightly
scoped change to the installer:

- **binary + systemd**: the drop-in `10-portunus.conf` switches from
  inert `Environment=` lines to a real `ExecStart=` override:

  ```
  [Service]
  ExecStart=
  ExecStart=/usr/local/bin/portunus-server --data-dir <DATA_DIR> serve \
      [--operator-http-listen <OP>] [--advertised-endpoint <ADV>]
  ```

  (`ExecStart=` cleared then re-set is the standard systemd override
  idiom.) This also finally makes `--data-dir` / `--operator-http-listen`
  effective for systemd — a net correctness gain, kept to exactly these
  three flags (no broader config rework).
- **docker**: append `--advertised-endpoint <ADV>` to the generated
  compose `command:` (the `--operator-http-listen 0.0.0.0:<port>`
  override added during the Caddy work stays).
- The inert `PORTUNUS_ADVERTISED_ENDPOINT` / `PORTUNUS_OPERATOR_HTTP_LISTEN`
  env lines are removed from the generated drop-in/.env (dead config;
  the CLI args are authoritative). `render_dropin` test seam updated
  accordingly.

Scope guard: this fixes only the three flags the installer already
collects. The broader "installer env vars are ignored by the server"
observation remains out of scope beyond these.

## Installer / wizard wiring

- Globals: reuse existing `ADVERTISED`, `DOMAIN`.
- After arg parse, for `ROLE=server`: if `DOMAIN` set and `ADVERTISED`
  empty → `ADVERTISED="${DOMAIN}:7443"` (precedence rule). Applies to
  `install` (both deploy forms) and the `domain` verb.
- Wizard (server branch) reorder: **ask domain first**.
  - domain entered (valid FQDN) → set `DOMAIN`, set
    `ADVERTISED="${DOMAIN}:7443"`, **skip the advertised prompt**.
  - domain blank → ask the advertised prompt exactly as today
    (detected-IP default, `-` = loopback).
- Summary: the existing `sum_advertised` line now shows
  `<domain>:7443`; the existing `sum_domain` line still shows the Web UI
  domain. No new i18n keys required for the core flow; one new key
  `adv_from_domain` (EN/ZH) annotates the advertised line when derived
  ("from domain") for transparency.
- `domain` verb on an existing install: also re-derives `ADVERTISED`
  (when not explicitly given) and rewrites the drop-in/compose so the
  server picks up the new endpoint + regenerates the SAN on restart;
  prints a note: "client bundles must be re-issued
  (`portunus-server enroll-client …`)".
- `uninstall` already removes the drop-in dir / compose; the autogen
  marker + cert live under data-dir and are removed by `--purge` only
  (unchanged).

## Error handling

- Domain validation reuses `valid_fqdn` (Caddy work). Bad FQDN → die
  with `bad_domain`.
- Cert SAN parse failure on a present cert is treated as
  "cannot verify" → for autogen certs, regenerate; for operator certs,
  WARN and continue (never block startup on a parse quirk).
- Key-perms enforcement (`enforce_key_perms`, ≤ 0600) is preserved on
  the regenerate path (re-written via existing `write_secret`, mode
  0600).
- All cert backups: `server.crt.portunus.<ts>.bak` /
  `server.key.portunus.<ts>.bak` next to the originals before any
  overwrite.

## Components & boundaries

- `crates/portunus-server/src/tls.rs` — owns cert lifecycle. New:
  `desired_san: Option<SanHost>` param threaded through
  `load_or_generate`; `generate_self_signed(extra: Option<SanHost>)`;
  `leaf_san_covers(cert_pem, host) -> bool`; marker read/write;
  IP-vs-DNS classification. One file, one responsibility (TLS material).
- `crates/portunus-server/src/serve.rs` — computes `desired_host` from
  `opts.advertised_endpoint` **before** `load_or_generate` (line 98),
  passes it in. ~5 lines + a host-extract helper (mirror of client
  `extract_host`, IPv6-bracket aware).
- `scripts/install.sh` — precedence assignment, wizard reorder,
  ExecStart-override drop-in, compose `command:` advertised flag, the
  `domain`-verb re-derive + note, one i18n key. Test seams:
  `--render-dropin` (now emits ExecStart form),
  `--effective-advertised` (prints the resolved advertised for given
  flags, network-free).

## Testing

**Rust unit (tls.rs), network-free:**
- generate with DNS extra SAN ⇒ leaf SAN contains the DnsName.
- generate with IPv4 extra SAN ⇒ leaf SAN contains the IpAddress.
- present autogen cert whose SAN already covers host ⇒ bytes unchanged
  (no regen), same fingerprint.
- present autogen cert missing host ⇒ regenerated, fingerprint changes,
  `.bak` written, marker updated.
- present cert, **no marker** (operator-supplied), missing host ⇒ NOT
  regenerated, returns Ok, fingerprint stable.
- key perms > 0600 still rejected; regenerate path writes 0600.
- `leaf_san_covers` true/false matrix (DNS exact, IP exact, absent).

**install.sh network-free (`scripts/install.test.sh`):**
- `server --domain d --dry-run` plan shows `advertised: d:7443`.
- `server --domain d --advertised-endpoint x:7443 --dry-run` ⇒
  advertised `x:7443` (explicit wins).
- `--render-dropin` with ADVERTISED+DATA_DIR+OP set ⇒ contains
  `ExecStart=` (clear) then an `ExecStart=…serve…--advertised-endpoint…`
  line; no `Environment=PORTUNUS_ADVERTISED_ENDPOINT=`.
- `--effective-advertised` seam: domain→`d:7443`; explicit→explicit;
  neither→empty.
- wizard (`--menu-stdin`): domain entered ⇒ advertised auto, advertised
  prompt skipped; domain blank ⇒ advertised prompt asked.
- EN/ZH i18n parity (existing coverage test) incl. `adv_from_domain`.
- shellcheck clean.

**VPS smoke (`serverbee-test.900040.xyz` → `207.241.173.217`), covers
every point:**
1. network-free harness on the VPS ⇒ PASS.
2. binary install `--domain serverbee-test.900040.xyz` (no explicit
   advertised): drop-in is the ExecStart-override form with
   `--advertised-endpoint serverbee-test.900040.xyz:7443`; start server;
   `server.crt` SAN contains the DnsName (`openssl x509 -text`);
   `leaf_fingerprint`/bundle endpoint = domain; Web UI HTTPS still 200
   via Caddy.
3. real client enrollment over `serverbee-test.900040.xyz:7443`:
   `portunus-server enroll-client`, drive the client binary, push a TCP
   rule, verify traffic — proves end-to-end TLS verifies against the
   domain SAN with the bundle pin (the crux).
4. explicit `--advertised-endpoint 207.241.173.217:7443` alongside
   `--domain` ⇒ advertised/SAN use the IP (IpAddress SAN), domain only
   for Web UI (precedence).
5. idempotent restart ⇒ no regen (fingerprint stable, no `.bak` churn).
6. `domain` verb re-point to a different domain
   (`*.900040.xyz` alt) ⇒ cert regenerated to new SAN, `.bak` written,
   WARN logged, installer prints re-issue note; old bundle now fails,
   freshly issued bundle works.
7. operator-supplied cert (drop a hand-made server.crt/key, remove
   marker) with a mismatching SAN ⇒ server starts, WARN
   `tls.cert_san_mismatch`, cert untouched.
8. docker install `--domain …` ⇒ compose `command:` carries
   `--advertised-endpoint <domain>:7443`; container regenerates SAN in
   its volume; client enrollment over the domain works; Web UI 200.
9. interactive wizard (PTY): domain entered ⇒ advertised prompt skipped,
   end state == step 2; second run blank domain ⇒ advertised prompt
   asked, no domain SAN.
10. regression: install with neither `--domain` nor
    `--advertised-endpoint` ⇒ behavior byte-identical to pre-feature
    (loopback advertised, no extra SAN, no marker-driven regen);
    `--dry-run` zero writes. Full host cleanup.

## Risks

- **Re-pin on advertised change** breaks already-issued bundles. By
  design; mitigated by greenfield-no-clients, WARN log, installer note,
  idempotent self-heal. Documented in output.
- **ExecStart override** changes the systemd unit's start command. Kept
  to the three already-collected flags; `uninstall` removes the drop-in;
  regression step 10 guarantees no-flag installs are unchanged.
- **Operator-supplied cert** must never be clobbered — the marker file
  is the explicit guard; step 7 verifies.
- **IPv6 advertised host** parsing — the host-extract helper is
  bracket-aware; covered by a tls.rs unit case if feasible (IPv4 is the
  smoke focus; IPv6 unit-only).
