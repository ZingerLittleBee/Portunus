`portunus-standalone` 是一个完全由 TOML 文件驱动的 TCP/UDP 转发器，**单机自给
自足、不依赖控制面**。它与 `portunus-client` 共享同一套数据面代码，但无需运行
`portunus-server` —— 适用于单机部署、边缘设备以及不需要控制面服务器的场景。

<Callout>
初次使用？[安装 → 安装 Standalone](/zh/docs/getting-started/installation#安装-standalone)
提供了快速上手以及"standalone 与 server + client 如何取舍"的决策指南。本页是完整
参考：配置 schema、CLI 标志、信号、示例、stats TUI 与运维注意事项。
</Callout>

## 安装

只有两种受支持的安装方式：容器宿主选 **Docker**，普通 Linux + systemd 选
**一键脚本**。

### Docker

每次发布都会向 GHCR 推送多架构镜像
`ghcr.io/zingerlittlebee/portunus-standalone:<version>`
（支持 `linux/amd64`、`linux/arm64`）。

```sh
# 1. 在工作目录放一份起步配置并编辑：
curl -fsSL -o portunus.toml \
  https://raw.githubusercontent.com/ZingerLittleBee/Portunus/main/crates/portunus-standalone/contrib/portunus.example.toml
chmod 0644 portunus.toml
# 用任意编辑器修改 portunus.toml（vi、nano 等）。
${EDITOR:-vi} portunus.toml

# 2. 启动容器：
docker run -d --network host \
  --name portunus-standalone \
  --restart unless-stopped \
  -v "$PWD/portunus.toml:/etc/portunus/standalone.toml:ro" \
  ghcr.io/zingerlittlebee/portunus-standalone:latest
```

注意事项：

- 生产环境固定到具体版本（`:1.5.0` 而非 `:latest`）。
- 容器以 **UID 65532**（`nonroot`）运行，宿主机文件必须对该 UID 可读 ——
  `chmod 0644 portunus.toml` 是最简单的做法。如果看到
  `error: io error: Permission denied (os error 13)` 就是这个原因。
- `--network host` 让端口段和任意端口规则不必逐端口映射。Docker Desktop
  或不支持 host 网络时改成 bridge 模式并显式列出端口，参考
  [`contrib/docker-compose.yml`](https://github.com/ZingerLittleBee/Portunus/blob/main/crates/portunus-standalone/contrib/docker-compose.yml)。
- 容器**内部**路径 `/etc/portunus/standalone.toml` 是镜像默认 `CMD` 写死的，
  宿主机路径任意。
- `docker compose` 和 Kubernetes manifests 见
  [`crates/portunus-standalone/contrib/`](https://github.com/ZingerLittleBee/Portunus/tree/main/crates/portunus-standalone/contrib)。

### 一键脚本（二进制 + systemd）

不用 Docker 的 Linux 服务器走这条。脚本会安装二进制、创建系统用户、写入
起步配置，以及一份加固版 systemd unit。

```sh
curl -fsSL https://raw.githubusercontent.com/ZingerLittleBee/Portunus/main/scripts/install.sh | bash -s -- standalone
```

它依次做这些事：

1. 检测 OS / 架构，下载最新 release 产物并校验 SHA-256。
2. 把二进制安装到 `/usr/local/bin/portunus-standalone`（mode `0755`）。
3. 若不存在，创建 `portunus` 系统用户/组。
4. 在 `/etc/portunus/standalone.toml` 写入起步配置（mode `0640`，
   属主 `root:portunus`），内容来自
   [`contrib/portunus.example.toml`](https://github.com/ZingerLittleBee/Portunus/blob/main/crates/portunus-standalone/contrib/portunus.example.toml)。
5. 在 `/etc/systemd/system/portunus-standalone.service` 写入加固版
   unit（`LimitNOFILE=65535`、`AmbientCapabilities=CAP_NET_BIND_SERVICE`、
   `ProtectSystem=strict`、`NoNewPrivileges=true`）。
6. 重新加载 systemd。**不会**自动启动 —— 初始配置是带示例上游的模板。

编辑配置后启动：

```sh
sudo ${EDITOR:-vi} /etc/portunus/standalone.toml
sudo systemctl enable --now portunus-standalone
sudo systemctl status portunus-standalone --no-pager
```

不带参数运行 `bash install.sh` 进入交互菜单，选 `3` 即 standalone。

## 配置 schema

```toml
# [global] 段 —— 所有字段可选
[global]
label              = "my-forwarder"   # 人类可读标签，仅作信息用途
log_level          = "info"           # tracing EnvFilter（默认 "info"）
log_format         = "json"           # "json"（默认）或 "pretty"
shutdown_drain_secs = 30              # SIGTERM 后优雅关闭的预算秒数（默认 30）

# [defaults] 段 —— 给未显式声明该字段的规则做兜底
[defaults]
protocol          = "tcp"    # 可选；规则可覆盖
udp_max_flows     = 1024     # 每规则 UDP flow 表容量（默认 1024）
udp_flow_idle_secs = 60      # idle flow 过期秒数（默认 60）
prefer_ipv6       = false    # DNS 地址族偏好（默认 false）

# [[rule]] —— 至少一条
[[rule]]
name          = "ssh-tunnel"          # 唯一名，写入日志
protocol      = "tcp"                 # "tcp" 或 "udp"
listen_port   = 2222                  # 单端口  }  二选一
# listen_ports = "8000-8009"          # 端口段   }
target        = "10.0.0.5:22"        # 单目标  }  二选一
# targets = [...]                     # 多目标  }
prefer_ipv6   = false                 # 单规则覆盖（可选）
udp_max_flows = 1024                  # 单规则覆盖（仅 UDP，可选）
udp_flow_idle_secs = 60              # 单规则覆盖（仅 UDP，可选）
```

### 端口段

用 `listen_ports = "lo-hi"` 把连续 listen 端口段一一对应映射到目标端口段：

```toml
[[rule]]
name         = "web-range"
protocol     = "tcp"
listen_ports = "8000-8009"
target       = "10.0.0.10:8000-8009"
```

listen 和 target 端口段宽度必须一致。

### 多目标 failover

用 `targets` 列表实现优先级 failover：

```toml
[[rule]]
name        = "ha-https"
protocol    = "tcp"
listen_port = 8443
targets = [
  { host = "primary.internal",   port = 443, priority = 0,  proxy_protocol = "v2" },
  { host = "secondary.internal", port = 443, priority = 10, proxy_protocol = "v2" },
]
```

`priority` 值越小优先级越高。转发器先尝试最高优先级的健康目标，连接失败时
按优先级回落。

### PROXY protocol

在任意目标（单目标或 `targets = [...]` 中的某项）上设
`proxy_protocol = "v1"` 或 `"v2"`，转发器会在上游连接前注入 PROXY protocol
头。后端需要拿到原始客户端地址时用。

## CLI 参数

| 参数 | 默认 | 说明 |
| --- | --- | --- |
| `--config <PATH>` |（自动搜索）| TOML 配置文件路径 |
| `--check` | false | 仅验证配置后退出（0 = 合法，2 = 非法）|
| `--log-level <LEVEL>` | 取自配置 | 覆盖日志级别 |
| `--log-format <FMT>` | 取自配置 | 覆盖日志格式（`json` 或 `pretty`）|

不传 `--config` 时，按顺序搜索 `$PORTUNUS_STANDALONE_CONFIG`、
`./portunus.toml`，都不存在时以 2 退出。

## 信号

| 信号 | 行为 |
| --- | --- |
| `SIGTERM` / `SIGINT` | 优雅关闭 —— 已建立连接最多 drain `shutdown_drain_secs` 秒，进程以 0 退出 |
| `SIGHUP` | 当前版本不处理（暂不支持 config reload）|

shipped 的 systemd unit 故意不写 `ExecReload`，因此
`systemctl reload portunus-standalone` 会报 "Job type reload is not
applicable" 而不是把服务杀掉。改 TOML 后请用
`systemctl restart portunus-standalone`。

## 示例配置

### 最简

二进制能接受的最小配置：一条规则，没有 `[global]` 或 `[defaults]` 块（所有
可选字段走默认）。

```toml
[[rule]]
name        = "ssh"
protocol    = "tcp"
listen_port = 2222
target      = "10.0.0.5:22"
```

### 完整

涵盖 schema 所有字段，且常见场景各一条规则（单端口 TCP、TCP 端口段、带
PROXY protocol 的多目标 failover、DNS 目标、单端口 UDP、UDP 端口段）。复制
后按需裁剪。

```toml
# ─────────── Global（所有字段可选） ───────────
[global]
label               = "edge-1"     # 写入日志的标识
log_level           = "info"       # EnvFilter；例如 "portunus=debug,info"
log_format          = "json"       # "json"（默认）或 "pretty"
shutdown_drain_secs = 30           # SIGTERM 后优雅关闭预算

# ─────────── Defaults（被未声明该字段的规则继承） ───────────
[defaults]
protocol           = "tcp"         # 规则未声明 protocol 时的默认值
udp_max_flows      = 1024          # 每规则 UDP flow 表容量
udp_flow_idle_secs = 60            # idle UDP flow 过期秒数
prefer_ipv6        = false         # DNS 地址族偏好

# ─────────── Rules ───────────

# 1) 单端口 TCP
[[rule]]
name        = "ssh-jump"
protocol    = "tcp"
listen_port = 2222
target      = "10.0.0.5:22"

# 2) TCP 端口段（listen 和 target 宽度必须一致）
[[rule]]
name         = "web-range"
protocol     = "tcp"
listen_ports = "8000-8009"
target       = "10.0.0.10:8000-8009"

# 3) 多目标 failover + PROXY protocol v2
[[rule]]
name        = "ha-https"
protocol    = "tcp"
listen_port = 8443
targets = [
  { host = "primary.internal",   port = 443, priority = 0,  proxy_protocol = "v2" },
  { host = "secondary.internal", port = 443, priority = 10, proxy_protocol = "v2" },
]

# 4) DNS 目标（每次 connect 时解析；prefer_ipv6 覆盖 defaults）
[[rule]]
name        = "api"
protocol    = "tcp"
listen_port = 9000
target      = "api.internal.example.com:443"
prefer_ipv6 = false

# 5) 单端口 UDP，提升 flow 容量和 idle 超时
[[rule]]
name               = "game-udp"
protocol           = "udp"
listen_port        = 27015
target             = "10.0.0.20:27015"
udp_max_flows      = 4096
udp_flow_idle_secs = 120

# 6) UDP 端口段
[[rule]]
name         = "voice-udp"
protocol     = "udp"
listen_ports = "30000-30019"
target       = "10.0.0.30:30000-30019"
```

启动前验证配置：

```sh
# 二进制
portunus-standalone --check --config /etc/portunus/standalone.toml

# Docker
docker run --rm \
  -v "$PWD/portunus.toml:/etc/portunus/standalone.toml:ro" \
  ghcr.io/zingerlittlebee/portunus-standalone:latest \
  --config /etc/portunus/standalone.toml --check
```

合法时打印 `ok` 并以 0 退出，非法时以 2 退出。

## 验证安装

确认规则已激活，并且流量真的能转发。

**systemd 安装：**

```sh
systemctl status portunus-standalone --no-pager
journalctl -u portunus-standalone -n 20 --no-pager
ss -ltn | grep 2222     # 检查 listen 端口是否已绑定
```

**Docker 安装：**

```sh
docker logs portunus-standalone | head -20
ss -ltn | grep 2222
```

启动时每条规则应输出一条事件：

```json
{"event":"standalone.rlimit_nofile","soft":65535,"hard":65535,…}
{"event":"rule.activated","rule_id":"…","listen_port":2222,…,"target":"10.0.0.5:22-22"}
```

端到端 TCP echo（把 7777/7778 换成你规则上的端口；先起 upstream 否则
connect 失败）：

```sh
sudo apt-get install -y socat
nohup socat TCP-LISTEN:7778,reuseaddr,fork EXEC:/bin/cat </dev/null >/tmp/up.log 2>&1 &
( echo "verify-payload"; sleep 1 ) | socat -t 3 - TCP:127.0.0.1:7777
#   → verify-payload
```

每条连接关闭时，转发器会输出 `rule.conn_closed`，包含非零的
`bytes_in` / `bytes_out`。

## 实时流量面板

`portunus-standalone stats` 是一个 TUI 看板，通过 Unix 域 socket
从运行中的 daemon 流式获取每条规则的实时计数器。无 HTTP、无
Prometheus、无额外进程——只是同一个二进制的子命令。

```sh
# 交互式 TUI：
portunus-standalone stats

# 一次性 JSON 快照（脚本场景）：
portunus-standalone stats --once | jq '.snapshot.r[] | {id, in, out, conns_active}'
```

Docker 安装：

```sh
docker exec -it portunus-standalone portunus-standalone stats
```

三个标签页：

- **Overview** —— 每规则一行：当前 in/out 速率、活跃连接、UDP flow 数。
- **Detail** —— 选中规则的 60s sparkline + 累计字节数 +（UDP）按
  `udp_max_flows` 显示的饱和 gauge。
- **Errors** —— 每条规则的非零失败累计计数（`port_in_use`、
  `upstream_connect_failed`、`icmp_evict`、`emsgsize`、`wouldblock`、
  `addflow_dropped`、`dns_failures`、`flows_dropped_overflow`）。

键位：`q` 退出，`?` 帮助，`Tab`/`h`/`l`/`←`/`→` 切 tab，`↑↓jk` 选行，
`p` 暂停，`s` 切排序，`r` 翻转排序方向，`/` 过滤，`c` 重置基线。

### 配置

```toml
[stats]
enabled     = true                                # 默认 true
socket_path = "/run/portunus/standalone.sock"     # Linux 默认
refresh_ms  = 1000                                # 250..=5000
```

daemon CLI 覆盖：`--no-stats`、`--stats-socket <PATH>`。

TUI 客户端按平台选默认 socket：

- Linux → `/run/portunus/standalone.sock`（shipped systemd unit 的
  `RuntimeDirectory=portunus` 会自动创建）
- macOS → `$TMPDIR/portunus-standalone.sock`
- 都不合适时用 `--socket <PATH>` 覆盖。

快照频率由 daemon 决定，client 没有请求通道。要改频率得编辑
`[stats] refresh_ms` 后重启 daemon。

## 可观测性

`portunus-standalone` 默认以 JSON 格式写 **stderr**（本地迭代时可切换到
`log_format = "pretty"`）。每条规则启动时写结构化事件，且有一个周期 reporter
每 30 秒输出一次按规则汇总的字节数与连接数。

不内嵌 HTTP / Prometheus / gRPC 任何接口 —— 那些是 `portunus-server` 的范畴。
用日志聚合器（`journald`、`docker logs`、sidecar log shipper 等）来收集 JSON 输出。

## 运维注意

- **不支持热重载。** `SIGHUP` 是 no-op，shipped 的 systemd unit 拒绝
  `systemctl reload`。改规则只能编辑 TOML 再
  `systemctl restart portunus-standalone`（或
  `docker restart portunus-standalone`）。已建立的 TCP 连接在退出时
  drain（不超过 `shutdown_drain_secs`）。
- **配置文件权限敏感。**
  - systemd：进程以 `User=portunus` 运行，并带
    `ReadOnlyPaths=/etc/portunus`。Installer 会设
    `chown root:portunus` + `chmod 0640`。严于此（如
    `chmod 0600 root:root`）会导致启动失败。
  - Docker：容器以 UID 65532 运行，宿主机文件必须对该 UID 可读
    —— `chmod 0644` 最省事。
- **绑定特权端口（`< 1024`）。** systemd unit 已带
  `AmbientCapabilities=CAP_NET_BIND_SERVICE`，所以 22/53/80/443 这些端口
  不用 root 也能绑。Docker 镜像声明了 `NET_BIND_SERVICE` capability；
  如果你额外约束了 capabilities，加 `--cap-add NET_BIND_SERVICE`。
- **文件描述符上限。** 启动时输出 `standalone.rlimit_nofile`。systemd
  unit 设 `LimitNOFILE=65535`；Docker 端可加
  `--ulimit nofile=65535:65535`。
- **TCP `splice(2)` 零拷贝**：未配置 PROXY-protocol 前缀时在 Linux 上
  自动启用。规则首次连接时会输出 `proxy.splice_selected` 事件。
- **端口段规则原子绑定。** `listen_ports = "8000-8009"` 启动时一次绑
  10 个端口；任何一个不可用就整条规则启动失败、其它已绑端口释放。
- **Ubuntu snap 版 Docker** 限制 bind-mount source 只能在 `/home`、
  `/media`、`/mnt` 等白名单路径下；从 `/root`、`/tmp`、`/etc` 挂载会
  在容器内表现为 `No such file or directory`。把 `portunus.toml` 放
  `/home` 下，或安装非 snap 的 `docker-ce` / `docker.io`。

## 与 portunus-client 的差异

| 功能 | `portunus-client` | `portunus-standalone` |
| --- | --- | --- |
| 控制面 | 到 `portunus-server` 的 gRPC 流 | 无 —— 仅 TOML |
| 动态规则更新 | 支持（server push） | 不支持 —— 需重启 |
| RBAC / 多用户 | 支持 | 不支持 |
| 限流 / QoS | 支持 | 不支持 |
| Prometheus 指标 | 支持（loopback HTTP） | 不支持 |
| 审计日志 | 通过 server | 结构化 stderr 日志 |
| SNI 路由 | 支持 | 不支持 |
| TCP splice 零拷贝（Linux） | 支持 | 支持（同一份数据面代码） |
| 上游 PROXY protocol | 支持 | 支持 |

## 升级与卸载

如果用 **一键脚本** 安装：

```sh
# 升级到最新 release（或固定 --version v1.5.0）：
bash install.sh upgrade standalone

# 状态（二进制版本、服务是否 active、部署形态）：
bash install.sh status standalone

# 服务控制透传：
bash install.sh service standalone restart

# 卸载（默认保留 /etc/portunus）：
bash install.sh uninstall standalone

# Purge：连 /etc/portunus 和 portunus 用户一起清掉：
bash install.sh uninstall standalone --purge
```

如果用 **Docker** 安装：

```sh
docker pull ghcr.io/zingerlittlebee/portunus-standalone:latest
docker stop portunus-standalone && docker rm portunus-standalone
# 然后用「安装」一节里的 docker run … 重新启动
```
