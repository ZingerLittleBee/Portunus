`portunus-standalone` 是一个完全由 TOML 文件驱动的独立 TCP/UDP 转发器。它与
`portunus-client` 共享同一套数据面代码，但无需运行 `portunus-server` ——
适用于单机部署、边缘设备以及不需要控制面服务器的场景。

## 快速上手（本地试跑）

适合本地开发或快速冒烟测试：

```sh
cargo build --release -p portunus-standalone
# →  target/release/portunus-standalone
```

创建一个最简配置 `portunus.toml`：

```toml
[[rule]]
name = "ssh"
protocol = "tcp"
listen_port = 2222
target = "10.0.0.5:22"
```

运行：

```sh
./target/release/portunus-standalone --config portunus.toml
```

仅验证配置，不启动服务：

```sh
./target/release/portunus-standalone --check --config portunus.toml
# 配置合法时打印 "ok" 并以 0 退出，配置有误时以 2 退出
```

需要部署到 Linux 服务器请直接看 [安装](#安装)。

## 安装

下面三种路径任选其一。**Installer** 是 Linux 服务器上的推荐方式 ——
帮你处理系统用户、目录、文件权限、systemd unit。

### 方式 A：Installer（二进制 + systemd）

```sh
curl -fsSL https://raw.githubusercontent.com/ZingerLittleBee/Portunus/main/scripts/install.sh | bash -s -- standalone
```

它依次做这些事：

1. 检测 OS / 架构，解析最新 release。
2. 下载 `portunus-<version>-<target>.tar.gz`，对照官方 checksums
   做 SHA-256 校验。
3. 把二进制安装到 `/usr/local/bin/portunus-standalone`（mode `0755`）。
4. 若不存在，创建 `portunus` 系统用户/组
   （`useradd --system --no-create-home --shell /usr/sbin/nologin`）。
5. 创建 `/etc/portunus/` 目录，并在
   `/etc/portunus/standalone.toml` 写入起步配置（mode `0640`，属主
   `root:portunus`），内容来自
   [`contrib/portunus.example.toml`](https://github.com/ZingerLittleBee/Portunus/blob/main/crates/portunus-standalone/contrib/portunus.example.toml)。
6. 在 `/etc/systemd/system/portunus-standalone.service` 写入加固版
   unit（来自
   [`contrib/portunus-standalone.service`](https://github.com/ZingerLittleBee/Portunus/blob/main/crates/portunus-standalone/contrib/portunus-standalone.service)），
   含 `LimitNOFILE=65535`、`AmbientCapabilities=CAP_NET_BIND_SERVICE`、
   `ProtectSystem=strict`、`NoNewPrivileges=true` 等。
7. 打印 next-step 提示。

Installer **不会**自动启动服务。编辑 `/etc/portunus/standalone.toml`
之后：

```sh
sudo systemctl enable --now portunus-standalone
```

不带参数运行 `install.sh` 进入交互菜单，选 `3` 即 standalone。

### 方式 B：手动安装（不用 installer）

适合你不想让脚本碰你系统，或安装环境无法访问 GitHub 的场景：

```sh
# 1. 获取二进制（下载 release 产物，或在工作站交叉编译后 scp 过来 ——
#    在小内存 VPS 上跑 cargo build 容易 OOM）：
sudo install -m 0755 ./portunus-standalone /usr/local/bin/portunus-standalone

# 2. 创建系统用户（幂等）：
sudo useradd --system --no-create-home --shell /usr/sbin/nologin portunus

# 3. 放置配置（先下载示例再改）：
sudo mkdir -p /etc/portunus
sudo curl -fsSL -o /etc/portunus/standalone.toml \
  https://raw.githubusercontent.com/ZingerLittleBee/Portunus/main/crates/portunus-standalone/contrib/portunus.example.toml
sudo chown root:portunus /etc/portunus/standalone.toml
sudo chmod 0640 /etc/portunus/standalone.toml
sudo "$EDITOR" /etc/portunus/standalone.toml

# 4. 安装加固版 systemd unit：
sudo curl -fsSL -o /etc/systemd/system/portunus-standalone.service \
  https://raw.githubusercontent.com/ZingerLittleBee/Portunus/main/crates/portunus-standalone/contrib/portunus-standalone.service
sudo systemctl daemon-reload

# 5. 启动：
sudo systemctl enable --now portunus-standalone
```

启动前可以先校验配置以便快速失败：

```sh
sudo -u portunus /usr/local/bin/portunus-standalone --check \
  --config /etc/portunus/standalone.toml
# 合法时打印 "ok" 并以 0 退出
```

### 方式 C：容器（Docker 或 Kubernetes）

见下方 [生产环境部署](#生产环境部署)，包括 `docker compose`、原生
`docker run`、Kubernetes manifest。

## 配置文件查找顺序

省略 `--config` 时，按以下顺序查找：

1. `$PORTUNUS_STANDALONE_CONFIG`（环境变量）
2. `./portunus.toml`（当前工作目录）

如果所有路径均不存在，打印详细错误并以 2 退出。

## 配置文件结构

```toml
# [global] 节 — 所有字段均可选
[global]
label              = "my-forwarder"   # 人类可读标签，仅供参考
log_level          = "info"           # tracing EnvFilter 指令（默认："info"）
log_format         = "json"           # "json"（默认）或 "pretty"
shutdown_drain_secs = 30              # SIGTERM 后的优雅关闭等待时间（默认：30 秒）

# [defaults] 节 — 规则未指定字段时的默认值
[defaults]
protocol          = "tcp"    # 可选；规则仍可单独覆盖
udp_max_flows     = 1024     # 每条规则的 UDP 流表上限（默认：1024）
udp_flow_idle_secs = 60      # 空闲流过期时间，单位秒（默认：60）
prefer_ipv6       = false    # 地址族偏好（默认：false，即优先 IPv4）

# [[rule]] — 至少需要一条
[[rule]]
name          = "ssh-tunnel"          # 唯一名称，用于日志输出
protocol      = "tcp"                 # "tcp" 或 "udp"
listen_port   = 2222                  # 单端口  }  二选一
# listen_ports = "8000-8009"          # 端口范围 }
target        = "10.0.0.5:22"        # 单目标   }  二选一
# targets = [...]                     # 多目标   }
prefer_ipv6   = false                 # 每条规则的覆盖项（可选）
udp_max_flows = 1024                  # UDP 规则的覆盖项（可选）
udp_flow_idle_secs = 60              # UDP 规则的覆盖项（可选）
```

### 端口范围

使用 `listen_ports` 配合 `"lo-hi"` 字符串，将连续监听端口窗口映射到目标端的同偏移窗口：

```toml
[[rule]]
name         = "web-range"
protocol     = "tcp"
listen_ports = "8000-8009"
target       = "10.0.0.10:8000-8009"
```

监听端口范围与目标端口范围的宽度必须相同。

### 多目标故障转移

使用 `targets` 列表形式实现按优先级排列的故障转移：

```toml
[[rule]]
name        = "ha-https"
protocol    = "tcp"
listen_port = 8443
targets = [
  { host = "primary.internal",   port = 443, priority = 0,  proxy_protocol = "v2" },
  { host = "secondary.internal", port = 443, priority = 10, proxy_protocol = "v2" },
]
```

`priority` 值越小优先级越高。转发器会优先尝试优先级最高的健康目标；连接失败时按优先级顺序回退。

### PROXY 协议

在任意目标（单目标或 `targets = [...]` 列表内）设置 `proxy_protocol = "v1"` 或
`"v2"`，可在上游连接建立前发送 PROXY 协议头，使后端获得原始客户端地址。

## CLI 参数

| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| `--config <PATH>` | （自动查找） | TOML 配置文件路径 |
| `--check` | false | 仅验证配置并退出（0 = 合法，2 = 有误） |
| `--log-level <LEVEL>` | 读取配置 | 覆盖日志级别 |
| `--log-format <FMT>` | 读取配置 | 覆盖日志格式（`json` 或 `pretty`） |

## 信号处理

| 信号 | 行为 |
| --- | --- |
| `SIGTERM` / `SIGINT` | 优雅关闭 — 活跃连接在 `shutdown_drain_secs` 内排空后进程以 0 退出。 |
| `SIGHUP` | 当前版本为空操作（配置热重载尚未支持）。 |

附带的 systemd unit 故意没设 `ExecReload`，因此
`systemctl reload portunus-standalone` 会报告 "Job type reload is not
applicable" 而不是把服务杀掉。改配置请编辑 TOML 然后
`systemctl restart portunus-standalone`。

## 安装后验证

`systemctl enable --now portunus-standalone` 之后，确认规则已激活、
数据真的在流：

```sh
# 1. 服务 active + 最近日志无错：
systemctl status portunus-standalone --no-pager
journalctl -u portunus-standalone -n 20 --no-pager
```

每条规则会有一条 `rule.activated` 事件：

```json
{"event":"standalone.rlimit_nofile","soft":65535,"hard":65535,…}
{"event":"rule.activated","rule_id":"…","listen_port":2222,…,"target":"10.0.0.5:22-22"}
```

```sh
# 2. 监听端口已绑定：
ss -ltn | grep 2222

# 3. 端到端 TCP echo（把 7777 / 7778 换成你规则里的端口；先把
#    上游起来，否则 connect 失败）：
sudo apt-get install -y socat                 # 或任意 echo server
nohup socat TCP-LISTEN:7778,reuseaddr,fork EXEC:/bin/cat </dev/null >/tmp/up.log 2>&1 &
( echo "verify-payload"; sleep 1 ) | socat -t 3 - TCP:127.0.0.1:7777
#   → verify-payload
```

每条关闭的连接会有一条 `rule.conn_closed` 事件，`bytes_in` /
`bytes_out` 应该非零：

```json
{"event":"rule.conn_closed","rule_id":"…","listen_port":7777,
 "peer":"127.0.0.1:51340","bytes_in":15,"bytes_out":15}
```

## 可观测性

`portunus-standalone` 默认将日志以 **JSON** 格式输出到 **stderr**
（开发时可设置 `log_format = "pretty"` 以获得可读输出）。每条规则在启动时
发出结构化事件，周期性报告器每隔 30 秒记录各规则的字节数和连接数统计。

本二进制不内嵌 HTTP 服务、Prometheus 端点或 gRPC 流——这些能力属于
`portunus-server`。请使用日志聚合工具（如 `journald`、sidecar 日志采集器）
收集 JSON 日志。

## 配置示例

### 最简

二进制能接受的最小配置：一条 rule，省略 `[global]` 与 `[defaults]`
（所有可选字段使用默认值）。

```toml
# /etc/portunus/standalone.toml
[[rule]]
name        = "ssh"
protocol    = "tcp"
listen_port = 2222
target      = "10.0.0.5:22"
```

校验并启动：

```bash
portunus-standalone --check --config /etc/portunus/standalone.toml
systemctl restart portunus-standalone
```

### 完整

涵盖 schema 中的全部字段，并按常见场景各给一条 rule（单端口 TCP、
TCP 端口段、多目标故障转移 + PROXY 协议、域名 target、单端口 UDP、
UDP 端口段）。按需复制裁剪。

```toml
# /etc/portunus/standalone.toml
#
# 修改后先校验再重启：
#   portunus-standalone --check --config /etc/portunus/standalone.toml
#   systemctl restart portunus-standalone
# （不支持 SIGHUP / `systemctl reload`，见「运维注意事项」。）

# ─────────── 全局（全部可选） ───────────
[global]
label               = "edge-1"     # 仅用于日志标识
log_level           = "info"       # EnvFilter；如 "portunus=debug,info"
log_format          = "json"       # "json"（默认）或 "pretty"
shutdown_drain_secs = 30           # 收到 SIGTERM 后的优雅退出时长

# ─────────── 默认值（rule 未设置时继承） ───────────
[defaults]
protocol           = "tcp"         # rule 未指定协议时使用的默认值
udp_max_flows      = 1024          # 每条 rule 的 UDP flow 上限
udp_flow_idle_secs = 60            # UDP flow 空闲多久回收（秒）
prefer_ipv6        = false         # DNS 解析时的地址族偏好

# ─────────── 规则 ───────────

# 1) 单端口 TCP
[[rule]]
name        = "ssh-jump"
protocol    = "tcp"
listen_port = 2222
target      = "10.0.0.5:22"

# 2) TCP 端口段（listen 与 target 的段宽度必须一致）
[[rule]]
name         = "web-range"
protocol     = "tcp"
listen_ports = "8000-8009"
target       = "10.0.0.10:8000-8009"

# 3) 多目标故障转移 + PROXY 协议 v2
#    priority 数字小的优先；相同优先级轮询。
[[rule]]
name        = "ha-https"
protocol    = "tcp"
listen_port = 8443
targets = [
  { host = "primary.internal",   port = 443, priority = 0,  proxy_protocol = "v2" },
  { host = "secondary.internal", port = 443, priority = 10, proxy_protocol = "v2" },
]

# 4) 域名 target（每次 connect 时解析；prefer_ipv6 覆盖 defaults）
[[rule]]
name        = "api"
protocol    = "tcp"
listen_port = 9000
target      = "api.internal.example.com:443"
prefer_ipv6 = false

# 5) 单端口 UDP，提高 flow 上限和空闲超时
[[rule]]
name               = "game-udp"
protocol           = "udp"
listen_port        = 27015
target             = "10.0.0.20:27015"
udp_max_flows      = 4096
udp_flow_idle_secs = 120

# 6) UDP 端口段
[[rule]]
name         = "voice-udp"
protocol     = "udp"
listen_ports = "30000-30019"
target       = "10.0.0.30:30000-30019"
```

## 运维注意事项

部署到生产前需要知道的事：

- **不支持热加载配置。** `SIGHUP` 是空操作，附带的 systemd unit 也
  拒绝 `systemctl reload`。改规则需编辑 TOML 然后
  `systemctl restart portunus-standalone`。已建立的 TCP 连接会在
  `shutdown_drain_secs` 内排空。
- **配置文件权限有要求。** 通过 systemd unit 运行时进程以
  `User=portunus` 起，加 `ReadOnlyPaths=/etc/portunus`。配置文件
  必须该用户可读 —— installer 会设 `chown root:portunus`
  + `chmod 0640`。再严（如 `chmod 0600 root:root`）会导致启动失败。
- **特权监听端口（`< 1024`）。** unit 通过
  `AmbientCapabilities=CAP_NET_BIND_SERVICE` 授权，所以监听
  22、53、80、443 等无需 root。脱离 systemd 运行时需要：以 root 启
  动、或 `setcap cap_net_bind_service+ep /usr/local/bin/portunus-standalone`、
  或只用 `≥ 1024` 的端口。
- **文件描述符上限。** 启动时会发 `standalone.rlimit_nofile` 日志。
  附带的 unit 设了 `LimitNOFILE=65535`；如果看到
  `… "soft":1024 …` 请提升（每条并发流占用每侧一个 fd）。
- **TCP `splice(2)` 零拷贝**：Linux 上未配 PROXY-protocol 前置时自动
  选用。该规则首次连接的日志里会出现 `proxy.splice_selected` 事件。
- **端口范围规则原子绑定。** 例如 `listen_ports = "8000-8009"` 会一次
  绑全部 10 个端口；任一不可用则该规则启动失败、其余端口立即释放。
  部署前请先排查端口冲突。
- **别在小机型 VPS 上编译。** 1 vCPU / 1 GB 的 VPS 跑 `cargo build`
  几乎必 OOM。在工作站交叉编译
  （`cargo zigbuild --release --target x86_64-unknown-linux-gnu -p portunus-standalone`）
  然后 `scp` 二进制过去。
- **日志为 stderr 上的 JSON。** 通过 `journald`（systemd 自动收集）
  或 sidecar 日志采集器聚合。本地迭代时切到
  `log_format = "pretty"` 更易读。

## 与 portunus-client 的差异

| 功能 | `portunus-client` | `portunus-standalone` |
| --- | --- | --- |
| 控制面 | 连接 `portunus-server` 的 gRPC 流 | 无——仅 TOML |
| 动态规则更新 | 是（服务端推送） | 否——需重启 |
| RBAC / 多用户 | 是 | 否 |
| 限速 / QoS | 是 | 否 |
| Prometheus 指标 | 是（本地 HTTP） | 否 |
| 审计日志 | 经由服务端 | 结构化日志输出到 stderr |
| SNI 路由 | 是 | 否 |
| TCP splice 快速路径（Linux） | 是 | 是（共享同一数据面代码） |
| PROXY 协议输出 | 是 | 是 |

## 生产环境部署

四种部署方式的参考模板均位于
[`crates/portunus-standalone/contrib/`](https://github.com/ZingerLittleBee/Portunus/tree/main/crates/portunus-standalone/contrib)。

### Docker / docker-compose

参考
[`Dockerfile`](https://github.com/ZingerLittleBee/Portunus/blob/main/crates/portunus-standalone/contrib/Dockerfile)
是一个两阶段构建，运行时镜像基于 distroless。
[`docker-compose.yml`](https://github.com/ZingerLittleBee/Portunus/blob/main/crates/portunus-standalone/contrib/docker-compose.yml)
默认使用 host 网络，端口范围转发因此无需逐端口 mapping：

```sh
cd crates/portunus-standalone/contrib
cp portunus.example.toml portunus.toml   # 启动前编辑
docker compose up -d
docker compose logs -f
```

目前未发布 GHCR 镜像 —— 请本地构建或推送到自有 registry。

**注意事项**：

- 运行时镜像为 `gcr.io/distroless/cc-debian12:nonroot`（UID 65532）。
  bind-mount 进容器的配置文件需要该 UID 可读
  （宿主上 `chmod 0644`，或显式映射 UID）。
- 直接用 `docker run` 调用（而非 compose）时，传 `--check` 会覆盖镜像
  默认 `CMD`，从而丢失默认的 `--config` 参数。两个都要显式传：
  `docker run … portunus-standalone --config /etc/portunus/standalone.toml --check`。
- **Ubuntu 上的 snap Docker** 把 bind-mount 来源限制在 `/home`、
  `/media`、`/mnt` 等白名单路径下。来自 `/root`、`/tmp`、`/etc` 的
  bind-mount 在容器内会表现为 `No such file or directory`。请把配置
  文件放到 `/home` 下，或改用非 snap 的 `docker-ce` / `docker.io` 包。

### Kubernetes

[`contrib/k8s/`](https://github.com/ZingerLittleBee/Portunus/tree/main/crates/portunus-standalone/contrib/k8s)
中的 manifest 运行一个单副本 `Deployment`，启用 `hostNetwork: true`
（任意端口转发的硬性要求）：

```sh
kubectl apply -f crates/portunus-standalone/contrib/k8s/configmap.yaml
kubectl apply -f crates/portunus-standalone/contrib/k8s/deployment.yaml
```

Deployment 与 DaemonSet 的取舍以及 PodSecurityPolicy 注意事项见
[`contrib/k8s/README.md`](https://github.com/ZingerLittleBee/Portunus/blob/main/crates/portunus-standalone/contrib/k8s/README.md)。

### 加固版 systemd unit

[`contrib/portunus-standalone.service`](https://github.com/ZingerLittleBee/Portunus/blob/main/crates/portunus-standalone/contrib/portunus-standalone.service)
设置了 `LimitNOFILE=65535`、以 `User=portunus` 运行、新增
`AmbientCapabilities=CAP_NET_BIND_SERVICE` 以便绑定特权端口，并启用
`ProtectSystem=strict` / `PrivateTmp=true` / `NoNewPrivileges=true`。
二进制不支持 SIGHUP 热加载 —— 修改 TOML 后请执行
`systemctl restart portunus-standalone`。

## 升级与卸载

如果你通过 [方式 A](#方式-ainstaller二进制--systemd) 安装：

```sh
# 升级到最新 release（或 --version v1.5.0 指定版本）：
bash install.sh upgrade standalone

# 查看状态（二进制版本、服务是否 active、部署形式）：
bash install.sh status standalone

# 服务控制透传：
bash install.sh service standalone restart

# 卸载（清理二进制 + unit；默认保留 /etc/portunus）：
bash install.sh uninstall standalone

# 彻底清理：连 /etc/portunus 和 portunus 用户一起删：
bash install.sh uninstall standalone --purge
```

如果你通过 [方式 B](#方式-b手动安装不用-installer) 手动安装：

```sh
sudo systemctl disable --now portunus-standalone
sudo rm /etc/systemd/system/portunus-standalone.service
sudo rm /usr/local/bin/portunus-standalone
sudo systemctl daemon-reload
# 可选 —— 同时清理配置和用户：
sudo rm -rf /etc/portunus
sudo userdel portunus
```
