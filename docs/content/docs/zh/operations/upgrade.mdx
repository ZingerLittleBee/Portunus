线协议改动 **只增不删** —— 新字段按 proto3 默认值剥离，跨版本组合会优雅
降级。项目历史上唯一的破坏性步骤是 v0.7 → v0.8（JSON → SQLite 切换，见下文）；
而 v0.7 从未进过生产。

v1.0 起，任何升级都无须手动迁移 SQLite schema —— server 启动时会自动向前
跑内嵌迁移。

## 兼容矩阵

| 组合 | 行为 |
| --- | --- |
| 新 server，旧 client | 新字段在发送时被剥离；server 用 capability 守门拒绝 client 无法处理的下发。 |
| 旧 server，新 client | 新 client 透明省略新字段（proto3 默认值剥离）。表现与旧版 client 完全一致。 |
| 双方同版本 | 完整功能面可用。 |

## Capability 守门

server 会**预先**拒绝向不理解某项功能的对端 client 推送。规则不会在任何地方
激活 —— 操作员收到 HTTP 422 加一个具体的码：

| 码 | 触发条件 |
| --- | --- |
| `unsupported_protocol` | UDP 规则推给 v0.3 或更早的 client |
| `multi_target_unsupported_by_client` | `targets[]` 推给 v0.6 或更早的 client |
| `sni_unsupported_by_client` | `sni_pattern` 推给 v0.8 或更早的 client |
| `rate_limit_unsupported_by_client` | `rate_limit` 推给 v0.10 或更早的 client |

## 推荐升级顺序

server 与 client 混版本时，**先升 server**。server 是守门人，新 server 配旧
client 是安全的；旧 server 配新 client 则看不到新功能，但其它行为正常。

## v0.7 → v0.8（重新开始）

这是项目历史上**唯一一次**算得上 "破坏性" 的升级 —— 而且也只是因为 v0.7
从未真正进过生产。

v0.8 把各个 JSON 文件（`tokens.json`、`identity.json`、`rules.json`）以及
内存审计 ring 全部并入一个 SQLite 数据库 `<data-dir>/state.db`。

写过 JSON 文件的 v0.7 部署**没有迁移路径**。约定是改用一个全新的
`--data-dir`：

```sh
sudo systemctl stop portunus-server
sudo rm /var/lib/portunus/tokens.json /var/lib/portunus/identity.json
portunus-server reset --confirm
sudo systemctl start portunus-server
```

server 启动后，用日志中打印的 setup token 完成首次 Web onboarding。已下发的
client bundle（TLS 指纹固定）继续有效 —— 只需重建操作员侧的身份和规则定义。

## v0.4 → v0.5（RBAC）

操作员 API 开始强制 `Authorization: Bearer <token>`。现有 CLI 脚本二选一，要么：

```sh
export PORTUNUS_OPERATOR_TOKEN=<token>
```

…要么用 `server.toml` 中的引导捷径：

```toml
operator_token = "<43-char URL-safe-base64 token>"
```

数据面（gRPC client token、TCP/UDP 转发）**逐字节相同** —— client 侧无须改动。

## v1.0 → v1.1（本地 Web 密码）

Web UI 从浏览器输入 Bearer Token 改为本地用户 ID / 密码登录，并使用服务端
`portunus_session` cookie。现有 API Bearer Token 仍可用于 CLI 和自动化。已有
用户不会自动生成密码；`superadmin` 应在 Web UI 中创建或重置本地密码，或在停止
server 后用本地 `reset-password`。

## v1.x 数据面版本

所有 v1.x 升级都是 drop-in：无须手动迁移 SQLite、不破坏线协议、不移除任何
运维接口。升级二进制并重启即可。运维可能注意到的要点：

- **v1.3（Linux `splice(2)` 快路径）** —— 在 Linux 上，对没有带宽上限的
  规则，TCP 吞吐约翻倍。对运维不可见；见
  [故障排查 → 在排错时关闭 Linux 快路径](/zh/docs/operations/troubleshooting#在排错时关闭-linux-快路径)。
- **v1.4（流量配额）** —— 自动应用 SQLite 迁移 `V008`（`traffic_quotas`
  + 历史表）。不做 backfill、不插默认行；未配置的 pair 转发逻辑完全不变。见
  [流量配额运维手册](/zh/docs/operations/runbook-traffic-quotas)。
- **v1.5 —— UDP 每规则 flow 上限修正。** `udp_max_flows_per_rule` 现在在
  整条规则范围内只统计一次，不再按每个监听端口分别统计。**range** UDP 规则
  的有效容量会缩减 `range_size` 倍。如果你依赖旧的虚高容量，请按比例调高
  `udp_max_flows_per_rule` 或拆分 range；该字段上限为 `65535`，所以若
  `cap × range_size` 此前已超过该值，则**必须**拆分规则。单端口 UDP 规则
  不变。
- **v1.6 —— standalone 统计 TUI**（`portunus-standalone stats`）；无
  控制面改动。
- **v1.6.1 —— TCP listener 启用 `SO_REUSEADDR`。** 修复 `docker restart`
  / 进程快速重启时，`accept()` 出的子 socket 残留在 `TIME_WAIT` 引起的
  误报 `port_in_use`。splice 字节计数（`bytes_in` / `bytes_out`）现在对
  长连接增量更新，不再只在连接关闭时更新。
- **v1.7 —— forwarder 硬化。** 修复 UDP 队头阻塞、accept 循环退避、为
  PROXY 协议前导与 IP 目标拨号加超时、bounded DNS cache、多目标 failover
  路径上启用 splice + 字节计数，以及修复每规则限速器在规则移除时的泄漏。
  全部为内部改动，无须运维介入。

## 降级注意

- **v0.5 → v0.4**：`identity.json` 变成空壳；`operator_token` 被静默忽略；
  auth 层不存在，每条操作员请求重新回到未认证状态。RBAC 状态丢失（之后再升回
  v0.5+ 需要重新 bootstrap）。转发规则与 `tokens.json` 保留。
- **v0.8 → v0.7**：不支持。v0.7 写 JSON，v0.8 写 SQLite，没有自动降级路径。

## 查询 schema 版本

该查询返回已应用的最高迁移号（refinery 在 `schema_migrations` 中每条迁移
存一个整数）：

```sh
sqlite3 /var/lib/portunus/state.db \
    'SELECT MAX(version) FROM schema_migrations;'
```

| 版本 | 首次随版本发布 | 迁移 |
| --- | --- | --- |
| `1` | v0.8 | 初始 schema |
| `2` | v0.9 | TLS SNI 路由 |
| `3` | v0.10 | PROXY 协议（per-target） |
| `4` | v0.10 | 规则 runtime 列 |
| `5` | v0.11 | 限流 |
| `6` | v1.1 | 本地密码认证 |
| `7` | v1.3.1 | 每 client 入口地址 |
| `8` | v1.4 | 流量配额 + 历史 |
| `9` | v1.4.2 | client enrollment |
| `10` | v1.4.3 | 运行时通告地址 |

v1.5 – v1.7 不新增 SQLite 迁移，head 保持在 `10`。

较新的二进制启动时会自动应用缺失的迁移。比磁盘 schema 旧的二进制会以退出码
78（`startup.schema_version_too_new`）拒绝启动。
