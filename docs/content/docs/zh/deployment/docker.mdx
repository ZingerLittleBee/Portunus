Docker Compose 是大多数安装场景的推荐部署方式。release pipeline 会把多架构
Linux 镜像推送到 GitHub Container Registry，下面的示例默认使用 `:latest`：

```sh
docker pull ghcr.io/zingerlittlebee/portunus-server:latest
docker pull ghcr.io/zingerlittlebee/portunus-client:latest
```

如果需要完全可复现的生产部署，把 `:latest` 换成固定 release tag，例如
`:1.0.0`。

## Server Compose

在 Server 主机上创建 `compose.yaml`：

```yaml
services:
  server:
    image: ghcr.io/zingerlittlebee/portunus-server:latest
    container_name: portunus-server
    ports:
      - "7443:7443"
      - "127.0.0.1:7080:7080"
    volumes:
      - portunus-data:/var/lib/portunus
    command:
      - --data-dir
      - /var/lib/portunus
      - serve
      - --operator-http-listen
      - 0.0.0.0:7080
    restart: unless-stopped

  operator:
    profiles:
      - cli
    image: ghcr.io/zingerlittlebee/portunus-server:latest
    network_mode: "service:server"
    volumes:
      - portunus-data:/var/lib/portunus
    entrypoint:
      - portunus-server
      - --data-dir
      - /var/lib/portunus
    command:
      - list-clients

  volume-tools:
    profiles:
      - ops
    image: alpine:3.20
    volumes:
      - portunus-data:/data
      - .:/backup
    working_dir: /backup
    command:
      - "true"

volumes:
  portunus-data:
    name: portunus-data
```

启动 Server：

```sh
docker compose up -d server
```

named volume 只保存自动生成的 TLS 材料、SQLite 数据库和其他守护进程持有的
状态，路径是 `/var/lib/portunus`。如果 `/var/lib/portunus/server.toml`
不存在，Server 就使用内置默认值。
Web UI 和 operator API 在容器内绑定 `0.0.0.0:7080`，但只发布到 Server 主机的
loopback：`http://127.0.0.1:7080/`；Client 控制面 listener 仍然监听
`0.0.0.0:7443`。

首次 onboarding 时，从正在运行的 server 日志读取 setup token：

```sh
docker compose logs server | grep 'Portunus onboarding setup token'
```

打开 `http://127.0.0.1:7080/`，粘贴 setup token，并创建第一个 `superadmin`
密码。token 30 分钟后过期；只要 onboarding 还没完成，每次 server 启动都会
重新轮换 token。

如果需要覆写默认值，再把可选 `server.toml` 挂进同一个 data-dir：

```yaml
services:
  server:
    volumes:
      - portunus-data:/var/lib/portunus
      - ./server.toml:/var/lib/portunus/server.toml:ro
```

## 操作员命令

运行时镜像是 distroless，没有 shell。`operator` 服务是一次性 CLI sidecar，
共享 Server 的网络命名空间和同一个 named volume。`docker compose run` 会覆盖
服务里的 `command`，所以每次可以执行不同子命令：

```sh
docker compose run --rm \
  -e PORTUNUS_OPERATOR_TOKEN="$PORTUNUS_OPERATOR_TOKEN" \
  operator list-clients
```

大多数操作员命令走 HTTP API，需要 API Token。Web onboarding 完成后，在 Web UI
的 Credentials 页面创建一个 API Token，再用 `PORTUNUS_OPERATOR_TOKEN` 传给 CLI：

```sh
docker compose run --rm \
  -e PORTUNUS_OPERATOR_TOKEN="$PORTUNUS_OPERATOR_TOKEN" \
  operator provision-client edge-01 --out /tmp/edge-01.bundle.json
```

`provision-client`、`push-rule`、`revoke-client` 等命令也使用同样模式。

本地 break-glass 命令（`reset-password`、`onboarding-token`）会直接打开
SQLite store，因此要先停止 `server`。`onboarding-token` 主要用于离线维护；后续
启动 `serve` 时还会再次轮换 setup token，所以浏览器 onboarding 通常使用
`server` 日志里打印的 token。

```sh
docker compose stop server
docker compose run --rm operator reset-password admin --temporary
docker compose up -d server
```

这里要使用真实的超管用户 ID：`bootstrap-superadmin` 创建的是 `_superadmin`，
Web onboarding 创建的是初始化时选择的 ID，例如 `admin`。

## Client Compose

Server 配发 Client bundle 后，在每台边缘主机上创建 `compose.client.yaml`：

```yaml
services:
  client:
    image: ghcr.io/zingerlittlebee/portunus-client:latest
    container_name: portunus-client
    network_mode: host
    volumes:
      - ./edge-01.bundle.json:/etc/portunus/client.bundle.json:ro
    restart: unless-stopped
```

启动 Client：

```sh
docker compose -f compose.client.yaml up -d
```

`portunus-client` 用 host networking 最省心：operator 推送的规则会在 edge
宿主机上绑定 listener，而 Docker 事先不知道这些端口。如果不用 host networking，
需要显式发布每个转发监听端口。

## 备份与迁移

Server 状态都在一个 named volume 中时，备份和迁移就是卷操作，再加上很小的
Compose 项目文件：

```sh
tar -czf portunus-config.tar.gz compose.yaml
docker compose stop server
docker compose run --rm volume-tools \
  tar -czf /backup/portunus-data.tar.gz -C /data .
docker compose start server
```

在新主机上恢复：

```sh
tar -xzf portunus-config.tar.gz
docker compose run --rm volume-tools \
  tar -xzf /backup/portunus-data.tar.gz -C /data
docker compose up -d server
```

如果用了覆写文件，请把 `server.toml` 和 `compose.yaml` 一起备份。

## 卷与路径

推荐 Compose 示例使用一个可写 named volume：

| 卷 | 容器路径 | 用途 |
| --- | --- | --- |
| `portunus-data` | `/var/lib/portunus/` | 自动生成的 TLS 材料、可选 `server.toml`、`state.db` 与边车文件 |

## 本地镜像构建

仓库里仍然保留了 [`deploy/docker/docker-compose.yml`](https://github.com/ZingerLittleBee/Portunus/blob/main/deploy/docker/docker-compose.yml)，
用于从本地 checkout 构建运行时镜像。它适合开发和 release 验证，不是推荐安装路径：

```sh
./deploy/docker/prepare-build-context.sh
docker compose -f deploy/docker/docker-compose.yml build
```

这个 helper 会在 `rust:1.88-bookworm` 里编译，确保复制进镜像的二进制和 Debian
12 distroless 运行时兼容。不要把更新宿主发行版上编译出来的二进制直接塞进这些
镜像，否则容器启动时可能报 `GLIBC_2.38 not found`。

## 注意事项

- **Distroless 运行时没有 shell**。Portunus CLI 命令走
  `docker compose run --rm operator`；文件排查可以使用临时 sidecar 容器。
- **操作员 API / Web UI** 在容器内绑定 `0.0.0.0:7080`，这样 Docker 能发布端口；
  示例只把它映射到宿主机 loopback：`127.0.0.1:7080:7080`。
- **文件系统分类检查** 在容器中同样会触发 —— 请确保 bind-mount 源是真正
  持久化的本地存储。
- **`portunus-client`** 需要 bundle 落在
  `/etc/portunus/client.bundle.json`（或 `PORTUNUS_CLIENT_BUNDLE`）。

## 健康检查

server 的最小 `HEALTHCHECK`：

```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --start-period=5s --retries=3 \
  CMD ["portunus-server", "list-clients"]
```

（这需要容器环境中存在 `PORTUNUS_OPERATOR_TOKEN`；请通过编排系统的 secrets
机制注入。）

## 网络备注

- 跨主机连接 Client 时，控制面 listener（`7443`）是 **唯一必须暴露** 的
  端口。
- 操作员 HTTP listener（`7080`）在容器内绑定 `0.0.0.0` 以便 Docker 发布端口；
  示例只发布到宿主机 loopback。
- metrics listener（`7081`）仍然保持 loopback-only。
