Docker 是大多数安装场景的推荐部署方式。release pipeline 会把多架构 Linux
镜像推送到 GitHub Container Registry：

```sh
docker pull ghcr.io/zingerlittlebee/portunus-server:1.0.0
docker pull ghcr.io/zingerlittlebee/portunus-client:1.0.0
```

生产环境建议固定 `:1.0.0`。`:latest` 指向最新稳定版：

```sh
docker pull ghcr.io/zingerlittlebee/portunus-server:latest
docker pull ghcr.io/zingerlittlebee/portunus-client:latest
```

## Server

先创建宿主目录：

```sh
mkdir -p ./portunus-config ./portunus-state
```

启动 server：

```sh
docker run -d --name portunus-server \
  -p 7443:7443 \
  -v "$(pwd)/portunus-config:/etc/portunus" \
  -v "$(pwd)/portunus-state:/var/lib/portunus" \
  ghcr.io/zingerlittlebee/portunus-server:1.0.0
```

镜像入口是 `portunus-server`，默认命令是：

```sh
--config-dir /var/lib/portunus --config /etc/portunus/server.toml serve
```

如果 `/etc/portunus/server.toml` 不存在，Portunus 仍会使用 CLI / 编译期默认值
启动，并把生成的 TLS 材料和 SQLite 状态写到 `/var/lib/portunus`。

## 操作员命令

运行时镜像是 distroless，没有 shell。一次性操作员命令可以继续使用同一个镜像，
并加入 server 容器的网络和卷：

```sh
docker run --rm \
  --network container:portunus-server \
  --volumes-from portunus-server \
  ghcr.io/zingerlittlebee/portunus-server:1.0.0 \
  --config-dir /var/lib/portunus \
  --config /etc/portunus/server.toml \
  bootstrap-superadmin --name ops
```

保存打印出来的 token，后续命令通过环境变量传入：

```sh
docker run --rm \
  --network container:portunus-server \
  --volumes-from portunus-server \
  -e PORTUNUS_OPERATOR_TOKEN="$PORTUNUS_OPERATOR_TOKEN" \
  ghcr.io/zingerlittlebee/portunus-server:1.0.0 \
  --config-dir /var/lib/portunus \
  --config /etc/portunus/server.toml \
  list-clients
```

## Client

把 provision 出来的 client bundle 挂载到默认路径：

```sh
docker run -d --name portunus-client \
  --network host \
  -v "$(pwd)/edge-01.bundle.json:/etc/portunus/client.bundle.json:ro" \
  ghcr.io/zingerlittlebee/portunus-client:1.0.0
```

`portunus-client` 用 host networking 最省心：operator 推送的规则会在 edge
宿主机上绑定 listener，而 Docker 事先不知道这些端口。如果不用 host networking，
需要为每个转发监听端口显式加 `-p`。

## 本地试用：docker-compose

```sh
cd deploy/docker
docker compose up -d server
```

compose 文件 **仅供本地** —— 是为快速试玩准备的，不是生产编排。它把一个
server + 可选 client 接到 loopback 路径上，并使用 runtime-only Dockerfile，
所以本地 build 仍需要先把二进制放到 `docker-bin/<arch>/`：

```sh
cargo build --release -p portunus-server -p portunus-client
arch="$(docker version --format '{{.Server.Arch}}')"
mkdir -p "docker-bin/${arch}"
cp target/release/portunus-server target/release/portunus-client "docker-bin/${arch}/"
docker compose -f deploy/docker/docker-compose.yml build
```

## 卷与路径

容器内默认布局沿用 systemd unit 的 FHS：

| 宿主路径 | 容器路径 | 用途 |
| --- | --- | --- |
| `./portunus-config/` | `/etc/portunus/` | TLS 材料、可选 `server.toml`、Client bundle |
| `./portunus-state/` | `/var/lib/portunus/` | `state.db` 与边车文件 |

## 注意事项

- **Distroless 运行时没有 shell**。如需 `docker compose exec` 进入，需要
  compose 中已声明可执行的二进制；临时排查可以挂另一个 sidecar 容器
  `--volumes-from`，或者把构件拉回宿主。
- **操作员 API 的 loopback 绑定** 在容器内强制。要从工作站访问，请走 SSH
  隧道，或在容器内直接调操作员 CLI。
- **文件系统分类检查** 在容器中同样会触发 —— 请确保 bind-mount 源是真正
  的本地文件系统（不要 tmpfs）。
- **`portunus-client`** 需要 bundle 落在
  `/etc/portunus/client.bundle.json`（或 `PORTUNUS_CLIENT_BUNDLE`）。

## 健康检查

server 的最小 `HEALTHCHECK`：

```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --start-period=5s --retries=3 \
  CMD ["portunus-server", "list-clients"]
```

（这需要容器环境中存在 `PORTUNUS_OPERATOR_TOKEN`；请通过编排系统的 secrets
机制注入。）

## 网络备注

- 跨主机连接 Client 时，控制面 listener（`7443`）是 **唯一必须暴露** 的
  端口。
- 操作员 HTTP listener（`7080`）与 metrics listener（`7081`）应在容器内
  保持 loopback；按需通过反向代理 + 认证或 SSH 隧道暴露。
