绝大多数故障都会以结构化日志 **加上** 具体退出码或 HTTP 状态码的形式
出现。两者结合通常足以唯一确定原因。

## Client 无法连接

### `control.connect_failed`（TLS / 证书指纹不匹配）

```json
{"event":"control.connect_failed","error":"…"}
```

server 的叶证书指纹与 bundle 中的 `server_cert_sha256` 不匹配，因此 TLS
握手中止。没有专门的 pin 不匹配事件 —— 失败会以 `control.connect_failed`
的形式出现，证书 / TLS 细节放在其 `error=` 字段里。**client 不会退出**，
而是按退避重试。server 端不会有任何日志（TLS 握手根本没完成）。

**修复**：server 证书重新生成后，用 `enroll-client` 重新生成接入命令；
或在 server 端恢复原来的 `server.crt` + `server.key`。

### `auth.failure { reason: "token_revoked" }`

```sh
portunus-server revoke edge-01
# ↓
# Server 日志：client.disconnected reason=token_revoked
# 重启后 Client 日志：auth.failure reason=token_revoked
```

**修复**：重新生成接入命令，并在边缘主机执行。

### Connect refused / network unreachable

检查 server 端 `control_listen` 是否与 bundle 的 `server_endpoint` 一致，并
确认防火墙没有拦掉 `tcp/7443`。

## 规则激活失败

### `port_in_use`

```sh
portunus-server push-rule edge-01 18080 10.0.0.5:8080
# exit 5: port_in_use: port 18080 already in use
```

client 上有别的东西（另一条规则、无关进程）占用了监听端口。

**修复**：改用一个空闲端口；或先 `remove-rule` 移除占用者再重试。**没有自动
重试** —— 失败的规则会一直占住该端口（生命周期 Q4 决策有意为之），直到被
显式移除。

<Callout type="info">
  **v1.6.1** 起，TCP listener 以 `SO_REUSEADDR` 绑定，因此 `docker
  restart` 或进程快速重启时，已死进程 `accept()` 出的子 socket 残留在
  `TIME_WAIT` 不再导致误报 `port_in_use`（此前会占住端口约
  `tcp_fin_timeout`，默认 60 s）。`SO_REUSEADDR` 只放宽 `TIME_WAIT`
  绑定 —— 同一端口上两个存活的 `LISTEN` socket 仍会冲突，所以重复规则
  仍会被拒为 `port_in_use`。如果在全新启动时看到 `port_in_use`，说明端口
  确实被另一进程或规则占用了。
</Callout>

### `client_not_connected`

```sh
portunus-server push-rule edge-01 18080 …
# exit 4: client_not_connected
```

client 的 gRPC 流断了。规则在 server 端持久化；client 重连后 server 会自动
重推。确认状态：

```sh
portunus-server list-clients
```

## 目标解析（DNS）

目标是 DNS 名称的规则，会在每次新连接 / 新 flow 时惰性解析。解析失败**不会**
让规则失败 —— 只丢掉这次连接（TCP）或首包（UDP），并发出结构化事件：

```json
{"event":"rule.dns_failed","rule":"…","host":"…","reason":"…"}
```

UDP 用 `rule.udp_dns_failed`。每次失败还会递增每规则的 Prometheus
counter：

```text
portunus_rule_dns_failures_total{client="edge-01",owner="alice",rule="42"}
```

**修复**：确认目标主机名能从 client 主机解析（`dig`、`getent hosts`），且
client 的 resolver / `/etc/resolv.conf` 可达。上游 DNS 短暂中断会表现为
counter 上升、恢复后趋平；resolver 会缓存成功结果（遵守 TTL），并在刷新
失败时短暂返回 stale 条目，所以抖动的 resolver 不一定会丢掉每一条连接。

resolver 缓存有上界（v1.7 起加入的内存泄漏防护）：在高基数目标 workload
下，它会优先淘汰最接近过期的条目，而不是无限增长。这是内部保护，没有
运维开关。

## RBAC 拒绝

| 码 | 原因 |
| --- | --- |
| `unauthenticated` | 缺少或非法 Bearer API Token / Web 会话 |
| `not_owner` | 已认证，但调用方不拥有该资源 |
| `client_not_granted` | 调用方在所请求 Client 上没有 Grant |
| `port_outside_grant` | 监听端口落在任何一个 Grant 之外 |
| `protocol_not_granted` | 协议（TCP/UDP）不在 Grant 中 |
| `password_change_required` | Web 会话被限制，用户必须先修改临时密码 |

闭集匹配规则见 [RBAC](/zh/docs/server-client/features/rbac)。

## Server 启动失败

### `startup.unsupported_filesystem`

`--data-dir` 落在 NFS / tmpfs / ramfs。请改本地可写文件系统。

### `startup.store_in_use`（exit 1）

另一个 `portunus-server serve` 已经拿着数据库。找出并停掉那个进程；clustering
不在范围。

### `startup.schema_version_too_new`（exit 78）

用旧二进制对接新 schema（例如把 v0.11 备份恢复到 v0.10 二进制上）。要么保留
旧备份，要么用对应版本的二进制。

### `bootstrap_required`（HTTP 503）

server 还没有任何启用的超管。v1.1.0 起，启动 `serve`，从 stderr / 日志读取
setup token，然后打开 Web UI 创建第一个 `superadmin`：

```sh
journalctl -u portunus-server -n 100 --no-pager \
  | grep 'Portunus onboarding setup token'
```

setup token 30 分钟后过期；只要 onboarding 还没完成，每次 server 启动都会
轮换 token。如果已经存在 `superadmin`，onboarding 不会再次打开，请走下面的
密码恢复流程。

旧自动化仍可用 `bootstrap-superadmin` 或 `server.toml` 中的 `operator_token`
创建 API Token 访问，但这些路径不会创建 Web 密码。

## Web 登录与密码恢复

### `rate_limited`

登录、onboarding、密码重置都会按 subject 和远端 IP 限流。等 lockout 过期后再
试。暴力破解密码本就该枯燥乏味，这里的实现也确实如此。

### 最后一个 `superadmin` 忘记密码

先停止 server 进程，在本地重置现有账户，然后重启。这里要填真实的超管用户
ID：`bootstrap-superadmin` 创建的是 `_superadmin`，Web onboarding 创建的则是
初始化时选择的 ID，例如 `admin`。

```sh
sudo systemctl stop portunus-server
sudo -u portunus-server portunus-server \
  --data-dir /var/lib/portunus \
  reset-password admin --temporary
sudo systemctl start portunus-server
```

该命令只打印一次 `temporary_password=...`，撤销 Web 会话，默认撤销 API Token，
并标记账户下次登录必须改密码。这个场景没有远程 "forgot password" 接口。

## 性能问题

### "升级后吞吐回退"

跑 criterion 基准对比：

```sh
cargo bench -p portunus-client --bench data_plane
```

CI 的回归守门
（`.github/workflows/bench.yml` →
`scripts/bench_regression_gate.py`）只要任何基准的中位数比 v0.1.0 baseline
慢 25% 就会失败。怀疑回归时，可用 `git bisect run cargo bench …` 二分定位。

### "持续 UDP 负载下连接被丢"

查 `portunus_rule_flows_dropped_overflow_total`。非零就把 `server.toml` 中的
`udp_max_flows_per_rule` 调高（同时调高 client systemd unit 的 `LimitNOFILE`）。

**v1.5** 起，flow 上限按**每规则**统计，不再按每监听端口统计 —— 设了
`udp_max_flows_per_rule = N` 的 range 规则在其所有端口上合计只允许 `N`
条 flow（此前是 `N × range_size`）。如果一条在 v1.4 正常的 range 规则
现在溢出，请按比例调高上限或拆分 range；该字段上限为 `65535`。见
[升级指南](/zh/docs/server-client/operations/upgrade#v1x-数据面版本)。

**v1.7** 起，每规则 UDP listener 循环对队头阻塞做了硬化，单个慢上游不再
拖住同规则下其它 flow 的数据报。某条行为异常的 flow 仍可能在收到反射回来的
ICMP 错误时被提前驱逐（`rule.udp_flow_evicted_icmp`）—— 下一个数据报会
重建它。

### "SNI 模式 listener 上 TLS 握手好像变慢"

看 peek 直方图：

```text
histogram_quantile(0.99,
  rate(portunus_tls_client_hello_peek_duration_seconds_bucket[5m]))
```

长尾（接近 3 s）通常说明 client 在慢网络下分片发送 ClientHello —— 不是
Portunus 的问题。

## 流量配额（v1.4+）

完整说明见 [流量配额运维手册](/zh/docs/server-client/operations/runbook-traffic-quotas)。

### "终端用户报告连接在 GB 边界被断开"

怀疑这个 pair 撞到了月配额上限。先看实时 gauge：

```text
portunus_traffic_quota_exhausted{user="alice", client="edge-tokyo"} 1
```

值为 `1` 表示数据面已经把这个 pair 的转发硬杀，并拒绝新连接。用
HTTP status 端点二次确认：

```sh
curl -sS \
  -H 'Authorization: Bearer '"$PORTUNUS_API_TOKEN" \
  https://portunus.example.com/v1/users/alice/quotas/edge-tokyo/status
```

`exhausted: true` 且 `exhausted_at` 非 null 即可确认。

**修复**——二选一：

- 一次性补发本期额度（period 边界不变）：

  ```sh
  curl -sS -X PATCH \
    -H 'Authorization: Bearer '"$PORTUNUS_API_TOKEN" \
    -H 'Content-Type: application/json' \
    https://portunus.example.com/v1/users/alice/quotas/edge-tokyo \
    -d '{"clear_period_usage": true}'
  ```

- 或者永久抬高月上限：

  ```sh
  curl -sS -X PUT \
    -H 'Authorization: Bearer '"$PORTUNUS_API_TOKEN" \
    -H 'Content-Type: application/json' \
    https://portunus.example.com/v1/users/alice/quotas/edge-tokyo \
    -d '{"monthly_bytes": 2199023255552}'
  ```

Server 会立刻下发新的 `TrafficQuotaUpdate { action: SET }`，client
无须重连即可恢复接受连接。

### "配额状态卡住不动"

症状：PUT / PATCH 看起来成功，但 client 仍然在断连接（或仍然在放过
连接）。

检查 1 —— client 会话还活着吗？

```text
portunus_clients_connected{client="edge-tokyo"} 1
```

值为 `0` 说明 control-plane 流断了；server 的配额变更已经排队，
等 client 重连时会 replay 下发。

检查 2 —— client 真的处理了 update 吗？grep client 日志：

```sh
journalctl -u portunus-client | grep traffic_quota.applied_set
```

`traffic_quota.applied_set` 事件行表示 `QuotaHandle` 已装载或替换。
没有这行说明 `TrafficQuotaUpdate` 根本没到 —— 确认 client 二进制是
v1.4.0+。

如果 client 确实重连了，内存里的 `QuotaHandle` 会由 server 的
`TrafficQuotaUpdate` replay 步骤重建（这一步**先于 rule replay**，
所以第一条 rule 绑定 listener 时 handle 已经就位）。无须运维介入
—— 重连后的 StatsReport 周期内，server 端 `bytes_used` 可能短暂
比实际已送达字节略高。

### "流量图表是空的"

Traffic tab 的图表来自 `traffic_samples_1m` / `traffic_samples_1h`。
空有两种可能：根本没生成样本，或查询窗口超出 retention。

- **没生成样本** —— rollup 任务只在 pair 那一分钟 / 那一小时确实
  转发过字节时才写 sample 行。空闲 pair 自然会有缺口。把查询指向
  你确定有流量的时段验证一下。
- **超出 retention** —— `bucket=1m` 行保留 **7 天**；`bucket=1h`
  行保留 **90 天**。`from` 超过对应 retention 的查询会返回
  `422 quota_bucket_out_of_retention`。缩小窗口或切换到大 bucket。
- **刚过去那个小时** —— rollup 任务在 `H+1 min` 运行。`H` 到
  `H+1 min` 之间，小时 `H` 对应的 `1h` 行还不存在；这个范围请用
  `bucket=1m` 查询。

如果底层 gauge（`portunus_traffic_quota_bytes_used` 在涨）显示有
流量、但图表是空的，把 bucket 切到 `1m`，并把查询窗口缩到刚好覆盖那段
活动即可。

## 在排错时关闭 Linux 快路径

在 Linux 上，`portunus-client` 默认对没有 `bandwidth_in_bps` /
`bandwidth_out_bps`（per-rule 或 per-owner）的 TCP 规则启用 `splice(2)`
快路径。这个优化**对运维不可见**——没有规则字段、配置项或 CLI 开关。
为了诊断或压测对照，可用**内部**环境变量 `PORTUNUS_DISABLE_SPLICE=1`
强制所有连接走 userspace 通道：

```sh
PORTUNUS_DISABLE_SPLICE=1 ./portunus-client --bundle ./edge.bundle.json
```

`--help` 里特意不提这个变量。请把它当作**仅供排错的应急开关**，而非稳定 API。

### "我看到 `proxy.splice_unsupported_fallback` 事件"

```json
{"event":"proxy.splice_unsupported_fallback","errno_name":"ENOSYS"}
```

内核（或它前面的 sandbox / LSM / seccomp 策略）拒绝了 `splice` 系统调用。
连接已**透明回落到 userspace 通道**——功能不受影响，只是峰值吞吐略有下降。

**修复**：放宽策略，或接受回落并设置 `PORTUNUS_DISABLE_SPLICE=1` 来消掉
告警。

### "我看到反复 `proxy.splice_pipe_size_failed` 事件"

```json
{"event":"proxy.splice_pipe_size_failed",
 "requested_bytes":1048576,
 "actual_default_bytes":65536,
 "errno_name":"EPERM"}
```

`/proc/sys/fs/pipe-max-size` 低于请求的 1 MiB。快路径仍按内核默认 pipe
大小（多数发行版上是 64 KiB）工作，但大块 workload 的峰值吞吐会下降。

**修复**：调高这个 sysctl（按主机或在 systemd unit 里），或接受降低的
峰值吞吐。

### "Linux 上的吞吐跟 v1.2.0 baseline 一样"

快路径没有生效。依次检查：

1. client 进程里是否设了 `PORTUNUS_DISABLE_SPLICE`？取消它。
2. 规则有没有 `bandwidth_in_bps` 或 `bandwidth_out_bps`？带宽限速需要
   per-chunk userspace token 计数，所以这种情况下快路径会被正确禁用。
3. owner 有没有带宽上限？同 #2。
4. 看 `proxy.splice_selected`（info 级）——一条规则首次进入快路径时会
   触发一次。没有这条事件，说明规则不符合条件。

**v1.6.1** 起，`bytes_in` / `bytes_out` 在 splice 路径上增量更新（每方向
每 64 KiB 批次做一次 `Relaxed` 累加），不再只在连接关闭时更新 —— 长连接
（SSH、gRPC、WebSocket）的速率显示不会再一直停在 0。**v1.7** 起，splice
快路径与实时字节计数在多目标 failover 路径上同样生效。

### "RST 下 `bytes_in` 跟 upstream 实收字节数对不上"

行为与 v1.2.0 userspace 通道完全一致：计数器统计的是**已送达**字节，
不是"已收但未送达"的字节。中途 RST 会丢掉两条路径在途的字节；两侧
计数器对"已送达总数"的认定是一致的。这不是快路径引入的回归。

## 通告地址

完整模型见 [通告地址](/zh/docs/server-client/features/advertised-endpoint)。该地址按
fail-closed 方式解析，并按服务器证书 SAN 校验。

### `endpoint_invalid`（HTTP 422）

`PUT /v1/settings/advertised-endpoint`（或在 Web UI 保存）拒绝了该值：
它不是纯 `host:port`（带了 scheme、path、IPv6 字面量、空白……）。

**修复**：提交纯 `host:port`，例如 `proxy.example.com:34567`。不要带
`https://`，也不要带结尾 path。

### `endpoint_not_in_cert_san`（HTTP 422）

主机格式正确，但**未被服务器证书 SAN 覆盖**。该 override 会被刻意
*不予持久化* —— 服务器 fail closed，不通告无法通过 TLS 证明的主机。
GET 返回 `effective: null` 且带 SAN `diagnostic`、或启动日志报 seed 未
被覆盖，根因相同。

**修复**（operator 操作）：

1. 确认客户端可达的 `host:port`（client 连接的公网地址 —— 例如
   Railway TCP 代理域名）。
2. 检查已部署证书的 SAN：

   ```sh
   openssl x509 -in server.crt -noout -text \
     | grep -A1 "Subject Alternative Name"
   ```

3. 若主机**确实**在其中（注意：通配只匹配最左单标签；IP 字面量需要
   IP SAN，而非 DNS SAN），说明你填错了标签 —— 修正后重试。
4. 若不在，重新签发 / 获取一张 SAN 包含该主机的证书（一条 DNS SAN、
   一条覆盖它的单标签 `*.` 通配，或 IP 字面量对应的 IP SAN），重新
   部署 `server.crt` + `server.key`，再重启。
5. 重新应用 override（`PUT` / Web UI）或修正 seed
   （`PORTUNUS_ADVERTISED_ENDPOINT` / `--advertised-endpoint`）——
   此时返回 `200`。
6. 重建在地址错误期间创建的 enrollment（它们冻结了旧值）。legacy
   `NULL`-endpoint enrollment 无须重建 —— 配置修好后即可成功 redeem。

### 客户端 bundle 指向 `127.0.0.1` / loopback

既没设 override 也没设 seed 时，解析会回落到第 4 层 loopback
（`127.0.0.1:<control_port>`），远程 client 连不上。

**修复**：设置显式 override（Web UI / `PUT`）或 seed
（`PORTUNUS_ADVERTISED_ENDPOINT`）为一个被 SAN 覆盖的公网
`host:port`，然后重建受影响的 enrollment。

### legacy enrollment redeem 报 `failed_precondition`

一条升级前的 enrollment（存储 endpoint 为 NULL）在 redeem 时按 fail-closed
方式解析，而当前配置非法 / 未覆盖。该 enrollment **未被消费**，客户端
token **未轮换**。

**修复**：修正通告地址配置（见上面的 `endpoint_not_in_cert_san`）；
之后 client 可用同一个 code 再次 redeem —— 成功之前是幂等的。
