v0.11.0 起支持。每个上限都 **独立可选**；不设置任何上限时，行为与 v0.10 字节级别完全一致。

## 上限种类

| 上限 | 触发后行为 |
| --- | --- |
| `bandwidth_in_bps` | 入向拷贝半段使用 token bucket 限速。**永不关闭连接** —— 读半段在下一次填充前阻塞。 |
| `bandwidth_out_bps` | 出向同上。两个方向的限流墙钟时间分别计数。 |
| `new_connections_per_sec` | TCP accept-then-RST，UDP 首包丢弃。 |
| `concurrent_connections` | accept（TCP）/ 首包（UDP）时原子 `fetch_add` 后比对上限；RAII guard 在连接关闭时释放。 |

`bandwidth_in_bps`、`bandwidth_out_bps`、`new_connections_per_sec` 各自
支持可选的 burst 覆写；`concurrent_connections` 不支持（它不使用 token
bucket）。缺省时 `burst = 1 × rate`（一秒的 token 量）。Server 校验会把
burst 限制在 `[rate / 100, rate × 60]` 区间内，并拒绝为零的 rate。

<Callout type="info">
  **限流器永远不会主动关闭已有连接** —— 即使热更新把并发上限调到当前
  在线连接数以下，也只会等待其自然排空，仅对新连接按调低后的上限拒绝。
</Callout>

## 与 splice 快路径的交互

v1.3.0 的 Linux TCP 零拷贝快路径在 **accept 时按连接判定**，判定式就一行：

```text
启用 splice  ⇔  Linux  &&  TCP  &&  !PORTUNUS_DISABLE_SPLICE
              &&  规则未设  bandwidth_in_bps / bandwidth_out_bps
              &&  owner 未设 bandwidth_in_bps / bandwidth_out_bps
```

换句话说：

| 规则形态 | 走哪条数据面 | 原因 |
| --- | --- | --- |
| 规则与 owner **都未** 设置带宽 cap | splice 快路径 | 没有需要整形的字节，内核 pipe 直通 |
| 规则或 owner 任一方向设置了带宽 cap | userspace 路径 | token bucket 整形器，与 v1.2.0 字节级别完全一致 |

几个关键边界：

- **`new_connections_per_sec` 与 `concurrent_connections` 不会让 splice 失效。**
  它们是 accept 阶段的准入门控，**在选择数据面之前** 就已完成——只决定一条
  连接是被拒绝还是放行，不参与字节整形。设置了 `concurrent_connections
  = 1000` 但未设置带宽 cap 的规则，照样走 splice。
- 同一台 Server 上限速规则与非限速规则可以共存——每条连接独立判定。
- 热更新增加或移除带宽 cap **只对新连接生效**；已建立的连接保持原路径
  直到关闭。
- splice 仅在 Linux + TCP 上启用。UDP 规则与非 Linux 部署始终走 userspace
  路径，cap 行为完全相同。
- 所有 cap 相关的 metrics、审计字段、throttle 计数与 reject 行为，都与 splice
  是否启用 **无关**。详见 [指标](#指标)。

启用 splice 不会损失任何 QoS 能力：限速规则跳过这次提速，非限速规则则照常受益。

## 每条规则的上限

```sh
portunus-server push-rule edge-01 443 backend.local:8080 \
  --bandwidth-in-bps 1048576 \           # 1 MB/s 入
  --bandwidth-out-bps 524288 \           # 512 KB/s 出
  --new-connections-per-sec 100 \
  --concurrent-connections 1000 \
  --bandwidth-in-burst 2097152           # 可选的 burst 覆写
```

Burst 标志为 `--bandwidth-in-burst`、`--bandwidth-out-burst`、
`--new-connections-burst`。

`list-rules` 的可读输出里会多出一列紧凑的 `CAPS`，显示该规则当前生效的上限。

## 每个所有者的天花板

每个所有者天花板限制的是该用户在某个 Client 上 **所有规则的聚合用量**，
信封以 `(client, owner)` 为键。

每个所有者的上限 **先于** 每条规则的上限生效；触发拒绝时携带的原因带 `owner_*`
前缀。

```sh
# 设置 Alice 在 edge-01 上的信封
portunus-server owner-cap set edge-01 alice \
  --bandwidth-in-bps 10485760 \         # 10 MB/s 聚合
  --concurrent-connections 5000

# 检视
portunus-server owner-cap get edge-01 alice
portunus-server owner-cap list edge-01

# 移除
portunus-server owner-cap delete edge-01 alice
```

REST 接口：`/v1/clients/{client_id}/owners/{owner_id}/rate-limit`
（在 `/v1/clients/{client_id}/owners` 列出所有者）。

## 拒绝原因

`portunus_rate_limit_reject_total{reason=…}` 上的六个值：

- `conn_concurrent`
- `conn_rate`
- `udp_flow_rate`
- `owner_concurrent`
- `owner_conn_rate`
- `owner_udp_flow_rate`

按规则拒绝时 `owner` 标签为空；按所有者拒绝时该标签会被填充。

## 热更新

上限变更通过替换规则上的 `Arc<RuleRateLimiter>` 实现，但保留原有的
`tokens` 与 `last_refill_micros`：

- 调高上限不会凭空产生免费的 burst 额度。
- 调低上限不会让现有的 token pool 立刻失效。
- 把并发上限调到当前在线连接数以下时，等待连接自然排空，**绝不强行关闭**。

v1.7.0 起，规则被移除时会回收其 per-rule 限流器（此前会泄漏）。

## 能力门控

向 v0.10 或更早的 Client 推送 `rate_limit`（或任何 owner 上限变更）会在
任何规则激活之前返回 HTTP 422 `rate_limit_unsupported_by_client`。

## 拒绝路径：TCP accept-then-RST

Portunus **不会暂停 listener** 来拒绝被限流的规则，因为 v0.7 / v0.9 的
listener 在多条规则间共享 —— 暂停会牵连同一 listener 上的其他规则。被限流
拒绝的连接会先 accept，设置 `SO_LINGER {l_onoff: 1, l_linger: 0}`，然后
close，从而对客户端表现为一个干净的 RST。

## 指标

```
portunus_rate_limit_reject_total{client,rule,owner,reason}
portunus_rate_limit_throttle_seconds_total{client,rule,owner,direction}
portunus_rate_limit_active_connections{client,rule,owner}
```

数据面的拒绝 / 限流事件 **只记录在 tracing 中**，**不会** 写入 SQLite 操作员
审计表（沿用 v0.9 D13 不变量）。
