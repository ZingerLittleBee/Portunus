这一页演示原始 CLI 的端到端流程。它对应 v0.1 spec 中的 `quickstart.md`，也正是
`portunus-e2e` 集成测试逐步执行的脚本。

## 0. 前置条件

需要两个终端（或两台主机）：

- 主机 A：已安装 [`portunus-server`](/zh/docs/getting-started/installation#安装-server)，
  并位于 `PATH`。
- 主机 B：已安装 [`portunus-client`](/zh/docs/getting-started/installation#安装-client)，
  并位于 `PATH`。

如果只是单机试用，可在本机同时安装两个二进制，把两个终端分别当作主机 A 和主机 B。

## 1. 引导 API 操作员（主机 A）

```sh
mkdir -p ./srv
portunus-server --data-dir ./srv bootstrap-superadmin --name ops
# → superadmin user_id=_superadmin token=<bearer>
```

<Callout type="warn">
  这里走的是原始 CLI 示例所用的旧 API Token bootstrap 路径。如果是以浏览器为主的
  常规安装，请先启动 `serve`，再用运行中的 server 打印的 setup token 完成
  Web UI onboarding。
</Callout>

Bearer Token **只会输出一次**，请立即保存：

```sh
export PORTUNUS_OPERATOR_TOKEN=<paste-token-here>
```

下文所有操作员子命令都会从环境变量 `PORTUNUS_OPERATOR_TOKEN` 读取 Token。

## 2. 启动 Server（主机 A）

```sh
portunus-server --data-dir ./srv serve
```

Server 首次启动时会创建 `./srv/server.crt`、`./srv/server.key` 以及其余缺失的
状态文件。日志为 JSON：

```json
{"event":"tls.cert_generated","fingerprint_sha256":"…"}
{"event":"server.listening","grpc":"0.0.0.0:7443","operator_http":"127.0.0.1:7080","metrics":"127.0.0.1:7081"}
```

## 3. 配发一个 Client

```sh
portunus-server --data-dir ./srv enroll-client edge-01
```

<Callout type="warn">
`enroll-client` 直接打开 `state.db`，当同一 `--data-dir` 上已有 server
运行时会以 `store_in_use` 失败。上面启动的 server 仍在运行时，请改用
Web UI **Clients** 页面或 operator HTTP API 创建 client——两者都经由运行中的
server。离线 CLI 仅用于 `serve` 启动前的 bootstrap。参见
[Enroll](/zh/docs/configuration/client#enroll)。
</Callout>

在主机 B 上，先安装 client，再兑换上面输出的命令：

```sh
curl -fsSL https://raw.githubusercontent.com/ZingerLittleBee/Portunus/main/scripts/install.sh | sh -s -- client
portunus-client enroll 'portunus://...' --out ./edge-01.bundle.json
```

Client 会校验证书指纹，并以 0600 权限写入 bundle。

## 4. 启动 Client（主机 B）

```sh
portunus-client --bundle ./edge-01.bundle.json
```

数秒之内 Server 就会输出 `client.connected`。可以这样验证：

```sh
portunus-server list-clients
# CLIENT_ID                   NAME      STATE       REMOTE_ADDR          CONNECTED_AT
# 01J8X4ZKQ9A1B2C3D4E5F6G7H8  edge-01   connected   203.0.113.10:51324   2026-05-06T12:01:30+00:00
```

## 5. 推送一条转发规则

```sh
# edge-01:18080 → 10.0.0.5:8080
portunus-server push-rule edge-01 18080 10.0.0.5:8080
# → 1
```

`push-rule` 只在 stdout 上打印新规则的 `rule_id`（这里是 `1`）。

规则会在 2 秒的 ack 预算内由 `Pending` 转为 `Active`（由 `--ack-timeout`
控制，默认 2），随后 Client 开始监听 18080，并把每条新 TCP 连接转发到
`10.0.0.5:8080`。

## 6. 通过规则传输流量

```sh
# 发送 100 MB （目标主机上运行 nc -lk 8080 > /tmp/out.bin）
dd if=/dev/urandom of=/tmp/in.bin bs=1M count=100
nc <host-B> 18080 < /tmp/in.bin

sha256sum /tmp/in.bin /tmp/out.bin   # 哈希必须一致
```

## 7. 观测

```sh
# 通过操作员 CLI 看聚合计数
portunus-server rule-stats 1
# → rule_id=1 client=edge-01 protocol=tcp bytes_in=104857600 bytes_out=0 active=0 dns_failures=0 target_failovers_total=0 updated_at=2026-05-06T12:05:00Z

# 通过 Prometheus 看相同的数字
curl -s http://127.0.0.1:7081/metrics | grep '^portunus_rule'
```

## 8. 拆除

```sh
portunus-server remove-rule 1
# 规则进入 drain，未结束的连接会在 shutdown_drain_timeout_secs 内自然结束
```

向任一二进制发送 `SIGTERM` ：双方都会优雅地 drain 未完成的连接（默认 30 秒），
然后以 `0` 退出。
