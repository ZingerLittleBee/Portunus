`Portunus` 将**控制面**（规则下发、认证、指标聚合）与**数据面**（字节转发）
隔离开来。这两条路径不共享代码，这正是项目能够在版本之间保证数据面稳定的根本
原因。

完整的控制面部署是一个双进程系统：`portunus-server` 管理状态并把规则下发给
一个或多个 `portunus-client` 边缘进程。数据面本身位于独立的
`portunus-forwarder` 库中，client 与单二进制的
[`portunus-standalone`](/zh/docs/configuration/standalone) 转发器都复用它——
因此一台 TOML 驱动的 standalone 主机跑的是同一套转发代码，完全不需要控制面。

## 组件

```
              ┌──────────────────────────────────────┐
              │            portunus-server            │
              │                                      │
   操作员 ────┤  • CLI 子命令                        │
   (CLI/HTTP) │  • 操作员 HTTP API + Web session    │
              │  • Web UI（loopback）               │
              │  • RBAC：用户、凭据、Grant          │
              │  • SQLite 状态存储                  │
              │  • Prometheus /metrics              │
              └──────────────┬───────────────────────┘
                             │ gRPC / TLS-pinned
                             │ （双向流）
              ┌──────────────▼───────────────────────┐
              │            portunus-client            │
              │                                      │
   终端用户 ──▶ TCP accept 循环 / UDP recv_from 循环 │
   流量        │                                      │
              │  • 每条规则一个 Listener             │
              │  • 多目标健康跟踪                    │
              │  • 带缓存的 DNS 解析                 │
              │  • 双向拷贝 / NAT 风格 UDP 流量解   │
              │    复用                              │
              └──────────────┬───────────────────────┘
                             │ TCP / UDP
                             ▼
                       配置的上游目标
```

## Standalone 模式

只有一台主机、又想要同一套转发但不要控制面时，
[`portunus-standalone`](/zh/docs/configuration/standalone) 读取一份 TOML
文件，直接运行 `portunus-forwarder` 数据面：

```
                  portunus.toml
                       │ rules、defaults、[stats]
                       ▼
              ┌──────────────────────────────────────┐
              │          portunus-standalone          │
              │                                      │
   终端用户 ──▶ portunus-forwarder 数据面            │
   流量        │  • 每条规则 TCP / UDP Listener       │
              │  • 端口范围、DNS 目标                │
              │  • 多目标 failover                   │
              │  • PROXY protocol                    │
              │  • 可选的 Unix socket 统计           │
              │    → `portunus-standalone stats` TUI │
              └──────────────┬───────────────────────┘
                             │ TCP / UDP
                             ▼
                       配置的上游目标
```

它跑的是和 `portunus-client` **完全相同的数据面**，因此下面的
[数据面](#数据面) 一节同样适用。其余部分全部没有：没有 gRPC 控制流、
操作员 HTTP API、Web UI、RBAC、审计日志，也没有 Prometheus `/metrics`。
配置就是这份 TOML 文件，重启时重新加载；它的 schema 不暴露 TLS SNI 路由和
限流——这两者只能通过 Server 配置。实时吞吐通过可选的统计 socket 和内置的
`stats` TUI 查看。完整 schema 见
[standalone 指南](/zh/docs/configuration/standalone)。

## 控制面

- **TLS 指纹固定的 Client 认证**。Server 在首次启动时自签一张叶证书；每个
  Client 下发包都包含其 SHA-256 指纹，Client 在每次连接时校验该指纹。
- **操作员 HTTP API** 监听在 `operator_http_listen`（默认 loopback）。
  受保护路由接受 Web session cookie 或 `Authorization: Bearer <token>` API 凭据。
- **gRPC 控制流** 监听在 `control_listen`。规则下发、统计上报、健康更新、
  热更新事件都走 *每个 Client 一条* 的双向流。
- **SQLite 状态** 存放于 `<data-dir>/state.db`（WAL 模式 + refinery 迁移）。
  保存用户、凭据、Grant、多目标规则条目和审计日志。

## 数据面

- **TCP**：每条规则启动一个 `accept` 循环；每条连接做双向拷贝，用户态除内核
  send/recv 窗口外不加任何缓冲。在 Linux 上，未限速规则的字节走内核
  `splice + pipe` 快路径，负载根本不进用户态；任何带宽 cap、UDP 或非 Linux
  构建则回落到用户态拷贝。详见
  [性能报告](/zh/docs/getting-started/performance)。
- **UDP**：每条规则运行一个集中式 runtime——每个监听端口一个 listener 任务、
  一个按 `(监听端口, 源地址)` 键的共享流注册表、一个回程 demux 任务，以及一个
  空闲流回收器。每条流分配一个 `connect()` 过的上游 socket（多目标选择在建流时
  一次性完成），从而获得 NAT 风格的回程隔离。流上限
  （`udp_max_flows_per_rule`）按整条规则强制。在 Linux 上，ICMP 错误
  （`port` / `host` / `network unreachable`）会驱逐受影响的流；下一个数据报会
  对重新选定的目标重建该流。
- **DNS 目标**：首次连接时懒解析，按解析器返回的 TTL 缓存（夹紧到
  `[5 s, 5 min]`），刷新失败时享有 30 秒的 stale-while-error 宽限。
- **多目标故障切换**：每条规则 1..=8 个目标，按 priority 顺序选择。被动健康
  检测（连续 3 次失败切到 `Failed`，连续 2 次成功切回 `Healthy`）始终开启，
  也是 standalone 模式下唯一的健康机制。可选的主动 TCP-connect 探测是
  **仅控制面** 的能力：只有当 Server 用 `--health-check-interval-secs` 下发
  规则时才会运行，因此 TOML 驱动的 `portunus-standalone` 转发器只有被动故障
  切换，不做主动探测。
- **PROXY protocol**：可选的 v1/v2 头部，预置在上游连接前，让目标看到
  原始客户端地址（仅 TCP）。
- **TLS SNI 路由**：peek ClientHello、解析 SNI 扩展、按精确主机或
  单层通配符路由 —— 全程不解密。
- **限流**：带宽用 token bucket 限制；连接速率 / 并发用 accept-then-RST。

## Constitution 不变量

项目用一组小但严格的不变量约束所有版本：

1. **单点认证缝（Single-seam auth）** —— 每条操作员请求都从一个 `auth_layer`
   通过；不存在第二条认证路径。
2. **数据面稳定** —— 除非 spec 显式要求改动，TCP / UDP 转发热路径在版本之间
   字节相同。Criterion 基准在 ±25% 处守门。
3. **Contracts before implementation** —— 每一次线协议或 HTTP API 改动都伴随
   契约测试先于实现落地。
4. **日志中无原始密钥** —— Redaction 层会清洗已知敏感字段名；审计 emitter
   只能看到 verify 之后的身份信息。
5. **Server 端强制** —— SPA 仅渲染 Server 返回的内容；RBAC 隔离绝不放在客户端。

## 版本兼容

线协议改动 **只增不删**。新增字段使用 proto3 默认值剥离，新版 Client 与旧版
Server 通信时会自动剥掉新字段；旧版 Client 与新版 Server 通信时则因
Server 的 capability gate 而看不到任何新行为。

完整版本演进见 [`CHANGELOG.md`](https://github.com/ZingerLittleBee/Portunus/blob/main/CHANGELOG.md)。
