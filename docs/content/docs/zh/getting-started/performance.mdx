这份报告记录当前 **v1.6.1** 的性能状态。这里的数字是**同机实测证据**，不是跨所有
硬件都成立的吞吐承诺。转发性能强依赖内核版本、CPU 频率策略、NIC 路径、socket
buffer，以及路径是否留在内核态。

v1.6.1 的 TCP 数据面与 v1.3.0 引入 splice 时一致；本次修订**换了一台全新 Linux
机器重测**，并新增 **`portunus-standalone` 与 `portunus-client` 的直接对比**——
因为两者共用同一份 `portunus-forwarder` 数据面。v1.3.0 与 v0.11 的旧基线保留在
[历史性能](/zh/docs/getting-started/performance-history) 页面。

## 测试机

这是一台 4 vCPU 的 Debian 13 / kernel 6.12 机器——与
[v0.11 / v1.3.0 测试机](/zh/docs/getting-started/performance-history)
同 OS/内核家族，但硬件不同，所以**两台机器的绝对值不能直接比**，只有同机 delta 才有意义。

## v1.6.1 头条（Linux TCP splice 快路径）

单次 30 秒无带宽限制 TCP 测试（`--seconds 30 --omit-seconds 5`），splice 默认开启，
对同一 `iperf3` target 在 loopback 上背对背采集：

| 路径 | 吞吐 | vs direct | vs iptables |
| --- | ---: | ---: | ---: |
| Direct `iperf3` loopback | 30,890 Mbit/s | 100.0% | — |
| `iptables` `nat/OUTPUT REDIRECT` | 26,987 Mbit/s | 87.4% | 100.0% |
| **`portunus-standalone` uncapped** | **31,936 Mbit/s** | **103.4%** | **118.3%** |
| **`portunus-client` uncapped** | **26,644 Mbit/s** | **86.3%** | **98.7%** |
| Portunus 限制为 `1,048,576 B/s` | 8.388 Mbit/s | n/a | n/a |

有了 splice 快路径，Portunus 紧贴——某些轮次甚至超过——direct `iperf3` loopback
和内核态 `iptables` REDIRECT 基线。kernel `splice + pipe` 完全绕开了 userspace
`read → memcpy → write` 这条限速所有用户态 TCP 程序（包括 `iperf3` 自己）的瓶颈；
同时代理路径的三段 pipeline（iperf client → Portunus → iperf server）在 4 个 vCPU
上的并行度也高于直连两线程。限流测试实测 8.388 Mbit/s（目标 8.389），落在 +10%
验收线内。

> ~30 Gbit/s loopback 下的单次 uncapped 峰值被调度与核分配噪声主导。头条表里行的
> 先后顺序应理解为"三者在同一区间"，而不是固定排名——见下面
> [standalone vs client](#standalone-vs-client) 的重复测试，顺序逐次翻转。

## standalone vs client

`portunus-standalone`（TOML 驱动，无控制面）与 `portunus-client`（双向 gRPC 控制面
+ 规则下发）**端到端共用同一份 `portunus-forwarder` 数据面**——同一条 `proxy.rs` /
`splice` 热路径。规则一旦激活、字节开始流动，转发代码就是同一份。

为把真实差异与 loopback 噪声分开，我们做了六次背对背的 uncapped A/B 重复
（`--seconds 10 --omit-seconds 2`），加上 30 秒头条那次（共 7 对）：

| 轮次 | `portunus-standalone` | `portunus-client` | standalone / client |
| ---: | ---: | ---: | ---: |
| headline (30s) | 31,936 | 26,644 | 119.9% |
| rep 1 | 32,528 | 35,507 | 91.6% |
| rep 2 | 26,250 | 34,782 | 75.5% |
| rep 3 | 14,616 | 27,311 | 53.5% |
| rep 4 | 35,801 | 24,801 | 144.4% |
| rep 5 | 35,025 | 32,886 | 106.5% |
| rep 6 | 27,525 | 27,086 | 101.6% |
| **中位数** | **~30,027** | **~30,099** | **~100%** |

standalone/client 比值在 **53.5% 到 144.4%** 之间摆动——有时 standalone 赢、有时
client 赢——而两者中位数相差仅约 0.2%。**两个构建之间不存在系统性的数据面吞吐
差异。** 这种大幅摆动是 ~30 Gbit/s loopback 下 uncapped 峰值被调度与核分配噪声
主导的结果，与被测二进制无关。

干净、低噪声的证据是下面的 [offered-load sweep](#offered-load-sweep)：在 100 Mbit/s
到 20 Gbit/s 的每一个固定限速点上，standalone 与 client 彼此、以及与 direct /
iptables 之间，都落在 < 1% 以内。

唯一的结构性差异：`portunus-client` 额外运行 gRPC 控制面流和周期性 stats 上报。
在 CPU 饱和的机器上这是边际后台开销——低于数据面测量的噪声地板。**就纯转发吞吐
而言，按你的部署模型选构建（中央控制面 vs 静态 TOML）即可，这不是一个性能决策。**

## Offered-load sweep

下表用 `iperf3 -b <rate>` TCP 应用层 pacing（`--seconds 5 --omit-seconds 1`）回答
比"裸 loopback 峰值"更重要的运维问题："在我的 WAN 带宽下，转发选择会成为瓶颈吗？"
同样 11 个采样点，四条路径在测试机上背对背：

| Offered 带宽 | Direct | iptables REDIRECT | `portunus-standalone` | `portunus-client` |
| ---: | ---: | ---: | ---: | ---: |
| 100 Mbit/s | 100.02 | 100.02 | 100.02 | 100.02 |
| 500 Mbit/s | 499.82 | 499.92 | 499.90 | 500.05 |
| 1 Gbit/s | 999.94 | 999.83 | 999.82 | 999.78 |
| 2.5 Gbit/s | 2,499.94 | 2,499.75 | 2,499.89 | 2,499.75 |
| 5 Gbit/s | 4,999.57 | 4,999.71 | 4,999.48 | 4,999.68 |
| 7.5 Gbit/s | 7,499.46 | 7,499.55 | 7,499.48 | 7,499.30 |
| 10 Gbit/s | 9,998.99 | 9,999.29 | 9,999.61 | 9,999.33 |
| 12.5 Gbit/s | 12,499.27 | 12,499.22 | 12,499.17 | 12,501.03 |
| 15 Gbit/s | 14,999.09 | 14,997.84 | 14,999.13 | 14,999.24 |
| 18 Gbit/s | 17,846.25 | 18,001.85 | 18,013.34 | 18,030.70 |
| 20 Gbit/s | 19,998.82 | 19,924.28 | 19,481.14 | 20,180.43 |

在**整个 100 Mbit/s → 20 Gbit/s 区间，四条路径都在 iperf3 短样本噪声内打到了
offered 速率。** splice 快路径让两个 Portunus 构建在整段 sweep 上都与内核态
`iptables` REDIRECT 基线持平——在这台 4 vCPU 机器上，不存在哪个 offered 区间会让
更丰富的控制面付出可测量的吞吐代价。18-20 Gbit/s 行里的轻微起伏（如 20 Gbit/s 时
standalone 19,481 vs client 20,180）是短跑 pacing 噪声，不是排名——和上面 uncapped
峰值摆动是同一种噪声。

这正是 splice 之前的
[v0.11 基线](/zh/docs/getting-started/performance-history)
守不住的区间：那里 Portunus 在 12.5 Gbit/s 以上掉到 iptables 的 ~33-49%。splice
快路径（v1.3.0 引入）压平了它，v1.6.1 在全新硬件上再次确认。

## splice 之外保持不变

- 每连接 setup 延迟、半关闭语义、字节计数器、Prometheus 指标、审计事件、RBAC、
  限速。**capped 规则继续走 userspace 通道，与 v1.2.0 字节级一致。** splice 启用
  条件按连接判定：`Linux && TCP && !PORTUNUS_DISABLE_SPLICE && 规则与 owner 都没设
  带宽 cap`；`concurrent_connections` 与 `new_connections_per_sec` 是 accept-time
  gate，**不会** 让 splice 失效。详见
  [限流与 QoS — 与 splice 快路径的交互](/zh/docs/features/rate-limiting#%E4%B8%8E-splice-%E5%BF%AB%E8%B7%AF%E5%BE%84%E7%9A%84%E4%BA%A4%E4%BA%92)。
- 跨平台行为：macOS / Windows 构建未变更（splice 用 `#[cfg(target_os = "linux")]`
  门控；`nm` 确认 darwin release 二进制零 splice 符号）。
- 运维表面：无规则字段、无线协议字段、无 Web UI 控件、无 CLI 开关。
  `PORTUNUS_DISABLE_SPLICE=1` 仅用于故障排查 / 压测对比（参见
  [故障排查 — 在排错时关闭 Linux 快路径](/zh/docs/operations/troubleshooting#%E5%9C%A8%E6%8E%92%E9%94%99%E6%97%B6%E5%85%B3%E9%97%AD-linux-%E5%BF%AB%E8%B7%AF%E5%BE%84)）。

## 测试方法

成熟的代理/转发性能报告会先讲清楚方法，再给数字：

- 记录被测对象：commit、release/debug profile、CPU、OS、kernel、工具版本、关键 sysctl。
- 使用 release 二进制，热路径关闭 debug 日志。
- 先测 direct baseline，再在同一台机器上测 proxy path。
- 预热或省略开头几秒，避免 TCP slow start 和进程启动污染样本。
- 同时报告吞吐、RTT/建立延迟、连接行为、重传、拒绝/限流指标。
- uncapped loopback 峰值用**区间**报告，不要用单个数字——多个 10 Gbit/s 量级的
  loopback 峰值被调度噪声主导。
- CI 回归门和"产品性能宣称"分开。CI 负责抓漂移，固定硬件负责给可宣传的基线。

本仓库分三层测：

| 层级 | 命令 | 目的 |
| --- | --- | --- |
| Criterion TCP 数据面 | `cargo bench -p portunus-client --bench data_plane -- --quick` | 稳定的 loopback proxy primitive 回归信号。 |
| Criterion UDP 数据面 | `cargo bench -p portunus-client --bench udp_data_plane -- --quick` | UDP steady-state 与 RTT 回归信号。 |
| 真实进程对比 | `scripts/perf_compare.py` | 同机跑真实 `portunus-server` + `portunus-client` + `portunus-standalone` + `iptables` + `iperf3`。 |

`scripts/perf_compare.py` 是 v1.6.1 的测试 harness，端到端驱动当前 CLI：
`server.toml` 的 `operator_token` 自举、operator HTTP `POST /v1/client-enrollments`
注册流程、`portunus-client enroll` + bundle + `push-rule`，以及 TOML 驱动的
`portunus-standalone` 规则——把 direct、`iptables` REDIRECT、standalone、client
四条路径背对背测一遍。（旧的 `scripts/perf_loopback.py` 早于 v1.6.1 的一次性
enrollment URI 流程和 SQLite `state.db` 文件锁，它的 `provision-client`/bundle
路径已经跑不通。）

## 解读

Portunus 是用户态 L4 forwarder。它 accept TCP 连接，dial target，然后在两条 socket
之间搬字节。这让它比纯内核 NAT 做得多，但路径必然要跨用户态做内核不做的事。

每条 flow 实际付多少代价，要看规则形态：

- **纯 TCP、无带宽 cap、Linux**：字节搬运走内核 `splice + pipe`，字节根本不进
  用户态。这台机器上的 [offered-load sweep](#offered-load-sweep) 显示两个 Portunus
  构建在 100 Mbit/s 到 20 Gbit/s 全程贴近 direct `iperf3` 与 `iptables` REDIRECT
  （落在 iperf3 噪声内）。
- **有任何带宽 cap、UDP、macOS / Windows、或显式禁用 splice**：依然走经典 userspace
  路径。这种形态下 kernel NAT / nftables flowtables 仍然是性能天花板——userspace
  `read → memcpy → write` 有实打实的每 flow 代价。

无论哪条路径，需要「远端 client 注册 + 中央规则下发 + 多租户 RBAC + 可观测 + QoS +
SNI 路由 + PROXY 协议 + 规则生命周期」时，Portunus 才是合适的工具。一条 DNAT 规则
给不了你这些——只比纯吞吐数字是错位的比较，除非你确实只需要那一条。

## 常见转发方案对比

| 维度 | **Portunus（v1.6.1）** | `iptables` DNAT / MASQUERADE | `nftables` + flowtables | NGINX `stream` | HAProxy TCP mode | Envoy `tcp_proxy` | `rinetd` | `socat` | SSH `-L` / `-R` |
| --- | --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 性能预期 | 用户态 L4，给无带宽限制的 TCP 流配 Linux `splice` 快路径。测试机上贴近 `iptables` REDIRECT 到 20 Gbit/s；限速、UDP、非 Linux 仍走经典用户态 copy。 | 最高。内核路径，无用户态 copy。 | 最高。合格 flow 可绕开部分经典 forwarding path。 | 高性能用户态 proxy，事件循环成熟。 | 极高性能用户态 TCP proxy，连接处理成熟。 | 高性能但更重，面向 service mesh / xDS。 | 轻量用户态 redirector。 | 灵活诊断管道，不是托管式生产 proxy。 | 加密隧道，吞吐受 SSH crypto 影响。 |
| TCP | 有 | 有 | 有 | 有 | 有 | 有 | 有 | 有 | 有 |
| UDP | 有（首包强制） | 有 | 有 | 有 | 无通用 UDP 转发 | 有专门 UDP 场景 / 过滤器 | 有 | 有 | 无原生 UDP |
| 动态远端规则下发 | 一等公民：Server 通过 pinned TLS 向 edge Client 下发签名规则包；CLI、Operator HTTP API、内嵌 Web UI、热更新。 | 无内置控制面 | 无内置控制面 | reload/API 取决于版本和配置 | Runtime API 支持很多操作 | xDS 控制面 | 配置/reload 风格 | 无 | 按 session |
| RBAC / 审计 / 指标 | 原生：按用户 / Client / 协议 / 端口段 RBAC；结构化审计；Prometheus 指标；内嵌 SQLite。 | 依赖外部系统 | 依赖外部系统 | 模块化 metrics，无原生租户模型 | stats、ACL、stick tables 很强 | telemetry 和 policy 很强 | 极少 | 极少 | SSH auth/log |
| QoS / rate limit | 每条规则与每个 owner：`bandwidth_in/out_bps`、`new_connections_per_sec`、`concurrent_connections`。token bucket shaper；限速规则走经典用户态路径。 | 通过 `tc` / nftables 生态做基础整形 | nftables / `tc`，不是 app-owner 维度 | `limit_conn` / `limit_rate` 类控制 | 丰富的连接/速率控制 | filters / overload manager | 极少 | 极少 | 极少 |
| 适合场景 | 中央化管理边缘 listener，带租户 RBAC、per-owner QoS、可观测、SNI dispatch、PROXY 协议——TCP/UDP，单端口或端口段，IP 或 DNS target。 | 本机/本网络静态 NAT。 | 现代 Linux packet filtering / forwarding。 | 通用 TCP/UDP proxy。 | TCP 足够时的 L4/L7 负载均衡。 | service mesh 或 xDS 管理环境。 | 简单端口 redirect baseline。 | 实验和调试。 | 运维方便的加密隧道。 |

公平测试不是"找几篇网上数字拼表"，而是：

1. 同一台机器或同一组两机拓扑。
2. 同一 target workload，TCP/UDP 都用 `iperf3`。
3. 同样 duration、warm-up、socket buffer、CPU governor、MTU、kernel。
4. 先测 direct baseline。
5. 每次只启用一种转发实现。

## 复现流程

### 本机 / VPS 对比

```sh
# Debian/Ubuntu
sudo apt-get update
sudo apt-get install -y build-essential cmake pkg-config protobuf-compiler \
  git curl iperf3 iptables

curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh -s -- -y
source "$HOME/.cargo/env"

git clone https://github.com/ZingerLittleBee/Portunus.git
cd Portunus
PORTUNUS_SKIP_WEBUI=1 cargo build --release \
  -p portunus-standalone -p portunus-server -p portunus-client

# /tmp 可能是 tmpfs；portunus 拒绝在 tmpfs 上放 state.db。改用 $HOME。
export TMPDIR="$HOME"

# 头条：direct / iptables / standalone / client + 限流收敛。
# --with-iptables 仅 Linux，需要 root 或免密码 sudo。
python3 scripts/perf_compare.py --seconds 30 --omit-seconds 5 \
  --with-iptables --cap-bytes-per-sec 1048576 \
  --json-out perf-headline.json

# 四条路径的 offered-load sweep。
python3 scripts/perf_compare.py --seconds 5 --omit-seconds 1 \
  --with-iptables --cap-bytes-per-sec 0 \
  --offered-mbps 100,500,1000,2500,5000,7500,10000,12500,15000,18000,20000 \
  --json-out perf-sweep.json
```

常用开关：`--skip-client`（只测 standalone）、`--skip-standalone`（只测 client）、
`--server-bin` / `--client-bin` / `--standalone-bin` 指向预构建二进制。头条 JSON
输出形状：

```json
{
  "direct": { "mbps": 30889.985, "retransmits": 147 },
  "iptables_redirect": { "mbps": 26987.079, "retransmits": 16 },
  "iptables_vs_direct_pct": 87.37,
  "standalone_uncapped": { "mbps": 31935.997, "retransmits": 9 },
  "standalone_vs_direct_pct": 103.39,
  "standalone_vs_iptables_pct": 118.34,
  "client_uncapped": { "mbps": 26644.098, "retransmits": 4 },
  "client_vs_direct_pct": 86.25,
  "client_vs_iptables_pct": 98.73,
  "standalone_vs_client_pct": 119.86,
  "client_capped": {
    "cap_bytes_per_sec": 1048576, "mbps": 8.388,
    "target_mbps": 8.389, "within_plus_10pct": true
  }
}
```

两机测试时，把 `portunus-server` 放控制机，`portunus-client`（或
`portunus-standalone`）与 `iperf3 -s` 放 edge / target 机，然后从第三台执行
`iperf3 -c <edge-listen-ip> -p <listen_port> -t 30 -O 5 -J`。同一轮里必须再直连
target 跑一次 `iperf3` 作为 baseline。
