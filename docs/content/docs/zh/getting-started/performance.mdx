这份报告记录当前 v1.3.0 的性能状态。这里的数字是**同机实测证据**，不是跨所有硬件
都成立的吞吐承诺。转发性能强依赖内核版本、CPU 频率策略、NIC 路径、socket buffer、
以及路径是否留在内核态。

## v1.3.0 头条（Linux TCP splice 快路径）

在专用 Linux 测试机（Debian 13、kernel 6.12.74-2、glibc 2.41，与
[v0.11 Linux iptables baseline](/zh/docs/getting-started/performance-history#linux-iptables-baseline)
同一台硬件）上，v1.3.0 的 splice 快路径让**未带宽限制的 TCP 规则单流吞吐翻倍以上**：

| 模式 | iperf3 direct loopback | Portunus uncapped | Portunus / direct | vs v1.2.0 userspace |
| --- | ---: | ---: | ---: | ---: |
| splice OFF（`PORTUNUS_DISABLE_SPLICE=1`，v1.2.0 userspace） | 18,288 Mbit/s | 9,954 Mbit/s | 54.4% | （基线） |
| splice ON（v1.3.0 默认） | 18,820 Mbit/s | **21,922 Mbit/s** | **116.5%** | **2.20×** |

splice ON 的数字**超过** direct `iperf3` loopback，是因为 kernel
`splice + pipe` 完全绕开了 userspace `read → memcpy → write`
这条限速所有用户态 TCP 程序（包括 `iperf3` 自己）的瓶颈；同时三段
pipeline（iperf client → portunus-client → iperf server）的并行度
也高于直连两线程。

### Offered-load sweep（v1.3.0 攻破 v0.11 saturation 区间）

v0.11 的报告明确指出 `≥ 12.5 Gbit/s` 是 Portunus 早期饱和、远远落后
iptables 的区间。v1.3.0 的 splice 快路径在这台机器上把这段差距压平了。
下表用与 v0.11 baseline 同一份 `scripts/perf_loopback.py --with-iptables
--offered-mbps …` —— 同样 11 个采样点、同样 iptables REDIRECT 对照、
同一台机器 —— 分别在 v1.3.0 splice ON（默认）与 splice OFF
（`PORTUNUS_DISABLE_SPLICE=1`）下背对背采集：

| Offered 带宽 | Direct | iptables REDIRECT | Portunus + splice OFF | Portunus + splice ON | splice 提速 | splice OFF / iptables | splice ON / iptables |
| ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| 100 Mbit/s | 100 | 100 | 100 | 100 | 1.00× | 100.0% | 100.0% |
| 500 Mbit/s | 500 | 500 | 500 | 500 | 1.00× | 100.0% | 100.0% |
| 1 Gbit/s | 1,000 | 1,000 | 1,000 | 1,000 | 1.00× | 100.0% | 100.0% |
| 2.5 Gbit/s | 2,500 | 2,500 | 2,500 | 2,498 | 1.00× | 100.0% | 99.9% |
| 5 Gbit/s | 4,999 | 5,005 | 5,207 | 4,999 | 0.96× | 104.0% | 99.9% |
| 7.5 Gbit/s | 7,476 | 7,492 | 6,775 | 7,499 | 1.11× | 90.4% | 100.1% |
| 10 Gbit/s | 9,999 | 10,484 | 8,870 | 10,035 | 1.13× | 84.6% | 95.7% |
| **12.5 Gbit/s** | 12,598 | 12,545 | **8,556** | **12,373** | **1.45×** | **68.2%** | **98.6%** |
| **15 Gbit/s** | 15,569 | 15,523 | **9,953** | **16,370** | **1.64×** | **64.1%** | **105.5%** |
| **18 Gbit/s** | 18,406 | 18,017 | **8,347** | **19,618** | **2.35×** | **46.3%** | **108.9%** |
| **20 Gbit/s** | 19,088 | 18,738 | **8,690** | **20,147** | **2.32×** | **46.4%** | **107.5%** |

对比 v0.11 [offered-load 表](/zh/docs/getting-started/performance-history#offered-load-sweep)：12.5 Gbit/s
offered 时，v0.11 跑出 6,529 Mbit/s（49% of iptables），v1.3.0 splice
OFF 跑出 8,556 Mbit/s——依然明显落后 iptables。v1.3.0 splice ON 达到
**12,373 Mbit/s（98.6% of iptables）——基本等于 offered 目标**。15-20
Gbit/s 整段区间里，splice ON 与 direct loopback、iptables REDIRECT 的
差距都收敛到 iperf3 短样本噪声范围内；而 splice OFF 的 userspace 路径
仍然被 v1.2.0 userspace memcpy 顶在 ~8.5-10 Gbit/s。

5,000 Mbit/s 行的 0.96× 是 iperf3 short-run pacing 噪声（此机 ±200
Mbit/s），不是回归；splice OFF 与 splice ON 在 < 10 Gbit/s 区间相差
≤5%。15-20 Gbit/s 区间里 `splice ON / iptables` 列略高于 100%，原因
与头条表里一致——三段 pipeline 能比 direct / iptables 两段路径多吃
一个核。

### splice 之外保持不变

- 每连接 setup 延迟、半关闭语义、字节计数器、Prometheus 指标、审计事件、
  RBAC、限速。**capped 规则继续走 userspace 通道，与 v1.2.0 字节级一致。**
  splice 启用条件按连接判定：`Linux && TCP && !PORTUNUS_DISABLE_SPLICE
  && 规则与 owner 都没设带宽 cap`；`concurrent_connections` 与
  `new_connections_per_sec` 是 accept-time gate，**不会** 让 splice 失效。
  详见 [限流与 QoS — 与 splice 快路径的交互](/zh/docs/features/rate-limiting#%E4%B8%8E-splice-%E5%BF%AB%E8%B7%AF%E5%BE%84%E7%9A%84%E4%BA%A4%E4%BA%92)。
- 跨平台行为：macOS / Windows 构建未变更（splice 用
  `#[cfg(target_os = "linux")]` 门控；`nm` 确认 darwin release 二进制零
  splice 符号）。
- 运维表面：无规则字段、无线协议字段、无 Web UI 控件、无 CLI 开关。
  `PORTUNUS_DISABLE_SPLICE=1` 仅用于故障排查 / 压测对比（参见
  [故障排查 — 在排错时关闭 Linux 快路径](/zh/docs/operations/troubleshooting#%E5%9C%A8%E6%8E%92%E9%94%99%E6%97%B6%E5%85%B3%E9%97%AD-linux-%E5%BF%AB%E8%B7%AF%E5%BE%84)）。

整个 `portunus-client` 测试套件（218 个 tests，含 30 个 splice 专项）在 splice on
**和** `PORTUNUS_DISABLE_SPLICE=1` 两种模式下字节级一致通过——所有现有规则形态下
的"字节稳定性"是发版门（012-tcp-zero-copy-splice SC-003）。

## 测试方法

成熟的代理/转发性能报告会先讲清楚方法，再给数字：

- 记录被测对象：commit、release/debug profile、CPU、OS、kernel、工具版本、关键 sysctl。
- 使用 release 二进制，热路径关闭 debug 日志。
- 先测 direct baseline，再在同一台机器上测 proxy path。
- 预热或省略开头几秒，避免 TCP slow start 和进程启动污染样本。
- 同时报告吞吐、RTT/建立延迟、连接行为、重传、拒绝/限流指标。
- CI 回归门和“产品性能宣称”分开。CI 负责抓漂移，固定硬件负责给可宣传的基线。

本仓库现在分三层测：

| 层级 | 命令 | 目的 |
| --- | --- | --- |
| Criterion TCP 数据面 | `cargo bench -p portunus-client --bench data_plane -- --quick` | 稳定的 loopback proxy primitive 回归信号。 |
| Criterion UDP 数据面 | `cargo bench -p portunus-client --bench udp_data_plane -- --quick` | UDP steady-state 与 RTT 回归信号。 |
| 真实进程 smoke | `scripts/perf_loopback.py` | 跑真实 `portunus-server`、`portunus-client`、`push-rule` 和 `iperf3`。 |

### 真实进程 Loopback

命令：

```sh
scripts/perf_loopback.py --seconds 3 --omit-seconds 1
```

结果：

| Path | Throughput |
| --- | ---: |
| Direct `iperf3` loopback | 80,064.984 Mbit/s |
| 通过 Portunus uncapped rule | 36,751.910 Mbit/s |
| Portunus / direct loopback | 45.90% |
| Portunus 限制为 `1,048,576 B/s` | 8.394 Mbit/s |

这组本机结果是在 macOS 上采集的，所以**不包含** Linux-only 的 iptables baseline。

`Portunus / direct loopback` 只是：

```text
portunus_uncapped / direct_iperf3_loopback * 100
```

它不是 iptables 对比。Linux 上可以加 `--with-iptables`，脚本会额外跑一组
同机 `iptables -t nat OUTPUT ... REDIRECT` 作为内核态端口重定向 baseline。
输出会多出：

| 字段 | 含义 |
| --- | --- |
| `iptables_redirect` | `iperf3` 通过内核 loopback 端口重定向的吞吐。 |
| `iptables_vs_direct_pct` | `iptables_redirect / direct * 100`。 |
| `portunus_vs_iptables_pct` | `portunus_uncapped / iptables_redirect * 100`。 |

限流目标是 8.389 Mbit/s，实测落在 +10% 验收线内。
uncapped 占比 18.04% → 45.90% 的提升来自下面 [近期优化](#近期优化)。

## 解读

Portunus 是用户态 L4 forwarder。它 accept TCP 连接，dial target，然后通过 Tokio
copy 字节。这让它比纯内核 NAT 做得多：控制面、RBAC、审计、owner QoS、metrics、
SNI dispatch、PROXY protocol、规则生命周期。但代价也明确：路径会跨用户态，
每条 flow 有更多工作。

如果只追求同机原始转发，内核 NAT / nftables flowtables 应该是性能天花板。
如果需求是“远端 client 注册 + 中央规则下发 + 多租户 RBAC + 可观测 + QoS”，
Portunus 才是这类问题的工具。拿它去和一条 DNAT 规则比纯吞吐，可以比，但
这有点像拿瑞士军刀和一根针比扎洞速度，针当然快，废话。

## 近期优化

上面的数字反映 v0.11 hot-path 落地的三处调优。每一项都是
byte-stable（不改语义、不改 wire format），未启用时回落到 v0.10 行为。

| 改动 | 文件 | 效果 |
| --- | --- | --- |
| Uncapped TCP copy buffer 8 KiB → 64 KiB | `crates/portunus-client/src/forwarder/proxy.rs` | `tokio::io::copy_bidirectional` → `copy_bidirectional_with_sizes(64 KiB, 64 KiB)`。bulk 转发 read/write 系统调用减少；每条热连接常驻内存 16 KiB → 128 KiB。 |
| 集中式 `dial()` + `SO_SNDBUF` / `SO_RCVBUF` | `crates/portunus-client/src/resolver/mod.rs` | 出向 TCP dial 走单一 helper：按地址族新建 `TcpSocket`，设置 `dial_send_buffer_bytes` / `dial_recv_buffer_bytes`（默认各 1 MiB），再 `connect`。loopback 收益小；高 BDP WAN 路径收益明显。best-effort，内核可能 clamp。 |
| 限速路径 chunk: 固定 16 KiB → 动态 | `crates/portunus-client/src/forwarder/rate_limit/copy.rs` | 单次 chunk = `clamp(tightest_cap_bps × 100 ms, 8 KiB, 64 KiB)`。高速 cap 享受快路径同等的 syscall 摊销；低速 cap 不会一次读大块再睡很久。10% 限速精度 gate（`t010_bandwidth_cap_shapes_to_target_rate_within_10pct`）依旧通过。 |

测得的变化：

* Criterion 1 MiB echo：~995 µs → ~575 µs（约 1.7× 吞吐）
* Criterion 64 KiB echo：~115 µs → ~90 µs（约 1.3× 吞吐）
* Criterion 1-byte 通过 proxy 的 RTT：45 µs → 46 µs（在噪声内）
* 真实进程 loopback uncapped：14.2 Gbit/s → 36.8 Gbit/s
  （direct loopback 18.0% → 45.9%）
* 真实进程限制 1 MiB/s：8.388 Mbit/s → 8.394 Mbit/s
  （仍在 +10% 验收线内）

不变的：单连接最低延迟、回归 gate 行为、RBAC / 审计 / per-owner QoS
语义、wire format、SQLite schema 范围。1 字节往返延迟在噪声内不变是因为
`TCP_NODELAY` 仍在两端开启，更大的 copy buffer 由内核按需填充。

后续待办（不在本次）：

* Linux-only `splice(2)` 零拷贝快路径，仅覆盖 “无 preread / 无 PROXY
  prelude / 无 bandwidth cap” 的流。需独立 feature flag，因为它会让数据面分叉。
* `dial_send_buffer_bytes` / `dial_recv_buffer_bytes` 接到 CLI / 配置。
  默认值合理；只有当运维上报真实链路被内核 clamp 时再暴露。

## 常见转发方案对比

| 维度 | `iptables` DNAT / MASQUERADE | `nftables` + flowtables | NGINX `stream` | HAProxy TCP mode | Envoy `tcp_proxy` | `rinetd` | `socat` | SSH `-L` / `-R` |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| 性能预期 | 最高。内核路径，无用户态 copy。 | 最高。合格 flow 可绕开部分经典 forwarding path。 | 高性能用户态 proxy，事件循环成熟。 | 极高性能用户态 TCP proxy，连接处理成熟。 | 高性能但更重，面向 service mesh / xDS。 | 轻量用户态 redirector。 | 灵活诊断管道，不是托管式生产 proxy。 | 加密隧道，吞吐受 SSH crypto 影响。 |
| TCP | 有 | 有 | 有 | 有 | 有 | 有 | 有 | 有 |
| UDP | 有 | 有 | 有 | 无通用 UDP 转发 | 有专门 UDP 场景 / 过滤器 | 有 | 有 | 无原生 UDP |
| 动态远端规则下发 | 无内置控制面 | 无内置控制面 | reload/API 取决于版本和配置 | Runtime API 支持很多操作 | xDS 控制面 | 配置/reload 风格 | 无 | 按 session |
| RBAC / 审计 / 指标 | 依赖外部系统 | 依赖外部系统 | 模块化 metrics，无 Portunus 这种租户模型 | stats、ACL、stick tables 很强 | telemetry 和 policy 很强 | 极少 | 极少 | SSH auth/log |
| QoS / rate limit | 通过 `tc` / nftables 生态做基础整形 | nftables / `tc`，不是 app-owner 维度 | `limit_conn` / `limit_rate` 类控制 | 丰富的连接/速率控制 | filters / overload manager | 极少 | 极少 | 极少 |
| 适合场景 | 本机/本网络静态 NAT。 | 现代 Linux packet filtering / forwarding。 | 通用 TCP/UDP proxy。 | TCP 足够时的 L4/L7 负载均衡。 | service mesh 或 xDS 管理环境。 | 简单端口 redirect baseline。 | 实验和调试。 | 运维方便的加密隧道。 |

公平测试不是“找几篇网上数字拼表”，而是：

1. 同一台机器或同一组两机拓扑。
2. 同一 target workload，TCP/UDP 都用 `iperf3`。
3. 同样 duration、warm-up、socket buffer、CPU governor、MTU、kernel。
4. 先测 direct baseline。
5. 每次只启用一种转发实现。

## 复现流程

### 本机 smoke

```sh
# macOS
brew install iperf3

# Linux
sudo apt-get update
sudo apt-get install -y iperf3

PORTUNUS_SKIP_WEBUI=1 cargo build --release -p portunus-server -p portunus-client
scripts/perf_loopback.py --seconds 10 --omit-seconds 2 --json-out /tmp/portunus-perf.json

# 仅 Linux，需要 root 或免密码 sudo 执行 iptables。
scripts/perf_loopback.py --seconds 10 --omit-seconds 2 --with-iptables \
  --json-out /tmp/portunus-perf.json
```

输出形状：

```json
{
  "direct": { "mbps": 80064.984, "retransmits": null },
  "portunus_uncapped": { "mbps": 36751.910, "retransmits": null },
  "portunus_vs_direct_pct": 45.9,
  "portunus_capped": {
    "cap_bytes_per_sec": 1048576,
    "mbps": 8.394,
    "target_mbps": 8.389,
    "within_plus_10pct": true
  }
}
```

Linux 上加 `--with-iptables` 后，JSON 还会包含：

```json
{
  "iptables_redirect": { "mbps": 0.0, "retransmits": null },
  "iptables_vs_direct_pct": 0.0,
  "portunus_vs_iptables_pct": 0.0
}
```

### VPS

SSH 凭证走线下或密码管理器，不要写进仓库。

```sh
ssh forward-bench@example-edge
sudo apt-get update
sudo apt-get install -y build-essential curl git iperf3 protobuf-compiler
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
```

两机测试时，把 `portunus-server` 放控制机，`portunus-client` 与 `iperf3 -s` 放 edge /
target 机，然后从第三台或控制机执行：

```sh
iperf3 -J -c <edge-listen-ip> -p <listen_port> -t 30 -O 5
```

同一轮里必须再直连 target 跑一次 `iperf3`，否则没有 direct baseline，数字就没法用。
