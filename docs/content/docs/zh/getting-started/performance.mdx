这份报告记录 **v1.6.1** 当前的性能表现。请把这里的数字理解为**同机实测证据**——
在某一台具体机器上测得的结果——而不是适用于所有硬件的吞吐承诺。转发性能主要取决于
内核版本、CPU 频率策略、网络路径、socket 缓冲区大小，以及流量是留在内核态、还是要
进入用户态（应用自己的内存空间）。

就 TCP 而言，v1.6.1 的数据面与引入 splice 快路径的 v1.3.0 完全一致。本次修订**在同一台
机器上端到端重测**，并新增 **`portunus-standalone` 与 `portunus-client`
的直接对比**——因为两个二进制共用同一份 `portunus-forwarder` 数据面。v1.3.0 与
v0.11 的旧基线保留在
[历史性能](/zh/docs/getting-started/performance-history) 页面。

> **一句话结论。** 在没有带宽限制的纯 TCP 规则上，Portunus 一路到 20 Gbit/s 都能
> 跟上内核自带的 `iptables` 转发；而 standalone 与 client 两个构建之间，转发吞吐
> 没有差异。

## 测试机

下面所有 v1.6.1 数字均采集于：

```text
Linux host 6.12.38+deb13-amd64 #1 SMP PREEMPT_DYNAMIC Debian 6.12.38-1 x86_64
AMD EPYC 7B13 (4 vCPU), 7.8 GiB RAM
rustc 1.96.0 (ac68faa20 2026-05-25)
iperf 3.18 (cJSON 1.7.15)
```

就是跑出 [v0.11 / v1.3.0 基线](/zh/docs/getting-started/performance-history)
的**同一台 4 vCPU Debian 13 / kernel 6.12 机器**。请把每个结果当作**同机、背靠背
测出的 delta**（同一次运行里 `iperf3`、`iptables`、Portunus 三者对比），而不是
跨报告比绝对值——不同测试会话间工具链和内核小版本会变。

所有测试都跑在 **loopback 回环接口**（`lo`——流量完全不出本机）上，这样网络本身
永远不会成为瓶颈，测到的就是转发器自身的开销。

## v1.6.1 头条（Linux TCP splice 快路径）

单次 30 秒、无带宽限制的 TCP 测试（`--seconds 30 --omit-seconds 5`；其中
`--omit-seconds` 会丢弃开头几秒，避免 TCP 慢启动拉低平均值），splice 默认开启，对
同一个 `iperf3` target 背对背采集：

每个转发器各占一列，方便与 `iptables` 基线并排对比：

| | `iptables` REDIRECT（基线） | `portunus-standalone` | `portunus-client` | `iperf3` |
| --- | ---: | ---: | ---: | ---: |
| 吞吐 | 26,987 Mbit/s | **31,936 Mbit/s** | **26,644 Mbit/s** | 30,890 Mbit/s |
| vs `iptables` | 100.0% | **118.3%** | **98.7%** | 114.5% |

我们以 **`iptables` REDIRECT 为 100% 基线**：它是 Linux 内核在内核态完成整个转发
（一种静态 NAT 重定向——没有任何用户态进程碰到这些字节），也就是这台机器上一个
转发器现实中能达到的最快速度。**vs `iptables`** 行就是该列吞吐除以这条基线——100%
表示与内核转发持平，高于 100% 表示比它还快。于是 `portunus-standalone` 跑到内核
iptables 的 **118.3%**，`portunus-client` 为 **98.7%**（基本持平）。`iperf3` 列
（完全不经转发器）则是吞吐上限，为 114.5%。

有了 splice 快路径，Portunus 跟上了——某些轮次甚至超过——内核 `iptables` 转发。`splice` 是
一个 Linux 系统调用，它把字节从一个 socket 搬到另一个 socket，**全程不拷贝进应用
内存**（即"零拷贝"）；Portunus 再配一个内核 `pipe`（管道），于是被转发的 TCP 连接
的负载根本不会被拖进用户态。这就绕开了 `read → memcpy → write` 这条限速本机上
*每一个*普通用户态 TCP 程序（包括 `iperf3` 自己）的往返路径。此外，代理路径以三个
进程运行（iperf client → Portunus → iperf server），在 4 个 vCPU 上的分摊比直连
两线程的 `iperf3` 基线更充分——这正是 Portunus 能略微*超过* 100% 的原因。

一条限流的 client 规则（上限 `1,048,576 B/s`）实测 8.388 Mbit/s，目标 8.389
Mbit/s——落在 +10% 验收线内，说明带宽限速是准确的。

> ~30 Gbit/s 下的单次 uncapped 峰值主要被**调度噪声**主导：在这个速度上，结果取决于
> 操作系统恰好把每个进程放到哪个核上，因此逐次波动。上表里各列的高低应理解为
> "三者在同一区间"，而不是固定排名——见下面 [standalone vs client](#standalone-vs-client)
> 的重复测试，顺序逐次翻转。

## standalone vs client

`portunus-standalone`（由静态 TOML 文件驱动，没有控制面）与 `portunus-client`
（额外接一条双向 gRPC 控制流，用来接收下发的规则）**端到端共用同一份
`portunus-forwarder` 数据面**——同一条 `proxy.rs` / `splice` 热路径。规则一旦激活、
字节开始流动，转发代码就是逐字节一致的。

为了把真实差异和 loopback 噪声区分开，我们做了六次背对背的 uncapped A/B 重复
（`--seconds 10 --omit-seconds 2`），再加上 30 秒头条那一次，共 7 对：

| 轮次 | `portunus-standalone` | `portunus-client` | standalone / client |
| ---: | ---: | ---: | ---: |
| headline (30s) | 31,936 | 26,644 | 119.9% |
| rep 1 | 32,528 | 35,507 | 91.6% |
| rep 2 | 26,250 | 34,782 | 75.5% |
| rep 3 | 14,616 | 27,311 | 53.5% |
| rep 4 | 35,801 | 24,801 | 144.4% |
| rep 5 | 35,025 | 32,886 | 106.5% |
| rep 6 | 27,525 | 27,086 | 101.6% |
| **中位数** | **~30,027** | **~30,099** | **~100%** |

standalone 对 client 的比值在 **53.5% 到 144.4%** 之间摆动——有时 standalone 赢、
有时 client 赢——但两者的中位数相差仅约 0.2%。**两个构建之间不存在系统性的数据面
吞吐差异。** 这种大幅摆动只是 ~30 Gbit/s 的 uncapped 峰值受调度与核分配的运气支配，
与被测的是哪个二进制无关。

干净、低噪声的证据是下面的 [offered-load sweep](#offered-load-sweep)：在 100 Mbit/s
到 20 Gbit/s 的每一个固定限速点上，standalone 与 client 彼此、以及与 `iperf3` /
iptables 基线之间，差距都在 1% 以内。

唯一的结构性差异是：`portunus-client` 还会运行 gRPC 控制流、并周期性上报统计。在
CPU 饱和的机器上这是微乎其微的后台开销——低于数据面测量的噪声水平。**就纯转发吞吐而言，
按你的部署模型选构建即可（中央控制面 vs 静态 TOML），这不是一个性能决策。**

## Offered-load sweep

裸峰值回答的是"它最快能跑多少？"，而更有用的运维问题是"在*我的*链路速度下，转发器
会不会碍事？"。为此我们用 `iperf3 -b <rate>`，它会**把发送端限速到一个固定速率**
（offered load，模拟一条该速度的真实 WAN/VPS 链路），再看每条路径能否打满它
（`--seconds 5 --omit-seconds 1`）。同样 11 个采样点，四条路径背对背：

| Offered 带宽 | `iperf3` | iptables REDIRECT | `portunus-standalone` | `portunus-client` |
| ---: | ---: | ---: | ---: | ---: |
| 100 Mbit/s | 100.02 | 100.02 | 100.02 | 100.02 |
| 500 Mbit/s | 499.82 | 499.92 | 499.90 | 500.05 |
| 1 Gbit/s | 999.94 | 999.83 | 999.82 | 999.78 |
| 2.5 Gbit/s | 2,499.94 | 2,499.75 | 2,499.89 | 2,499.75 |
| 5 Gbit/s | 4,999.57 | 4,999.71 | 4,999.48 | 4,999.68 |
| 7.5 Gbit/s | 7,499.46 | 7,499.55 | 7,499.48 | 7,499.30 |
| 10 Gbit/s | 9,998.99 | 9,999.29 | 9,999.61 | 9,999.33 |
| 12.5 Gbit/s | 12,499.27 | 12,499.22 | 12,499.17 | 12,501.03 |
| 15 Gbit/s | 14,999.09 | 14,997.84 | 14,999.13 | 14,999.24 |
| 18 Gbit/s | 17,846.25 | 18,001.85 | 18,013.34 | 18,030.70 |
| 20 Gbit/s | 19,998.82 | 19,924.28 | 19,481.14 | 20,180.43 |

在**整个 100 Mbit/s → 20 Gbit/s 区间，四条路径都打满了 offered 速率**，差异都在
iperf3 短样本测量噪声之内。splice 快路径让两个 Portunus 构建在整段 sweep 上都与
内核态 `iptables` REDIRECT 基线持平——在这台 4 vCPU 机器上，不存在哪个链路速度会让
更丰富的控制面付出可测量的吞吐代价。18-20 Gbit/s 行里的轻微抖动（如 20 Gbit/s 时
standalone 19,481 vs client 20,180）是短跑 pacing 噪声，不是排名——和上面 uncapped
峰值的摆动是同一种噪声。

这正是 splice 之前的
[v0.11 基线](/zh/docs/getting-started/performance-history)
守不住的区间：在那里，Portunus 在 12.5 Gbit/s 以上跌到 iptables 的 ~33-49%。splice
快路径（v1.3.0 引入）抹平了这道差距，v1.6.1 又在同一台机器上再次确认。

## splice 之外保持不变

- 每连接的建立延迟、半关闭语义、字节计数器、Prometheus 指标、审计事件、RBAC、限速，
  全都不受影响。**capped（已限速）规则继续走原来的用户态路径，与 v1.2.0 逐字节
  一致。** splice 是否启用按连接判定，需要同时满足
  `Linux && TCP && !PORTUNUS_DISABLE_SPLICE && 规则与 owner 都没设带宽 cap`；
  `concurrent_connections` 与 `new_connections_per_sec` 限制在 accept（接受连接）
  时检查，**不会** 让 splice 失效。详见
  [限流与 QoS — 与 splice 快路径的交互](/zh/docs/features/rate-limiting#%E4%B8%8E-splice-%E5%BF%AB%E8%B7%AF%E5%BE%84%E7%9A%84%E4%BA%A4%E4%BA%92)。
- 跨平台行为不变：macOS / Windows 构建从不使用 splice（它用
  `#[cfg(target_os = "linux")]` 限定在 Linux；`nm` 确认 macOS release 二进制里零个
  splice 符号）。
- 运维表面不变：没有新增规则字段、没有线协议字段、没有 Web UI 控件、没有 CLI 开关。
  `PORTUNUS_DISABLE_SPLICE=1` 环境变量只用于故障排查和压测 A/B 对比（参见
  [故障排查 — 在排错时关闭 Linux 快路径](/zh/docs/operations/troubleshooting#%E5%9C%A8%E6%8E%92%E9%94%99%E6%97%B6%E5%85%B3%E9%97%AD-linux-%E5%BF%AB%E8%B7%AF%E5%BE%84)）。

## 测试方法

成熟的代理/转发性能报告会先把方法讲清楚，再给数字：

- 记录被测对象：commit、release/debug profile、CPU、OS、kernel、工具版本、关键 sysctl。
- 使用 release 二进制，热路径关闭 debug 日志。
- 先测 direct baseline，再在同一台机器上测 proxy path。
- 用预热 / 省略开头几秒，避免 TCP 慢启动和进程启动污染样本。
- 同时报告吞吐、建立 / RTT 延迟、连接行为，以及重传、拒绝/限流等指标。
- uncapped loopback 峰值用**区间**报告，不要用单个数字——几十 Gbit/s 量级的 loopback
  峰值会被调度噪声主导。
- CI 回归门和"对外宣传的绝对数字"分开。CI 负责抓漂移，固定硬件负责给可宣传的基线。

本仓库分三层测：

| 层级 | 命令 | 目的 |
| --- | --- | --- |
| Criterion TCP 数据面 | `cargo bench -p portunus-client --bench data_plane -- --quick` | 稳定的 loopback proxy 回归信号。 |
| Criterion UDP 数据面 | `cargo bench -p portunus-client --bench udp_data_plane -- --quick` | UDP 稳态与 RTT 回归信号。 |
| 真实进程对比 | `scripts/perf_compare.py` | 同机跑真实的 `portunus-server` + `portunus-client` + `portunus-standalone` + `iptables` + `iperf3`。 |

`scripts/perf_compare.py` 是 v1.6.1 的测试 harness，端到端驱动当前 CLI——
`server.toml` 的 `operator_token` 自举、operator HTTP `POST /v1/client-enrollments`
注册流程、`portunus-client enroll` + bundle + `push-rule`，以及一条 TOML 驱动的
`portunus-standalone` 规则——把 `iperf3`、`iptables` REDIRECT、standalone、client
四条路径背对背测一遍。（旧的 `scripts/perf_loopback.py` 早于 v1.6.1 的一次性
enrollment URI 流程和 SQLite `state.db` 文件锁，它的 `provision-client` / bundle
路径已经跑不通。）

## 解读

Portunus 是一个**用户态 L4 转发器**：它 accept 一个 TCP 连接，dial 目标，然后在两条
socket 之间搬字节。这让它比一条纯内核 NAT 规则能做的多得多，但它仍然要跨越内核不必
跨的那道用户态边界。

每条连接的代价取决于规则要求什么：

- **纯 TCP、无带宽 cap、在 Linux 上**：字节搬运走内核 `splice + pipe` 路径，负载根本
  不进用户态。这台机器上的 [offered-load sweep](#offered-load-sweep) 显示，两个
  Portunus 构建一路到 20 Gbit/s 都贴着 `iperf3` 与 `iptables` REDIRECT
  （落在 iperf3 噪声内）。
- **有任何带宽 cap、或 UDP、或 macOS / Windows、或显式禁用 splice**：走原来的用户态
  路径。这些情况下，内核 NAT 和 nftables flowtables 仍然是性能天花板——用户态的
  `read → memcpy → write` 循环带来实打实的每连接代价。

无论哪条路径，当你需要远端 client 注册、中央规则下发、RBAC、审计轨迹、per-owner QoS、
指标、SNI 路由、PROXY 协议和受管的规则生命周期时，Portunus 才是合适的工具。一条静态
DNAT 规则（固定的内核目标改写）这些都给不了你——所以只比裸吞吐数字就是问错了问题，
除非你确实只需要裸吞吐。

公平的基准测试不是"找几篇网上数字拼表"，而是：

1. 同一台机器，或同一组两机拓扑。
2. 同一个 target workload（TCP 用 `iperf3`，支持的话 UDP 也用）。
3. 同样的 duration、warm-up、socket buffer、CPU governor、MTU、kernel。
4. 先测 direct baseline。
5. 每次只启用一种转发实现。

## 复现流程

### 本机 / VPS 对比

```sh
# Debian/Ubuntu
sudo apt-get update
sudo apt-get install -y build-essential cmake pkg-config protobuf-compiler \
  git curl iperf3 iptables

curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh -s -- -y
source "$HOME/.cargo/env"

git clone https://github.com/ZingerLittleBee/Portunus.git
cd Portunus
PORTUNUS_SKIP_WEBUI=1 cargo build --release \
  -p portunus-standalone -p portunus-server -p portunus-client

# /tmp 可能是 tmpfs；portunus 拒绝在 tmpfs 上放 state.db。改用 $HOME。
export TMPDIR="$HOME"

# 头条：direct / iptables / standalone / client + 限流收敛。
# --with-iptables 仅 Linux，需要 root 或免密码 sudo。
python3 scripts/perf_compare.py --seconds 30 --omit-seconds 5 \
  --with-iptables --cap-bytes-per-sec 1048576 \
  --json-out perf-headline.json

# 四条路径的 offered-load sweep。
python3 scripts/perf_compare.py --seconds 5 --omit-seconds 1 \
  --with-iptables --cap-bytes-per-sec 0 \
  --offered-mbps 100,500,1000,2500,5000,7500,10000,12500,15000,18000,20000 \
  --json-out perf-sweep.json
```

常用开关：`--skip-client`（只测 standalone）、`--skip-standalone`（只测 client），
以及 `--server-bin` / `--client-bin` / `--standalone-bin` 指向预构建二进制。头条 JSON
形状如下：

```json
{
  "direct": { "mbps": 30889.985, "retransmits": 147 },
  "iptables_redirect": { "mbps": 26987.079, "retransmits": 16 },
  "iptables_vs_direct_pct": 87.37,
  "standalone_uncapped": { "mbps": 31935.997, "retransmits": 9 },
  "standalone_vs_direct_pct": 103.39,
  "standalone_vs_iptables_pct": 118.34,
  "client_uncapped": { "mbps": 26644.098, "retransmits": 4 },
  "client_vs_direct_pct": 86.25,
  "client_vs_iptables_pct": 98.73,
  "standalone_vs_client_pct": 119.86,
  "client_capped": {
    "cap_bytes_per_sec": 1048576, "mbps": 8.388,
    "target_mbps": 8.389, "within_plus_10pct": true
  }
}
```

两机测试时，把 `portunus-server` 放控制机，`portunus-client`（或
`portunus-standalone`）和 `iperf3 -s` 放 edge / target 机，然后从第三台执行
`iperf3 -c <edge-listen-ip> -p <listen_port> -t 30 -O 5 -J`。同一轮里务必再直连
target 跑一次 `iperf3` 作为 baseline。

### Criterion 回归

```sh
cargo bench -p portunus-client --bench data_plane
python3 scripts/bench_regression_gate.py --max-regression-pct 50

cargo bench -p portunus-client --bench udp_data_plane -- --quick
cargo bench -p portunus-server --bench operator_api -- --quick
```
