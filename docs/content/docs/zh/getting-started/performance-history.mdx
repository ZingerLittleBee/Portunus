## Linux iptables Baseline

采集时间：2026-05-10。

```text
Linux 69-201-77-68 6.12.74+deb13+1-amd64 #1 SMP PREEMPT_DYNAMIC Debian 6.12.74-2 (2026-03-08) x86_64
rustc 1.95.0 (59807616e 2026-04-14)
iperf 3.18
```

命令：

```sh
TMPDIR=/root scripts/perf_loopback.py --seconds 10 --omit-seconds 2 \
  --with-iptables --iptables-bin /usr/sbin/iptables
```

这台机器的 `/tmp` 是 `tmpfs`，而 `portunus-server` 会拒绝把 TLS / persistence
状态放在 tmpfs 上，所以这里用 `TMPDIR=/root` 把临时 config / data 目录放到
ext4。脚本结束后已清理 iptables 规则；`iptables -t nat -S OUTPUT` 只剩默认
`-P OUTPUT ACCEPT` policy。

| Path | Throughput | vs direct | vs iptables | Retransmits |
| --- | ---: | ---: | ---: | ---: |
| Direct `iperf3` loopback | 18,362-18,770 Mbit/s | 100.00% | 98-100% | 0 |
| `iptables` `nat/OUTPUT REDIRECT` | 18,429-19,190 Mbit/s | 100-102% | 100.00% | 0 |
| 通过 Portunus uncapped rule | 7,273-9,844 Mbit/s | 39-53% | 39-53% | 0-21 |
| Portunus 限制为 `1,048,576 B/s` | 8.387 Mbit/s | n/a | n/a | 0 |

限流目标是 8.389 Mbit/s，实测落在 +10% 验收线内。
uncapped 极限值刻意写成范围：在这个速度上，结果会明显受 scheduler noise、
TCP pacing、用户态 copy 的 CPU placement 影响。这台机器的一次样本是
7,531.560 Mbit/s（iptables 的 39.25%），但重复运行落在 7,273 到
9,844 Mbit/s 之间。面向用户带宽判断时，下面的 offered-load sweep 更有用，
因为它更接近有限 WAN 链路，而不是裸 loopback 极限。

## Offered-Load Sweep

下面这组用 `iperf3 -b <rate>` 做 TCP application pacing，回答更贴近用户的问题：
“我的公网 / VPS 带宽就这么多，Portunus 和 iptables 到底有没有差？”

命令形状：

```sh
TMPDIR=/root scripts/perf_loopback.py --seconds 5 --omit-seconds 1 \
  --with-iptables --offered-mbps 100,500,1000,2500,5000,7500,10000,12500,15000,18000,20000 \
  --iptables-bin /usr/sbin/iptables
```

| Offered load | Direct | iptables REDIRECT | Portunus | Portunus / iptables | 解读 |
| ---: | ---: | ---: | ---: | ---: | --- |
| 100 Mbit/s | 100.012 | 100.007 | 99.999 | 99.99% | 没有实质差异。 |
| 500 Mbit/s | 499.986 | 500.011 | 500.138 | 100.03% | 没有实质差异。 |
| 1 Gbit/s | 999.850 | 999.737 | 999.694 | 100.00% | 没有实质差异。 |
| 2.5 Gbit/s | 2,499.613 | 2,499.853 | 2,499.678 | 99.99% | 没有实质差异。 |
| 5 Gbit/s | 4,999.610 | 4,999.626 | 5,351.013 | 107.03% | 达到目标；短样本 pacing 噪声主导。 |
| 7.5 Gbit/s | 7,471.440 | 7,499.448 | 8,172.746 | 108.98% | 达到目标；短样本 pacing 噪声主导。 |
| 10 Gbit/s | 10,035.228 | 9,999.445 | 9,422.687 | 94.23% | 开始有小差距，但仍接近目标。 |
| 12.5 Gbit/s | 12,753.687 | 13,198.878 | 6,528.820 | 49.46% | 差距打开；Portunus 受 CPU / 用户态 copy 限制。 |
| 15 Gbit/s | 15,482.215 | 16,335.550 | 7,836.651 | 47.97% | 明显差距。 |
| 18 Gbit/s | 18,228.201 | 17,948.877 | 7,919.914 | 44.12% | 明显差距。 |
| 20 Gbit/s | 18,600.141 | 18,265.199 | 6,105.970 | 33.43% | 超出这台机器上 Portunus 的舒适区。 |

这组数据应该按区间理解，不要当成所有机器的固定阈值：

- **≤ 2.5 Gbit/s**：这台机器上 Portunus 和 iptables 基本一样。用户如果是
  100M、500M、1G、2.5G 的 WAN / VPS 端口，转发器通常不是瓶颈。
- **5-10 Gbit/s**：Portunus 还能接近目标速率，但已经进入"最好在自己机器上实测"的区域。
- **≥ 12.5 Gbit/s**：iptables 仍接近 offered load，Portunus 明显更早饱和。
  这个区间里，Portunus 的控制面、QoS、审计、metrics 等能力会有实打实的吞吐代价。

> ⚠ "12.5 G+ Portunus 早期饱和" 这条结论是
> [v1.3.0 splice 快路径](/zh/docs/getting-started/performance#v130-%E5%A4%B4%E6%9D%A1linux-tcp-splice-%E5%BF%AB%E8%B7%AF%E5%BE%84)
> 为无带宽限制的 TCP 规则消除的瓶颈。

> 是当前运维参考；这页记录的是 splice 之前的 baseline。
