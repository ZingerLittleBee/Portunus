`scripts/install.sh` 是一个自包含的 **POSIX sh** 脚本，覆盖 Portunus 节点的
完整生命周期：

```
install · uninstall · upgrade · status · service · config · env
```

它支持 `server`、`client`、`standalone` 三种角色,并同时管理两种部署形态——
**release 二进制 + 服务 unit** 与 **Docker Compose** ,界面支持中英双语
（English / 中文）。默认安装会同时**启动**服务;加 `--no-service` 则只放置
二进制。二进制形态下,脚本会按主机的 init 系统选择 —— **systemd** 或
**OpenRC**（Alpine）;两者都没有的主机仍会装好二进制并打印手动启动说明。
安装器**不会**替你写配置——配置由你自行创建（见 standalone 指南）,配置缺失
时服务会直接退出。对于 server,它还能为 Web UI 配置一个带自动 Let's Encrypt HTTPS
的 **Caddy** 反向代理。CI、自动化以及文档中 `curl … | sh` 一行命令所依赖的
非交互 flag 接口保持不变;交互式菜单只是新增的能力。

同一个脚本有两种使用方式：

```sh
# 始终最新的副本，管道方式（交互菜单或非交互 verb）：
curl -fsSL https://raw.githubusercontent.com/ZingerLittleBee/Portunus/main/scripts/install.sh | sh

# 已 checkout / 下载的本地副本，直接运行：
sh scripts/install.sh <verb> [flags]
```

<Callout type="info">
纯 **POSIX sh** —— 可在 `dash` 与 busybox `ash` 下运行,无需 `bash`。
`curl … | sh` 在任意发行版（含 Alpine/musl）都能用。
</Callout>

## 两种模式

**交互模式** —— 未给出可执行的 verb/role 且终端可用（stdin 是 TTY，或在管道
调用时 `/dev/tty` 可读）。此时进入菜单：

```text
Portunus Manager
  [1] Install
  [2] Uninstall
  [3] Upgrade
  [4] Status
  [5] Service (start/stop/restart)
  [6] Config
  [7] Env
  [0] Exit
Select [0-7]:
```

因为即使被管道调用也能从 `/dev/tty` 读取，所以 `curl … | sh` 同样能进入菜单。选 `1`
进入引导式安装向导：角色（`server` / `client` / `standalone`）→ 部署形态
→（仅 server）HTTPS 域名 → 通告地址 → 计划预览 → 确认。

**非交互模式** —— 出现任意 verb/role/可识别的 flag，或带 `--yes`，或没有可用
终端时进入。这是 CI 与自动化所依赖的契约，行为相当于在原下载器之上新增了若干 verb。

## 用法

```text
install.sh <client|server|standalone|install|uninstall|upgrade|status|service|config|env|domain>
           [start|stop|restart] [get|set <key> [value]] [<fqdn>]
           [--version V] [--deploy binary|docker]
           [--bin-dir DIR] [--compose-dir DIR]
           [--advertised-endpoint HOST:PORT] [--data-dir DIR]
           [--operator-http-listen ADDR]
           [--config PATH] [--no-service]
           [--domain FQDN] [--acme-email ADDR] [--skip-dns-check]
           [--lang en|zh] [--reset-lang]
           [--yes] [--purge] [--dry-run]
```

裸角色是安装的简写：`client` ≡ `install client`，`server` ≡
`install server`，`standalone` ≡ `install standalone`。

## Verb

| Verb | 作用 |
| --- | --- |
| `install <client\|server\|standalone>` | 引导式或 flag 驱动安装。默认部署形态为 `binary`,且默认会启动服务（除非 `--no-service`）。 |
| `uninstall [client\|server\|standalone]` | 删除二进制 + 服务 unit（systemd unit/drop-in 或 OpenRC init.d/conf.d），或 `docker compose down`。会先确认。 |
| `upgrade [client\|server\|standalone]` | 解析最新版本，与记录版本比较，复用记录的配置。已是最新则空操作。 |
| `status` | 记录的元数据 + 实时探测（`systemctl is-active` / `rc-service status` / `docker compose ps`）。只读。 |
| `service <start\|stop\|restart>` | 按记录的 init / 部署形态分发到 `systemctl`、`rc-service` 或 `docker compose`。 |
| `config <get\|set> <key> [value]` | 读写单个限定键（见下）。 |
| `env` | 打印全部限定键及其持久化值。 |
| `domain <fqdn>` | 仅 server。为已有安装配置 / 刷新 Caddy HTTPS 前端（见 [域名与 HTTPS](#域名与-https)）。 |

## Flag

| Flag | 含义 |
| --- | --- |
| `--version <X.Y.Z\|vX.Y.Z>` | 固定某个 release。省略 ⇒ 运行时解析最新。 |
| `--deploy <binary\|docker>` | 部署形态。默认 `binary`。 |
| `--bin-dir <DIR>` | 二进制安装目录（默认 `/usr/local/bin`）。 |
| `--compose-dir <DIR>` | Docker compose 目录（默认：当前目录）。 |
| `--advertised-endpoint <HOST:PORT>` | server 通告地址（见下）。留空 ⇒ 运行时自动。 |
| `--data-dir <DIR>` | server 数据目录（默认 `/var/lib/portunus`）。 |
| `--operator-http-listen <ADDR>` | Operator HTTP 监听地址覆写。 |
| `--config <PATH>` | **仅 standalone。** 服务读取的配置文件（默认 `/etc/portunus/standalone.toml`）。让 unit/conf.d 指向 `PATH` 并修正其权限——**文件由你创建**（缺失时服务直接退出）。对 `client` / `server` 传入会直接报错——它们用 `--bundle` / `--data-dir`。 |
| `--no-service` | 安装二进制（及服务 unit）但**不**启用/启动,并打印手动启动命令。 |
| `--domain <FQDN>` | 仅 server。为 Web UI 配置 Caddy HTTPS 前端；当未设置 `--advertised-endpoint` 时，还会把通告地址推导为 `<FQDN>:7443`。 |
| `--acme-email <ADDR>` | Caddy 域名的 ACME / Let's Encrypt 联系邮箱。 |
| `--skip-dns-check` | 跳过域名 DNS 是否解析到本机的预检查。 |
| `--systemd` | 已废弃的空操作,为向后兼容保留——现在服务默认就会安装。 |
| `--lang <en\|zh>` | 强制界面语言（也可用 `PORTUNUS_LANG`）。 |
| `--reset-lang` | 清除缓存的语言偏好并退出；下次交互运行会重新询问。 |
| `--yes` | 普通确认默认为是。**不会**绕过 `--purge` 的输入令牌。 |
| `--purge` | 配合 `uninstall`，额外删除数据目录 / compose 卷。需键入确认令牌。 |
| `--dry-run` | 打印计划；**不**联网、**不**写文件。 |

`--version`、`--bin-dir`、`--yes`、`--dry-run` 以及裸
`client` / `server` 形式是原下载器接口，行为与之前完全一致。`--systemd`
仍被接受,但已不再需要。

## 部署形态

### 二进制 + 服务（systemd / OpenRC）

```sh
# 二进制 + 服务,并启动（默认）
curl -fsSL .../scripts/install.sh | sudo sh -s -- server \
  --advertised-endpoint host.example:7443
# 仅二进制,不启动
curl -fsSL .../scripts/install.sh | sudo sh -s -- server --no-service --version 1.4.2
```

下载 release 归档,校验其 SHA-256,并将 `portunus-<role>` 安装到
`--bin-dir`。随后探测主机 init 并安装服务:在 **systemd** 上是加固 unit
（对 server 还会在 `/etc/systemd/system/portunus-server.service.d/10-portunus.conf`
写入含通告地址的 **drop-in**）;在 **OpenRC** 上是 `/etc/init.d` 里的
`supervise-daemon` 脚本,其参数放在 `/etc/conf.d`。两种情况都会创建
`portunus` / `portunus-server` / `portunus-client` 服务用户与目录,然后
启用并启动服务——除非带 `--no-service`,此时放置好 unit 后即停止。两者都
没有的主机会装好二进制并打印手动启动说明。出厂 unit 不会被改写,server
配置走 drop-in / conf.d。

对 **standalone**,安装器不会替你写配置;请先创建它（默认
`/etc/portunus/standalone.toml`,或 `--config` 指定的路径）。只有该文件存在
时服务才会自动启动——若裸装且没有配置,则只放置 unit 并打印如何创建配置、
如何启动。无配置时直接启动,二进制会以退出码 2 退出。

### Docker Compose

```sh
mkdir -p ~/portunus && cd ~/portunus
curl -fsSL .../scripts/install.sh | sh -s -- server --deploy docker \
  --compose-dir ~/portunus --advertised-endpoint host.example:7443
```

写入 `compose.yml`（若已存在则保留）与同级 `.env`（保存限定变量），然后执行
`docker compose pull && up -d`。需要 Docker Compose **v2**（也能识别旧版
`docker-compose` ，但优先使用 v2）。server 配置走 `.env` 旁挂文件，
已存在的 compose 文件不会被改写。

## 安装元数据

每次安装会记录一个 `.install-meta` 文件（shell `key=value`），供后续
子命令判断如何动作：

| 部署形态 / 角色 | 路径 |
| --- | --- |
| 二进制 server | `<data-dir>/.install-meta`（默认 `/var/lib/portunus/.install-meta`） |
| 二进制 client / standalone | `/etc/portunus/.install-meta` |
| docker | `<compose-dir>/.install-meta` |

字段包括 `role`、`deploy`、`version`、`lang`、`init`
（`systemd` / `openrc` / `none`）、`advertised_endpoint_set`、`domain`、
`installed_at`、`installer_version`。生命周期 verb 会读回 `init` 以选择
正确的服务管理器。

生命周期 verb 按此顺序解析元数据：显式 `--compose-dir`、然后**当前
目录**、然后 `/var/lib/portunus`、最后 `/etc/portunus`。对 Docker
部署，请**在 compose 目录中**运行生命周期命令（或传 `--compose-dir`），
以免被旧的二进制安装抢先匹配。

## 限定 config 与 env

`config get/set` 不适用于 `standalone` 角色,请直接编辑其配置文件
（默认 `/etc/portunus/standalone.toml`,或 `--config` 指定的路径）。
对 `server` 与 `client`，`config` 与
`env` 仅接受以下四个键（传入其它键会直接报错）：

| 键 | 映射的环境变量 |
| --- | --- |
| `advertised-endpoint` | `PORTUNUS_ADVERTISED_ENDPOINT` |
| `data-dir` | `PORTUNUS_DATA_DIR` |
| `operator-http-listen` | `PORTUNUS_OPERATOR_HTTP_LISTEN` |
| `version-pin` | `PORTUNUS_VERSION_PIN` |

`set` 把值写入 systemd drop-in（二进制）或 compose `.env`（docker），
并**合并保留**其它键，随后询问是否重启服务。`get` 打印持久化的值
（未设置时显示 `<unset>`）。`env` 打印全部四个键。

```sh
# 二进制 server（任意目录运行——meta 在 /var/lib/portunus）
sudo sh install.sh config set advertised-endpoint host.example:7443
sh install.sh config get advertised-endpoint
sh install.sh env

# docker（在 compose 目录中运行，或传 --compose-dir）
cd ~/portunus
sh install.sh config set operator-http-listen 0.0.0.0:7080
sh install.sh env
```

## server 通告地址

安装 **server** 时，向导会提示填写通告地址——也就是 Client 连接的公网
`host:port`。server 证书 SAN 必须覆盖该 host。留空则让 server 在运行
时自动解析。安装器只做轻量的 `host:port` 合法性检查，并把值持久化
（systemd drop-in 或 compose `.env` 中的 `PORTUNUS_ADVERTISED_ENDPOINT`），
权威的 SAN / 语法校验由 server 在启动时完成。

## 域名与 HTTPS

仅 server。当你提供 `--domain`（或在向导的 HTTPS 提示中作答）时，安装器
会用 **Caddy** 加自动 Let's Encrypt HTTPS 为 Web UI 提供前端：

```sh
# 安装时
curl -fsSL .../scripts/install.sh | sudo sh -s -- server \
  --domain portunus.example.com --acme-email admin@example.com

# 或为已有的 server 安装附加 / 刷新 HTTPS
sudo sh install.sh domain portunus.example.com
```

它的行为：

- 预检查域名的 DNS A 记录是否解析到本机（用 `--skip-dns-check` 跳过）。
- 若未安装 Caddy 则先安装（apt / dnf / yum），随后向
  `/etc/caddy/Caddyfile` 写入一个以 `# >>> portunus >>>` /
  `# <<< portunus <<<` 为界的受管区块，将该域名反向代理到 operator HTTP 端口。已存在
  的 Caddyfile 会先备份，且只重写受管区块。
- `--acme-email` 用于设置证书签发的 ACME 联系邮箱。
- 重载 Caddy 并轮询 `https://<domain>/` 直到签发完成（通常约 30 秒）。

提供 `--domain` 时，若未设置 `--advertised-endpoint`，还会把通告地址推导
为 `<domain>:7443`。一旦 `domain` 重新对齐了通告地址，server 会在重启时刷新
其 gRPC 证书 SAN，此时**已有的 client bundle 必须重新签发**
（`portunus-server enroll-client <name>`）。

这样一来 Web UI 可经 HTTPS 公开访问，但仍受运维登录 / 令牌保护。

## 生命周期示例

```sh
# 状态（只读）：记录的 meta + 实时探测
sh install.sh status

# 服务控制（分发到 systemctl 或 docker compose）
sudo sh install.sh service restart
sh install.sh service stop

# 原地升级，复用记录的配置（已是最新则空操作）
sudo sh install.sh upgrade

# 卸载（保留数据）——会先确认
sudo sh install.sh uninstall

# 卸载并删除数据——需键入令牌 'purge'
sudo sh install.sh uninstall --purge
```

## 语言

解析顺序：`--lang` / `PORTUNUS_LANG` → `LC_ALL` / `LANG` 自动探测
（`zh*` ⇒ 中文，否则英文）→ 交互模式下首次运行时询问 → 否则用英文。所选语言
会存入 `.install-meta`，供后续子命令复用。

```sh
PORTUNUS_LANG=zh curl -fsSL .../scripts/install.sh | sh
sh install.sh --lang en status
```

## 安全

- **`--dry-run`** 打印计划，对每个 verb 都**不**联网、**不**写文件。
- **`--purge`** 额外删除数据目录 / compose 卷，并要求你键入字面令牌
  `purge`。**`--yes` 永不绕过该输入令牌。**
- 破坏性操作都会先确认；`uninstall` 默认选项为 `[y/N]`。
- 出厂 systemd unit 与已存在的基础 compose 文件都不会被改写，server
  配置通过 drop-in / `.env` 旁挂文件下发。
- 在已有节点上重复执行 `install` 会通过 `.install-meta` 被识别出来；想在复用
  原有配置的同时切换版本，请用 `upgrade`。
