# Portunus

面向真实运维场景的远程 TCP / UDP 转发。边缘 Client 运行在远端主机上，Server
集中下发转发规则；数据面保持 L4 字节直通，同时补上 RBAC、指标、审计日志和
流量配额。

## 性能

大多数用户并不是要榨干本机 20 Gbit/s loopback 的极限吞吐，而是在有限带宽下
稳定转发。实测 Linux 主机上，`Portunus` 在 2.5 Gbit/s 以内与内核态转发基线
基本无差异；5-10 Gbit/s 仍然接近目标值，但建议在目标机器上复测；超过
10 Gbit/s 后，用户态控制面带来的吞吐成本开始明显。

| 目标带宽 | 结论 |
| --- | --- |
| 100 Mbit/s - 2.5 Gbit/s | 实测与内核态转发没有有意义的差距。 |
| 5-10 Gbit/s | 通常仍接近目标值，但上线前应该在目标机器复测。 |
| 12.5 Gbit/s+ | 用户态转发的控制能力会开始换掉一部分极限吞吐。 |

## 它提供什么

| 领域 | 能力 |
| --- | --- |
| 转发 | TCP / UDP 规则、端口范围、DNS 域名目标、多目标故障切换。 |
| L4 路由 | TLS SNI 路由但不终结 TLS，可选 PROXY protocol。 |
| 控制面 | CLI、操作员 HTTP API、内置 Web UI、基于 TLS 指纹固定的热更新规则流。 |
| 多租户 | 按用户、Client、协议与端口范围做 RBAC，规则归属由 Server 强制执行。 |
| QoS | 按规则或 owner 设置带宽、新连接速率、并发连接 / flow 上限。 |
| 运维 | Prometheus 指标、结构化日志、审计记录、内置 SQLite、备份与恢复。 |

## 适合什么场景

当你需要集中管理远端监听端口、给不同用户分配可控范围、观察转发流量、或按 owner
设置配额时，`Portunus` 比一条静态本机转发规则更像一个可运维的系统。若场景
只是单机上的单条固定规则，并且唯一目标是极限吞吐，内核态转发路径仍然更简单、
更快。这里的关键是按你的目标带宽实测，而不是只看无上限 loopback 的极限值。
