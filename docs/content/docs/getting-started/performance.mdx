This report records the current v1.3.0 performance posture. Treat the numbers
as **same-host evidence**, not universal throughput claims. Forwarding
performance is dominated by kernel version, CPU frequency policy, NIC path,
socket buffer sizing, and whether the path stays in-kernel or crosses
userspace.

## v1.3.0 Headline (Linux TCP splice fast path)

On the dedicated Linux bench host (Debian 13, kernel 6.12.74-2, glibc 2.41,
same hardware as the
[v0.11 Linux iptables baseline](/en/docs/getting-started/performance-history#linux-iptables-baseline)),
the v1.3.0 splice fast path **more than doubles single-flow uncapped
throughput** on plain TCP rules with no per-rule and no per-owner bandwidth
cap:

| Mode | Direct iperf3 loopback | Portunus uncapped | Portunus vs direct | Speedup vs v1.2.0 userspace |
| --- | ---: | ---: | ---: | ---: |
| splice OFF (`PORTUNUS_DISABLE_SPLICE=1`, v1.2.0 userspace) | 18,288 Mbit/s | 9,954 Mbit/s | 54.4% | (baseline) |
| splice ON (v1.3.0 default) | 18,820 Mbit/s | **21,922 Mbit/s** | **116.5%** | **2.20×** |

The splice ON case exceeds the direct `iperf3` loopback baseline because the
kernel-to-kernel `splice + pipe` path avoids the userspace `read → memcpy →
write` round-trip that bounds *every* userspace TCP application on this host
— including `iperf3` itself. The proxied path's three pipelines (iperf
client → portunus-client → iperf server) parallelise across CPUs in a way
the direct two-thread iperf3 baseline cannot.

### Offered-load sweep (v1.3.0 vs v0.11 saturation point)

The v0.11 report identified `≥ 12.5 Gbit/s` as the offered-load band where
Portunus saturates early and falls well behind iptables. The v1.3.0 splice
fast path closes that gap on this host. The table below is the same
`scripts/perf_loopback.py --with-iptables --offered-mbps …` shape used in
the v0.11 baseline — same 11-point sweep, same iptables REDIRECT
comparison, same host — captured under v1.3.0 with splice ON (default)
and splice OFF (`PORTUNUS_DISABLE_SPLICE=1`) back-to-back:

| Offered load | Direct | iptables REDIRECT | Portunus + splice OFF | Portunus + splice ON | splice speedup | splice OFF vs iptables | splice ON vs iptables |
| ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| 100 Mbit/s | 100 | 100 | 100 | 100 | 1.00× | 100.0% | 100.0% |
| 500 Mbit/s | 500 | 500 | 500 | 500 | 1.00× | 100.0% | 100.0% |
| 1 Gbit/s | 1,000 | 1,000 | 1,000 | 1,000 | 1.00× | 100.0% | 100.0% |
| 2.5 Gbit/s | 2,500 | 2,500 | 2,500 | 2,498 | 1.00× | 100.0% | 99.9% |
| 5 Gbit/s | 4,999 | 5,005 | 5,207 | 4,999 | 0.96× | 104.0% | 99.9% |
| 7.5 Gbit/s | 7,476 | 7,492 | 6,775 | 7,499 | 1.11× | 90.4% | 100.1% |
| 10 Gbit/s | 9,999 | 10,484 | 8,870 | 10,035 | 1.13× | 84.6% | 95.7% |
| **12.5 Gbit/s** | 12,598 | 12,545 | **8,556** | **12,373** | **1.45×** | **68.2%** | **98.6%** |
| **15 Gbit/s** | 15,569 | 15,523 | **9,953** | **16,370** | **1.64×** | **64.1%** | **105.5%** |
| **18 Gbit/s** | 18,406 | 18,017 | **8,347** | **19,618** | **2.35×** | **46.3%** | **108.9%** |
| **20 Gbit/s** | 19,088 | 18,738 | **8,690** | **20,147** | **2.32×** | **46.4%** | **107.5%** |

Read this against the v0.11 [offered-load table](/en/docs/getting-started/performance-history#offered-load-sweep):
at 12.5 Gbit/s offered, v0.11 landed 6,529 Mbit/s (49% of iptables) and the
v1.3.0 userspace path (splice OFF) lands 8,556 Mbit/s — still well below
iptables. v1.3.0 splice ON reaches **12,373 Mbit/s (98.6% of iptables)** —
effectively the offered target on this host. Through the entire 15-20 Gbit/s
band the splice ON path stays within iperf3 short-run noise of both the
direct loopback and the iptables REDIRECT baseline, while the splice OFF
userspace path saturates around the v1.2.0 userspace-memcpy ceiling of
~8.5-10 Gbit/s.

The 0.96× row at 5,000 Mbit/s is iperf3 short-run pacing noise (±200 Mbit/s
on this host), not a regression; splice OFF and splice ON track within ±5%
across the sub-10 Gbit/s band. The `splice ON vs iptables` column rising
above 100% in the 15-20 Gbit/s rows is the same parallelisation effect
called out under the headline table — the three-process pipeline can
saturate cores the two-process direct/iptables paths cannot.

### What does NOT change with splice

- Per-connection setup latency, half-close semantics, byte counters,
  Prometheus metrics, audit events, RBAC, rate limiting. **Capped
  rules remain on the canonical userspace path, byte-identical to
  v1.2.0.** splice eligibility is decided per connection by
  `Linux && TCP && !PORTUNUS_DISABLE_SPLICE && no bandwidth cap on
  rule or owner`; `concurrent_connections` and `new_connections_per_sec`
  are accept-time gates and do **not** disable splice. See
  [Rate Limiting & QoS — Interaction with the splice fast path](/en/docs/features/rate-limiting#interaction-with-the-splice-fast-path).
- Cross-platform behaviour: macOS / Windows builds are unchanged
  (splice is `#[cfg(target_os = "linux")]`-gated; `nm` confirms zero
  splice symbols in the darwin release binary).
- Operator surface: no rule field, no wire field, no Web UI control,
  no CLI flag. The `PORTUNUS_DISABLE_SPLICE=1` env var exists only
  for triage and bench A/B (see
  [Disabling the Linux fast path for triage](/en/docs/operations/troubleshooting#disabling-the-linux-fast-path-for-triage)).

The full test suite (`portunus-client` 218 tests including the 30
splice-specific ones) passes bit-identically with splice on AND
`PORTUNUS_DISABLE_SPLICE=1` — byte stability across every existing
rule shape is enforced as a release gate (012-tcp-zero-copy-splice
SC-003).

## Method

Mature proxy performance reports separate methodology from results:

- Record the system under test: commit, build profile, CPU, OS, kernel,
  tool versions, and relevant sysctls.
- Use release binaries. Disable debug logging on the hot path.
- Measure a direct baseline first, then the proxy path on the same host.
- Use warm-up / omitted seconds so TCP slow start and process startup do not
  dominate the sample.
- Report throughput, setup / RTT latency, connection behavior, and any
  retransmits or rejects.
- Keep regression gates separate from absolute marketing numbers. CI catches
  drift; dedicated hardware establishes product claims.

The repo therefore uses three layers:

| Layer | Command | Purpose |
| --- | --- | --- |
| Criterion TCP data plane | `cargo bench -p portunus-client --bench data_plane -- --quick` | Stable loopback proxy primitive regression signal. |
| Criterion UDP data plane | `cargo bench -p portunus-client --bench udp_data_plane -- --quick` | UDP steady-state and RTT regression signal. |
| Real process smoke | `scripts/perf_loopback.py` | Real `portunus-server` + `portunus-client` + `push-rule` + `iperf3`. |

`Portunus as percent of direct loopback` is only:

```text
portunus_uncapped / direct_iperf3_loopback * 100
```

It is not an iptables comparison. On Linux, add `--with-iptables` to run a
same-host `iptables -t nat OUTPUT ... REDIRECT` baseline alongside the
Portunus rule. That adds:

| Field | Meaning |
| --- | --- |
| `iptables_redirect` | `iperf3` through an in-kernel loopback port redirect. |
| `iptables_vs_direct_pct` | `iptables_redirect / direct * 100`. |
| `portunus_vs_iptables_pct` | `portunus_uncapped / iptables_redirect * 100`. |

The capped case targeted 8.389 Mbit/s and landed within the +10 percent
acceptance bound. The uncapped percent-of-direct improved from 18.04 % to
45.90 % after the v0.11 tuning recorded under
[v0.11 hot-path tuning](#v011-hot-path-tuning).

## Interpretation

Portunus is a userspace L4 forwarder. It accepts a TCP connection, dials
the target, and moves bytes between the two sockets. That makes it much
richer than plain kernel NAT, but it still has to cross userspace
boundaries the kernel doesn't.

How the per-flow cost lands depends on what the rule asks for:

- **Plain TCP, no bandwidth cap, Linux**: v1.3.0 routes the byte copy
  through a kernel `splice + pipe` pair. The bytes never enter userspace
  on `portunus-client`. On this host the [v1.3.0 sweep](#offered-load-sweep-v130-vs-v011-saturation-point)
  shows splice ON tracking both direct `iperf3` and `iptables` REDIRECT
  within iperf3 noise through 20 Gbit/s.
- **Any bandwidth cap, or UDP, or macOS / Windows, or splice disabled**:
  the canonical userspace path runs. Kernel NAT / nftables flowtables
  remain the performance ceiling for *this* shape — userspace
  `read → memcpy → write` adds real per-flow cost.

Either way, Portunus is the right tool when you need remote client
registration, rule push, RBAC, auditability, per-owner QoS, metrics,
SNI dispatch, PROXY protocol, and graceful rule lifecycle. A static
DNAT rule cannot give you any of these — comparing raw throughput
alone is the wrong axis unless that's literally all you need.

## v0.11 hot-path tuning

The macOS Criterion and Real Process Loopback numbers above reflect
three tuning changes that landed in the v0.11 hot path (`copy_bidirectional`
size, centralised `dial()`, dynamic throttling chunk). v1.3.0 does **not**
re-tune these knobs — splice runs in addition to them and only on Linux
TCP rules without bandwidth caps. Each item below is byte-stable (no
functional or wire change) and falls back to v0.10 behaviour when its
toggle is unset.

| Change | File | Effect |
| --- | --- | --- |
| Uncapped TCP copy buffer 8 KiB → 64 KiB | `crates/portunus-client/src/forwarder/proxy.rs` | `tokio::io::copy_bidirectional` → `copy_bidirectional_with_sizes(64 KiB, 64 KiB)`. Cuts read/write syscall count on bulk transfers; resident cost rises from 16 KiB to 128 KiB per hot connection. |
| Centralised `dial()` with `SO_SNDBUF` / `SO_RCVBUF` | `crates/portunus-client/src/resolver/mod.rs` | Outbound TCP dials go through one helper that creates a `TcpSocket` of the right family, sets `dial_send_buffer_bytes` / `dial_recv_buffer_bytes` (default 1 MiB each), then `connect`s. Loopback shows little gain; high-BDP WAN paths benefit. Best-effort — kernel may clamp. |
| Throttling copy chunk: fixed 16 KiB → dynamic | `crates/portunus-client/src/forwarder/rate_limit/copy.rs` | Per-iteration chunk = `clamp(tightest_cap_bps × 100 ms, 8 KiB, 64 KiB)`. High caps amortise like the uncapped fast path; low caps stay smooth (no large read followed by long sleep). The 10 % bandwidth-shaping accuracy gate (`t010_bandwidth_cap_shapes_to_target_rate_within_10pct`) still passes. |

What changed in the measurements:

* Criterion 1 MiB echo: ~995 µs → ~575 µs (≈ 1.7× throughput)
* Criterion 64 KiB echo: ~115 µs → ~90 µs (≈ 1.3× throughput)
* Criterion 1-byte RTT through proxy: 45 µs → 46 µs (within noise)
* Real process loopback uncapped: 14.2 Gbit/s → 36.8 Gbit/s
  (18.0 % → 45.9 % of direct loopback)
* Real process capped at 1 MiB/s: 8.388 Mbit/s → 8.394 Mbit/s
  (still inside the +10 % acceptance bound)

What did *not* change: per-connection latency floor, regression-gate
behaviour, RBAC / audit / per-owner QoS semantics, the wire format,
and the SQLite schema range. The added latency on a 1-byte
round-trip is unchanged within noise because `TCP_NODELAY` is still
set on both sides and the larger copy buffer is filled
opportunistically by the kernel.

Open follow-ups (not in this revision):

* CLI / config plumbing for `dial_send_buffer_bytes` /
  `dial_recv_buffer_bytes`. Defaults are reasonable; expose only
  if operators report a kernel ceiling clamp on a real link.

The "Linux-only `splice(2)` zero-copy fast path" item that lived here
in v0.11 shipped in v1.3.0 — see the
[v1.3.0 headline](#v130-headline-linux-tcp-splice-fast-path) at the
top of this page.

| Best fit | Centrally managed edge listeners with tenant-aware RBAC, per-owner QoS, observability, SNI dispatch, and PROXY protocol — TCP/UDP, single ports or ranges, IP or DNS targets. | Static local host/network NAT. | Modern Linux packet filtering and forwarding. | Generic TCP/UDP proxying. | L4/L7 load balancing where TCP is enough. | Service mesh or xDS-managed environments. | Simple port redirection baseline. | Experiments and debugging. | Operationally convenient encrypted tunnels. |

The fair benchmark is not "compare published numbers." It is:

1. Same host or same two-host topology.
2. Same target application (`iperf3` TCP, UDP where supported).
3. Same duration, warm-up, socket buffers, CPU governor, MTU, and kernel.
4. Direct baseline first.
5. One forwarding implementation at a time.

## Reproduction

### Local Smoke

```sh
# macOS
brew install iperf3

# Linux
sudo apt-get update
sudo apt-get install -y iperf3

PORTUNUS_SKIP_WEBUI=1 cargo build --release -p portunus-server -p portunus-client
scripts/perf_loopback.py --seconds 10 --omit-seconds 2 --json-out /tmp/portunus-perf.json

# Linux-only, requires root or passwordless sudo for iptables.
scripts/perf_loopback.py --seconds 10 --omit-seconds 2 --with-iptables \
  --json-out /tmp/portunus-perf.json
```

Expected output shape:

```json
{
  "direct": { "mbps": 80064.984, "retransmits": null },
  "portunus_uncapped": { "mbps": 36751.910, "retransmits": null },
  "portunus_vs_direct_pct": 45.9,
  "portunus_capped": {
    "cap_bytes_per_sec": 1048576,
    "mbps": 8.394,
    "target_mbps": 8.389,
    "within_plus_10pct": true
  }
}
```

With `--with-iptables` on Linux, the JSON also includes:

```json
{
  "iptables_redirect": { "mbps": 0.0, "retransmits": null },
  "iptables_vs_direct_pct": 0.0,
  "portunus_vs_iptables_pct": 0.0
}
```

```sh
cargo bench -p portunus-client --bench data_plane

cargo bench -p portunus-client --bench udp_data_plane -- --quick
```

### VPS Flow

Use SSH credentials out of band. Do not commit passwords or tokens.

```sh
ssh forward-bench@example-edge
sudo apt-get update
sudo apt-get install -y build-essential curl git iperf3 protobuf-compiler
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source "$HOME/.cargo/env"

git clone https://github.com/ZingerLittleBee/Portunus.git
cd Portunus
PORTUNUS_SKIP_WEBUI=1 cargo build --release -p portunus-server -p portunus-client
scripts/perf_loopback.py --seconds 30 --omit-seconds 5 --json-out perf-vps-loopback.json
```

For a two-host test, run `portunus-server` on the control host, run
`portunus-client` and `iperf3 -s` on the edge/target host, then drive
`iperf3 -c <edge-listen-ip> -p <listen_port> -t 30 -O 5 -J` from a
third host or from the control host. Keep direct `iperf3` to the target
as the baseline.
