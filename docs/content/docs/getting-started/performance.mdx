This report records the current **v1.6.1** performance posture. Treat the
numbers as **same-host evidence**, not universal throughput claims.
Forwarding performance is dominated by kernel version, CPU frequency
policy, NIC path, socket buffer sizing, and whether the path stays
in-kernel or crosses userspace.

The v1.6.1 data plane is unchanged from the v1.3.0 splice introduction
for TCP; this revision **re-measures on a fresh Linux host** and adds a
direct **`portunus-standalone` vs `portunus-client`** comparison, since
both ship the same `portunus-forwarder` data plane. The prior v1.3.0 and
v0.11 baselines are preserved on the
[Performance History](/en/docs/getting-started/performance-history) page.

## Bench host

All v1.6.1 numbers below were captured on:

```text
Linux ip-207-241-173-217 6.12.38+deb13-amd64 #1 SMP PREEMPT_DYNAMIC Debian 6.12.38-1 x86_64
AMD EPYC 7B13 (4 vCPU), 7.8 GiB RAM
rustc 1.96.0 (ac68faa20 2026-05-25)
iperf 3.18 (cJSON 1.7.15)
```

This is a 4-vCPU Debian 13 / kernel 6.12 host — the same OS/kernel family
as the [v0.11 / v1.3.0 bench host](/en/docs/getting-started/performance-history),
but different hardware, so **absolute numbers are not comparable across
the two hosts** — only same-host deltas are.

## v1.6.1 Headline (Linux TCP splice fast path)

Single 30-second uncapped TCP runs (`--seconds 30 --omit-seconds 5`),
splice ON (default), measured back-to-back against the same `iperf3`
target on loopback:

| Path | Throughput | vs direct | vs iptables |
| --- | ---: | ---: | ---: |
| Direct `iperf3` loopback | 30,890 Mbit/s | 100.0% | — |
| `iptables` `nat/OUTPUT REDIRECT` | 26,987 Mbit/s | 87.4% | 100.0% |
| **`portunus-standalone` uncapped** | **31,936 Mbit/s** | **103.4%** | **118.3%** |
| **`portunus-client` uncapped** | **26,644 Mbit/s** | **86.3%** | **98.7%** |
| Portunus capped at `1,048,576 B/s` | 8.388 Mbit/s | n/a | n/a |

With the splice fast path, Portunus tracks — and in some runs exceeds —
both the direct `iperf3` loopback and the in-kernel `iptables` REDIRECT
baseline. The kernel-to-kernel `splice + pipe` path avoids the userspace
`read → memcpy → write` round-trip that bounds *every* userspace TCP
application on this host (including `iperf3` itself), and the proxied
path's three pipelines (iperf client → Portunus → iperf server)
parallelise across the 4 vCPUs in a way the direct two-thread `iperf3`
baseline cannot. The capped case landed at 8.388 Mbit/s against an 8.389
Mbit/s target — inside the +10% acceptance bound.

> The single-run uncapped maximum at ~30 Gbit/s loopback is dominated by
> scheduler and core-placement noise. Read the headline row ordering as
> "all three are in the same band", not as a fixed ranking — see the
> [standalone vs client](#standalone-vs-client) repeats below, where the
> ordering flips run to run.

## standalone vs client

`portunus-standalone` (TOML-driven, no control plane) and
`portunus-client` (bidi gRPC control plane + rule push) consume the
**same `portunus-forwarder` data plane end-to-end** — the same
`proxy.rs` / `splice` hot path. Once a rule is active and bytes are
flowing, the forwarding code is identical.

To separate a real difference from loopback noise we ran six
back-to-back uncapped A/B repeats (`--seconds 10 --omit-seconds 2`),
plus the 30-second headline run (7 pairs total):

| Run | `portunus-standalone` | `portunus-client` | standalone / client |
| ---: | ---: | ---: | ---: |
| headline (30s) | 31,936 | 26,644 | 119.9% |
| rep 1 | 32,528 | 35,507 | 91.6% |
| rep 2 | 26,250 | 34,782 | 75.5% |
| rep 3 | 14,616 | 27,311 | 53.5% |
| rep 4 | 35,801 | 24,801 | 144.4% |
| rep 5 | 35,025 | 32,886 | 106.5% |
| rep 6 | 27,525 | 27,086 | 101.6% |
| **median** | **~30,027** | **~30,099** | **~100%** |

The standalone/client ratio swings from **53.5% to 144.4%** — sometimes
standalone wins, sometimes client wins — while the two medians land
within ~0.2% of each other. **There is no systematic data-plane
throughput difference between the two builds.** The wide swing is the
uncapped ceiling at ~30 Gbit/s loopback being dominated by scheduler and
core-placement noise, not by the binary under test.

The clean, low-noise proof is the [offered-load sweep](#offered-load-sweep)
below: at every fixed paced rate from 100 Mbit/s to 20 Gbit/s, standalone
and client both deliver the requested rate within < 1% of each other and
of direct / iptables.

The only structural difference: `portunus-client` additionally runs the
gRPC control-plane stream and periodic stats reporting. On a
CPU-saturated host that is marginal background overhead — below the
data-plane measurement noise floor. **For raw forwarding throughput,
pick the build that fits your deployment model (central control plane vs
static TOML); it is not a performance decision.**

The table below uses `iperf3 -b <rate>` TCP application pacing
(`--seconds 5 --omit-seconds 1`) to answer the operator question that
matters more than raw loopback maximums: "at my WAN speed, does the

back-to-back on the bench host:

| Offered load | Direct | iptables REDIRECT | `portunus-standalone` | `portunus-client` |
| ---: | ---: | ---: | ---: | ---: |
| 100 Mbit/s | 100.02 | 100.02 | 100.02 | 100.02 |
| 500 Mbit/s | 499.82 | 499.92 | 499.90 | 500.05 |
| 1 Gbit/s | 999.94 | 999.83 | 999.82 | 999.78 |
| 2.5 Gbit/s | 2,499.94 | 2,499.75 | 2,499.89 | 2,499.75 |
| 5 Gbit/s | 4,999.57 | 4,999.71 | 4,999.48 | 4,999.68 |
| 7.5 Gbit/s | 7,499.46 | 7,499.55 | 7,499.48 | 7,499.30 |
| 10 Gbit/s | 9,998.99 | 9,999.29 | 9,999.61 | 9,999.33 |
| 12.5 Gbit/s | 12,499.27 | 12,499.22 | 12,499.17 | 12,501.03 |
| 15 Gbit/s | 14,999.09 | 14,997.84 | 14,999.13 | 14,999.24 |
| 18 Gbit/s | 17,846.25 | 18,001.85 | 18,013.34 | 18,030.70 |
| 20 Gbit/s | 19,998.82 | 19,924.28 | 19,481.14 | 20,180.43 |

Through the **entire 100 Mbit/s → 20 Gbit/s band, all four paths hit the
offered rate within iperf3 short-run noise.** The splice fast path keeps
both Portunus builds level with the in-kernel `iptables` REDIRECT
baseline across the full sweep — there is no offered-load band on this
4-vCPU host where the richer control plane costs measurable throughput.
The slight dips/overshoots in the 18-20 Gbit/s rows (e.g. standalone
19,481 vs client 20,180 at 20 Gbit/s) are short-run pacing noise, not a
ranking — the same noise that drives the uncapped-max swing above.

This is the band that the pre-splice
[v0.11 baseline](/en/docs/getting-started/performance-history)
could not hold: there, Portunus fell to ~33-49% of iptables above
12.5 Gbit/s. The splice fast path (introduced in v1.3.0) closed it, and
v1.6.1 confirms it on fresh hardware.

## What does NOT change with splice

- Per-connection setup latency, half-close semantics, byte counters,
  Prometheus metrics, audit events, RBAC, rate limiting. **Capped
  rules remain on the canonical userspace path, byte-identical to
  v1.2.0.** splice eligibility is decided per connection by
  `Linux && TCP && !PORTUNUS_DISABLE_SPLICE && no bandwidth cap on
  rule or owner`; `concurrent_connections` and `new_connections_per_sec`
  are accept-time gates and do **not** disable splice. See
  [Rate Limiting & QoS — Interaction with the splice fast path](/en/docs/features/rate-limiting#interaction-with-the-splice-fast-path).
- Cross-platform behaviour: macOS / Windows builds are unchanged
  (splice is `#[cfg(target_os = "linux")]`-gated; `nm` confirms zero
  splice symbols in the darwin release binary).
- Operator surface: no rule field, no wire field, no Web UI control,
  no CLI flag. The `PORTUNUS_DISABLE_SPLICE=1` env var exists only
  for triage and bench A/B (see
  [Disabling the Linux fast path for triage](/en/docs/operations/troubleshooting#disabling-the-linux-fast-path-for-triage)).

## Method

Mature proxy performance reports separate methodology from results:

- Record the system under test: commit, build profile, CPU, OS, kernel,
  tool versions, and relevant sysctls.
- Use release binaries. Disable debug logging on the hot path.
- Measure a direct baseline first, then the proxy path on the same host.
- Use warm-up / omitted seconds so TCP slow start and process startup do
  not dominate the sample.
- Report throughput, setup / RTT latency, connection behavior, and any
  retransmits or rejects.
- Report the uncapped loopback maximum as a **range**, not a single
  number — at multi-10-Gbit/s loopback it is scheduler-noise-dominated.
- Keep regression gates separate from absolute marketing numbers. CI
  catches drift; dedicated hardware establishes product claims.

The repo uses three layers:

| Layer | Command | Purpose |
| --- | --- | --- |
| Criterion TCP data plane | `cargo bench -p portunus-client --bench data_plane -- --quick` | Stable loopback proxy primitive regression signal. |
| Criterion UDP data plane | `cargo bench -p portunus-client --bench udp_data_plane -- --quick` | UDP steady-state and RTT regression signal. |
| Real process compare | `scripts/perf_compare.py` | Real `portunus-server` + `portunus-client` + `portunus-standalone` + `iptables` + `iperf3` on one host. |

`scripts/perf_compare.py` is the v1.6.1 harness. It drives the current
CLI end-to-end: `server.toml` `operator_token` bootstrap, the operator
HTTP `POST /v1/client-enrollments` enrollment flow, `portunus-client
enroll` + bundle + `push-rule`, and a TOML-driven `portunus-standalone`
rule — measuring direct, `iptables` REDIRECT, standalone, and client
paths back-to-back. (The older `scripts/perf_loopback.py` predates the
v1.6.1 one-time enrollment URI flow and the SQLite `state.db` file lock,
so its `provision-client`/bundle path no longer runs.)

## Interpretation

Portunus is a userspace L4 forwarder. It accepts a TCP connection, dials
the target, and moves bytes between the two sockets. That makes it much
richer than plain kernel NAT, but it still has to cross userspace
boundaries the kernel doesn't.

How the per-flow cost lands depends on what the rule asks for:

- **Plain TCP, no bandwidth cap, Linux**: the byte copy runs through a
  kernel `splice + pipe` pair. The bytes never enter userspace on the
  forwarder. On this host the [offered-load sweep](#offered-load-sweep)
  shows both Portunus builds tracking direct `iperf3` and `iptables`
  REDIRECT within iperf3 noise through 20 Gbit/s.
- **Any bandwidth cap, or UDP, or macOS / Windows, or splice disabled**:
  the canonical userspace path runs. Kernel NAT / nftables flowtables
  remain the performance ceiling for *this* shape — userspace
  `read → memcpy → write` adds real per-flow cost.

Either way, Portunus is the right tool when you need remote client
registration, rule push, RBAC, auditability, per-owner QoS, metrics,
SNI dispatch, PROXY protocol, and graceful rule lifecycle. A static
DNAT rule cannot give you any of these — comparing raw throughput
alone is the wrong axis unless that's literally all you need.

The fair benchmark is not "compare published numbers." It is:

1. Same host or same two-host topology.
2. Same target application (`iperf3` TCP, UDP where supported).
3. Same duration, warm-up, socket buffers, CPU governor, MTU, and kernel.
4. Direct baseline first.
5. One forwarding implementation at a time.

## Reproduction

### Local / VPS compare

```sh
# Debian/Ubuntu
sudo apt-get update
sudo apt-get install -y build-essential cmake pkg-config protobuf-compiler \
  git curl iperf3 iptables

curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh -s -- -y
source "$HOME/.cargo/env"

git clone https://github.com/ZingerLittleBee/Portunus.git
cd Portunus
PORTUNUS_SKIP_WEBUI=1 cargo build --release \
  -p portunus-standalone -p portunus-server -p portunus-client

# /tmp may be tmpfs; portunus refuses tmpfs-backed state.db. Use $HOME.
export TMPDIR="$HOME"

# Headline: direct / iptables / standalone / client + capped convergence.
# --with-iptables is Linux-only and needs root or passwordless sudo.
python3 scripts/perf_compare.py --seconds 30 --omit-seconds 5 \
  --with-iptables --cap-bytes-per-sec 1048576 \
  --json-out perf-headline.json

# Offered-load sweep across all four paths.
python3 scripts/perf_compare.py --seconds 5 --omit-seconds 1 \
  --with-iptables --cap-bytes-per-sec 0 \
  --offered-mbps 100,500,1000,2500,5000,7500,10000,12500,15000,18000,20000 \
  --json-out perf-sweep.json
```

Useful flags: `--skip-client` (standalone-only), `--skip-standalone`
(client-only), `--server-bin` / `--client-bin` / `--standalone-bin` to
point at prebuilt binaries. Output JSON shape (headline):

```json
{
  "direct": { "mbps": 30889.985, "retransmits": 147 },
  "iptables_redirect": { "mbps": 26987.079, "retransmits": 16 },
  "iptables_vs_direct_pct": 87.37,
  "standalone_uncapped": { "mbps": 31935.997, "retransmits": 9 },
  "standalone_vs_direct_pct": 103.39,
  "standalone_vs_iptables_pct": 118.34,
  "client_uncapped": { "mbps": 26644.098, "retransmits": 4 },
  "client_vs_direct_pct": 86.25,
  "client_vs_iptables_pct": 98.73,
  "standalone_vs_client_pct": 119.86,
  "client_capped": {
    "cap_bytes_per_sec": 1048576, "mbps": 8.388,
    "target_mbps": 8.389, "within_plus_10pct": true
  }
}
```

For a two-host test, run `portunus-server` on the control host, run
`portunus-client` (or `portunus-standalone`) and `iperf3 -s` on the
edge/target host, then drive `iperf3 -c <edge-listen-ip> -p <listen_port>
-t 30 -O 5 -J` from a third host. Keep direct `iperf3` to the target as
the baseline.
