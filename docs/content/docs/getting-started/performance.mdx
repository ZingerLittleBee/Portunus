This report records the current v1.3.0 performance posture. Treat the numbers
as **same-host evidence**, not universal throughput claims. Forwarding
performance is dominated by kernel version, CPU frequency policy, NIC path,
socket buffer sizing, and whether the path stays in-kernel or crosses
userspace.

## v1.3.0 Headline (Linux TCP splice fast path)

On the dedicated Linux bench host (Debian 13, kernel 6.12.74-2, glibc 2.41,
same hardware as the v0.11 baseline in [Linux iptables Baseline](#linux-iptables-baseline)
below), the v1.3.0 splice fast path **more than doubles single-flow uncapped
throughput** on plain TCP rules with no per-rule and no per-owner bandwidth
cap:

| Mode | Direct iperf3 loopback | Portunus uncapped | Portunus vs direct | Speedup vs v1.2.0 userspace |
| --- | ---: | ---: | ---: | ---: |
| splice OFF (`PORTUNUS_DISABLE_SPLICE=1`, v1.2.0 userspace) | 18,288 Mbit/s | 9,954 Mbit/s | 54.4% | (baseline) |
| splice ON (v1.3.0 default) | 18,820 Mbit/s | **21,922 Mbit/s** | **116.5%** | **2.20×** |

The splice ON case exceeds the direct `iperf3` loopback baseline because the
kernel-to-kernel `splice + pipe` path avoids the userspace `read → memcpy →
write` round-trip that bounds *every* userspace TCP application on this host
— including `iperf3` itself. The proxied path's three pipelines (iperf
client → portunus-client → iperf server) parallelise across CPUs in a way
the direct two-thread iperf3 baseline cannot.

### Offered-load sweep (v1.3.0 vs v0.11 saturation point)

The v0.11 report identified `≥ 12.5 Gbit/s` as the offered-load band where
Portunus saturates early and falls well behind iptables. v1.3.0 closes that
gap:

| Offered (Mbit/s) | Direct | Portunus + splice OFF | Portunus + splice ON | splice speedup |
| ---: | ---: | ---: | ---: | ---: |
| 100 | 100 | 100 | 100 | 1.00× |
| 500 | 500 | 500 | 500 | 1.00× |
| 1,000 | 1,000 | 1,000 | 1,000 | 1.00× |
| 2,500 | 2,500 | 2,499 | 2,498 | 1.00× |
| 5,000 | 4,999 | 5,000 | 5,000 | 1.00× |
| 7,500 | 7,499 | 7,996 | 7,500 | 0.94× |
| 10,000 | 10,215 | 10,401 | 9,999 | 0.96× |
| **12,500** | 13,345 | **9,701** | **12,499** | **1.29×** |
| **15,000** | 16,103 | **9,719** | **16,120** | **1.66×** |

Read this against the v0.11 [offered-load table](#offered-load-sweep): at
12.5 Gbit/s offered, v0.11 landed 6,529 Mbit/s (49% of iptables) and v1.3.0
splice OFF lands 9,701 Mbit/s (the v1.2.0 userspace tuning is still in
effect). v1.3.0 splice ON reaches **12,499 Mbit/s — effectively the offered
target** — closing the gap to iptables to within measurement noise on this
host. At 15 Gbit/s offered, splice ON reaches the requested rate while
both userspace paths are stuck around 9.7 Gbit/s, the userspace memcpy
ceiling.

The 0.94×/0.96× rows at 7.5G/10G are within iperf3 short-run pacing noise
(±200 Mbit/s on this host), not a regression; splice OFF and splice ON
are within ±5% across that band.

### What does NOT change with splice

- Per-connection setup latency, half-close semantics, byte counters,
  Prometheus metrics, audit events, RBAC, rate limiting (capped rules
  remain on the canonical userspace path, byte-identical to v1.1.0).
- Cross-platform behaviour: macOS / Windows builds are unchanged
  (splice is `#[cfg(target_os = "linux")]`-gated; `nm` confirms zero
  splice symbols in the darwin release binary).
- Operator surface: no rule field, no wire field, no Web UI control,
  no CLI flag. The `PORTUNUS_DISABLE_SPLICE=1` env var exists only
  for triage and bench A/B (see
  [Disabling the Linux fast path for triage](/en/docs/operations/troubleshooting#disabling-the-linux-fast-path-for-triage)).

The full test suite (`portunus-client` 218 tests including the 30
splice-specific ones) passes bit-identically with splice on AND
`PORTUNUS_DISABLE_SPLICE=1` — byte stability across every existing
rule shape is enforced as a release gate (012-tcp-zero-copy-splice
SC-003).

## Method

Mature proxy performance reports separate methodology from results:

- Record the system under test: commit, build profile, CPU, OS, kernel,
  tool versions, and relevant sysctls.
- Use release binaries. Disable debug logging on the hot path.
- Measure a direct baseline first, then the proxy path on the same host.
- Use warm-up / omitted seconds so TCP slow start and process startup do not
  dominate the sample.
- Report throughput, setup / RTT latency, connection behavior, and any
  retransmits or rejects.
- Keep regression gates separate from absolute marketing numbers. CI catches
  drift; dedicated hardware establishes product claims.

The repo therefore uses three layers:

| Layer | Command | Purpose |
| --- | --- | --- |
| Criterion TCP data plane | `cargo bench -p portunus-client --bench data_plane -- --quick` | Stable loopback proxy primitive regression signal. |
| Criterion UDP data plane | `cargo bench -p portunus-client --bench udp_data_plane -- --quick` | UDP steady-state and RTT regression signal. |
| Real process smoke | `scripts/perf_loopback.py` | Real `portunus-server` + `portunus-client` + `push-rule` + `iperf3`. |

Read this as bands, not a universal threshold:

- **≤ 2.5 Gbit/s**: Portunus and iptables are effectively identical on this

  forwarding choice is unlikely to be the bottleneck.
- **5-10 Gbit/s**: Portunus is near the requested rate, but the host is
  approaching the point where userspace copy cost can show up. Treat this as
  the "measure on your box" zone.
- **≥ 12.5 Gbit/s**: iptables stays near the offered load while Portunus
  saturates much earlier. In this range the richer control plane has a real
  throughput cost.

## Interpretation

Portunus is a userspace L4 forwarder. It accepts a TCP connection, dials
the target, then copies bytes through Tokio. That makes it much richer than
plain kernel NAT, but it necessarily crosses userspace and does more per-flow
work.

For raw same-host forwarding, kernel NAT / nftables flowtables should be the
performance ceiling. Portunus is the right tool when the operator needs
remote client registration, rule push, RBAC, auditability, per-owner QoS,
metrics, SNI dispatch, PROXY protocol, and graceful rule lifecycle.

## Recent Optimisations

The numbers above reflect three tuning changes landed in the v0.11
hot path. Each is byte-stable (no functional or wire change) and
falls back to v0.10 behaviour when its toggle is unset.

| Change | File | Effect |
| --- | --- | --- |
| Uncapped TCP copy buffer 8 KiB → 64 KiB | `crates/portunus-client/src/forwarder/proxy.rs` | `tokio::io::copy_bidirectional` → `copy_bidirectional_with_sizes(64 KiB, 64 KiB)`. Cuts read/write syscall count on bulk transfers; resident cost rises from 16 KiB to 128 KiB per hot connection. |
| Centralised `dial()` with `SO_SNDBUF` / `SO_RCVBUF` | `crates/portunus-client/src/resolver/mod.rs` | Outbound TCP dials go through one helper that creates a `TcpSocket` of the right family, sets `dial_send_buffer_bytes` / `dial_recv_buffer_bytes` (default 1 MiB each), then `connect`s. Loopback shows little gain; high-BDP WAN paths benefit. Best-effort — kernel may clamp. |
| Throttling copy chunk: fixed 16 KiB → dynamic | `crates/portunus-client/src/forwarder/rate_limit/copy.rs` | Per-iteration chunk = `clamp(tightest_cap_bps × 100 ms, 8 KiB, 64 KiB)`. High caps amortise like the uncapped fast path; low caps stay smooth (no large read followed by long sleep). The 10 % bandwidth-shaping accuracy gate (`t010_bandwidth_cap_shapes_to_target_rate_within_10pct`) still passes. |

What changed in the measurements:

* Criterion 1 MiB echo: ~995 µs → ~575 µs (≈ 1.7× throughput)
* Criterion 64 KiB echo: ~115 µs → ~90 µs (≈ 1.3× throughput)
* Criterion 1-byte RTT through proxy: 45 µs → 46 µs (within noise)
* Real process loopback uncapped: 14.2 Gbit/s → 36.8 Gbit/s
  (18.0 % → 45.9 % of direct loopback)
* Real process capped at 1 MiB/s: 8.388 Mbit/s → 8.394 Mbit/s
  (still inside the +10 % acceptance bound)

What did *not* change: per-connection latency floor, regression-gate
behaviour, RBAC / audit / per-owner QoS semantics, the wire format,
and the SQLite schema range. The added latency on a 1-byte
round-trip is unchanged within noise because `TCP_NODELAY` is still
set on both sides and the larger copy buffer is filled
opportunistically by the kernel.

Open follow-ups (not in this revision):

* Linux-only `splice(2)` zero-copy fast path on flows with no
  pre-read, no PROXY prelude, and no bandwidth cap. Would be a
  separate feature flag because it forks the data plane.
* CLI / config plumbing for `dial_send_buffer_bytes` /
  `dial_recv_buffer_bytes`. Defaults are reasonable; expose only
  if operators report a kernel ceiling clamp on a real link.

## Comparable Forwarders

| Dimension | `iptables` DNAT / MASQUERADE | `nftables` + flowtables | NGINX `stream` | HAProxy TCP mode | Envoy `tcp_proxy` | `rinetd` | `socat` | SSH `-L` / `-R` |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| Performance profile | Highest. Kernel path, no userspace copy. | Highest for eligible flows; bypasses parts of the classic forwarding path. | High userspace proxy with mature event loop. | Very high userspace TCP proxy with excellent connection handling. | High but heavier userspace proxy, designed for service mesh / xDS. | Lightweight userspace redirector. | Flexible diagnostic pipe, not tuned as a managed production proxy. | Encrypted tunnel; throughput pays SSH crypto overhead. |
| TCP | Yes | Yes | Yes | Yes | Yes | Yes | Yes | Yes |
| UDP | Yes | Yes | Yes | No generic UDP forwarding | Dedicated UDP filters / use cases | Yes | Yes | No native UDP |
| Dynamic remote rule push | No built-in control plane | No built-in control plane | Reload/API depends on edition/config | Runtime API supports many operations | xDS control plane | Config reload style | No | Per-session |
| RBAC / audit / metrics | External only | External only | Metrics via modules; no native tenant RBAC like Portunus | Strong stats, ACLs, stick tables | Rich telemetry and policy | Minimal | Minimal | SSH auth/logs |
| QoS / rate limit | Basic shaping via `tc` / nftables ecosystem | Via nftables / `tc`, not app-owner aware | `limit_conn` / `limit_rate` style controls | Rich connection/rate controls | Rich filters, overload manager | Minimal | Minimal | Minimal |
| Best fit | Static local host/network NAT. | Modern Linux packet filtering and forwarding. | Generic TCP/UDP proxying. | L4/L7 load balancing where TCP is enough. | Service mesh or xDS-managed environments. | Simple port redirection baseline. | Experiments and debugging. | Operationally convenient encrypted tunnels. |

The fair benchmark is not "compare published numbers." It is:

1. Same host or same two-host topology.
2. Same target application (`iperf3` TCP, UDP where supported).
3. Same duration, warm-up, socket buffers, CPU governor, MTU, and kernel.
4. Direct baseline first.
5. One forwarding implementation at a time.

## Reproduction

### Local Smoke

```sh
# macOS
brew install iperf3

# Linux
sudo apt-get update
sudo apt-get install -y iperf3

PORTUNUS_SKIP_WEBUI=1 cargo build --release -p portunus-server -p portunus-client
scripts/perf_loopback.py --seconds 10 --omit-seconds 2 --json-out /tmp/portunus-perf.json

# Linux-only, requires root or passwordless sudo for iptables.
scripts/perf_loopback.py --seconds 10 --omit-seconds 2 --with-iptables \
  --json-out /tmp/portunus-perf.json
```

Expected output shape:

```json
{
  "direct": { "mbps": 80064.984, "retransmits": null },
  "portunus_uncapped": { "mbps": 36751.910, "retransmits": null },

    "cap_bytes_per_sec": 1048576,
```

Use SSH credentials out of band. Do not commit passwords or tokens.

```sh

curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
```

For a two-host test, run `portunus-server` on the control host, run
`portunus-client` and `iperf3 -s` on the edge/target host, then drive
`iperf3 -c <edge-listen-ip> -p <listen_port> -t 30 -O 5 -J` from a
third host or from the control host. Keep direct `iperf3` to the target
as the baseline.
