`Portunus` has two runtime roles:

- `portunus-server` runs on the control-plane host. Operators talk to it, it
  stores state, and it pushes rules to connected clients.
- `portunus-client` runs on each edge host. It connects back to the server,
  binds forwarded ports, and moves traffic.

Install them on the hosts that will run those roles. In a real deployment that
usually means one server host and one or more client hosts.

## Prerequisites

- **Docker / Podman** for the recommended install path.
- **Rust 1.88+ stable** only when building from source. `protoc` is
  vendored via `prost-build`; no system install is required.
- **Two reachable hosts** for a real deployment. For local trials,
  one host with two terminals is enough.
- **Linux or macOS**. Linux is the supported production target.

## Quick Docker install

On a fresh Linux host, use Docker's official convenience script:

```sh
sudo curl -fsSL https://get.docker.com | sh
```

## Server install

Docker Compose is the recommended server install path.

### Copy-paste install on the server host

1. Create and enter the working directory:

```sh
mkdir -p ~/portunus-server && cd ~/portunus-server
```

2. Write `docker-compose.yml`:

```sh
cat > docker-compose.yml <<'EOF'
services:
  server:
    image: ghcr.io/zingerlittlebee/portunus-server:latest
    container_name: portunus-server
    network_mode: host
    volumes:
      - portunus-data:/var/lib/portunus
    restart: unless-stopped

volumes:
  portunus-data:
    name: portunus-data
EOF
```

3. Start the server and verify it:

```sh
docker compose up -d && docker compose logs -f -t
```

That is enough for a default server install. Generated TLS material and SQLite
state live in the `portunus-data` volume.

With host networking, the Web UI is available on the server host at
`http://127.0.0.1:7080/`. The client control-plane listener remains on
`0.0.0.0:7443`.

### Compose file reference

```yaml
services:
  server:
    image: ghcr.io/zingerlittlebee/portunus-server:latest
    container_name: portunus-server
    network_mode: host
    volumes:
      - portunus-data:/var/lib/portunus
    restart: unless-stopped

volumes:
  portunus-data:
    name: portunus-data
```
The examples use `:latest` by default; pin a version tag such as `:1.0.0`
when you need a fully repeatable production deploy.

If you need non-default listeners or other overrides, place an optional
`server.toml` at `<data-dir>/server.toml`, for example with
`./server.toml:/var/lib/portunus/server.toml:ro`.

See [Deploy with Docker](/en/docs/deployment/docker) for operator commands,
client Compose, backup, migration, and production notes.

For a host binary install, download the release archive for the server host,
extract it, and install only `portunus-server`:

```sh
tar -xzf portunus-1.0.0-x86_64-unknown-linux-gnu.tar.gz
sudo install -m 0755 portunus-1.0.0-x86_64-unknown-linux-gnu/portunus-server /usr/local/bin/
```

For a source build:

```sh
git clone https://github.com/ZingerLittleBee/Portunus.git
cd Portunus
cargo build --release -p portunus-server
sudo install -m 0755 target/release/portunus-server /usr/local/bin/
```

Verify on the server host:

```sh
portunus-server --version
```

## Client install

Docker Compose is also the recommended client install path.

### Copy-paste install on the edge host

First, use the server-side operator flow to provision a client bundle and copy
that bundle file to the edge host. The example below assumes the bundle file is
named `edge-01.bundle.json`.

1. Create and enter the working directory:

```sh
mkdir -p ~/portunus-client && cd ~/portunus-client
```

2. Place `edge-01.bundle.json` in this directory, then write
   `docker-compose.yml`:

```sh
cat > docker-compose.yml <<'EOF'
services:
  client:
    image: ghcr.io/zingerlittlebee/portunus-client:latest
    container_name: portunus-client
    network_mode: host
    volumes:
      - ./edge-01.bundle.json:/etc/portunus/client.bundle.json:ro
    restart: unless-stopped
EOF
```

3. Start the client and verify it:

```sh
docker compose up -d && docker compose logs -f -t
```

### Compose file reference

```yaml
services:
  client:
    image: ghcr.io/zingerlittlebee/portunus-client:latest
    container_name: portunus-client
    network_mode: host
    volumes:
      - ./edge-01.bundle.json:/etc/portunus/client.bundle.json:ro
    restart: unless-stopped
```

`--network host` is recommended for the client because forwarded listen ports
are created later by operator rules. If you avoid host networking, publish each
forwarded port explicitly.

For a host binary install, use the same release archive format and install only
`portunus-client` on the edge host:

```sh
tar -xzf portunus-1.0.0-x86_64-unknown-linux-gnu.tar.gz
sudo install -m 0755 portunus-1.0.0-x86_64-unknown-linux-gnu/portunus-client /usr/local/bin/
```

For a source build:

```sh
git clone https://github.com/ZingerLittleBee/Portunus.git
cd Portunus
cargo build --release -p portunus-client
sudo install -m 0755 target/release/portunus-client /usr/local/bin/
```

Verify on the client host:

```sh
portunus-client --version
```

Checksums are published alongside the archives:

```sh
curl -LO https://github.com/ZingerLittleBee/Portunus/releases/download/v1.0.0/portunus-1.0.0-checksums.txt
```

Extract the archive and copy both binaries into your `PATH`:

```sh
tar -xzf portunus-1.0.0-x86_64-unknown-linux-gnu.tar.gz
sudo install -m 0755 portunus-1.0.0-x86_64-unknown-linux-gnu/portunus-server /usr/local/bin/
sudo install -m 0755 portunus-1.0.0-x86_64-unknown-linux-gnu/portunus-client /usr/local/bin/
```

All release assets live at
[`github.com/ZingerLittleBee/Portunus/releases/tag/v1.0.0`](https://github.com/ZingerLittleBee/Portunus/releases/tag/v1.0.0).

## systemd

```sh
cd deploy/systemd
sudo ./install.sh
# Creates:
#   /var/lib/portunus/    state.db + generated TLS material + optional server.toml
#   /etc/portunus/        client bundle directory
#   portunus-server / portunus-client system users
```

The unit installer expects `portunus-server` and `portunus-client` to already
be on the host, usually copied from the release archive above.

See [Deploying with systemd](/en/docs/deployment/systemd) for the full unit
breakdown and hardening notes.
