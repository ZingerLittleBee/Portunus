Docker is the recommended deployment path for most installs. The release
pipeline publishes multi-arch Linux images to GitHub Container Registry:

```sh
docker pull ghcr.io/zingerlittlebee/portunus-server:1.0.0
docker pull ghcr.io/zingerlittlebee/portunus-client:1.0.0
```

Use `:1.0.0` for repeatable production deploys. `:latest` tracks the newest
stable release:

```sh
docker pull ghcr.io/zingerlittlebee/portunus-server:latest
docker pull ghcr.io/zingerlittlebee/portunus-client:latest
```

## Server

Create host directories first:

```sh
mkdir -p ./portunus-config ./portunus-state
```

Run the server:

```sh
docker run -d --name portunus-server \
  -p 7443:7443 \
  -v "$(pwd)/portunus-config:/etc/portunus" \
  -v "$(pwd)/portunus-state:/var/lib/portunus" \
  ghcr.io/zingerlittlebee/portunus-server:1.0.0
```

The image entrypoint is `portunus-server`; the default command is:

```sh
--config-dir /var/lib/portunus --config /etc/portunus/server.toml serve
```

If `/etc/portunus/server.toml` is absent, Portunus still starts with CLI /
compiled defaults and stores generated TLS material plus SQLite state under
`/var/lib/portunus`.

## Operator commands

The runtime image is distroless, so it has no shell. Run one-shot operator
commands with the same image, joined to the server container network and
volumes:

```sh
docker run --rm \
  --network container:portunus-server \
  --volumes-from portunus-server \
  ghcr.io/zingerlittlebee/portunus-server:1.0.0 \
  --config-dir /var/lib/portunus \
  --config /etc/portunus/server.toml \
  bootstrap-superadmin --name ops
```

Capture the printed token and pass it to later operator commands:

```sh
docker run --rm \
  --network container:portunus-server \
  --volumes-from portunus-server \
  -e PORTUNUS_OPERATOR_TOKEN="$PORTUNUS_OPERATOR_TOKEN" \
  ghcr.io/zingerlittlebee/portunus-server:1.0.0 \
  --config-dir /var/lib/portunus \
  --config /etc/portunus/server.toml \
  list-clients
```

## Client

Mount a provisioned client bundle into the expected path:

```sh
docker run -d --name portunus-client \
  --network host \
  -v "$(pwd)/edge-01.bundle.json:/etc/portunus/client.bundle.json:ro" \
  ghcr.io/zingerlittlebee/portunus-client:1.0.0
```

Host networking is the least surprising mode for `portunus-client`: pushed
rules bind listeners on the edge host, and Docker cannot know those ports
before the operator creates the rules. If you avoid host networking, publish
every forwarded listen port explicitly with `-p`.

## Local trial: docker-compose

```sh
cd deploy/docker
docker compose up -d server
```

The compose file is **local-only** — meant for kicking the tires, not
for production. It wires a single server + optional client with loopback paths
and uses the runtime-only Dockerfiles, so local builds still require staged
binaries in `docker-bin/<arch>/`.

```sh
cargo build --release -p portunus-server -p portunus-client
arch="$(docker version --format '{{.Server.Arch}}')"
mkdir -p "docker-bin/${arch}"
cp target/release/portunus-server target/release/portunus-client "docker-bin/${arch}/"
docker compose -f deploy/docker/docker-compose.yml build
```

## Volumes & paths

The default container layout follows the FHS pattern from the systemd
unit:

| Host path | Container path | Purpose |
| --- | --- | --- |
| `./portunus-config/` | `/etc/portunus/` | TLS material, optional `server.toml`, client bundles |
| `./portunus-state/` | `/var/lib/portunus/` | `state.db` and sidecar files |

## Considerations

- **Distroless runtime** has no shell. Use `docker compose exec` only
  if your compose file specifies an exec'able binary; for ad-hoc
  inspection, use a sibling container with `--volumes-from` or pull
  artefacts to the host.
- **Loopback bind for the operator API** is enforced inside the
  container; expose it to your workstation via SSH tunnel or run the
  operator CLI inside the container.
- **Filesystem class check** still fires inside containers — make sure
  your bind-mount source is a real local filesystem (not tmpfs).
- **`portunus-client`** needs the bundle present at
  `/etc/portunus/client.bundle.json` (or `PORTUNUS_CLIENT_BUNDLE`).

## Health checks

A simple `HEALTHCHECK` for the server:

```dockerfile
HEALTHCHECK --interval=30s --timeout=5s --start-period=5s --retries=3 \
  CMD ["portunus-server", "list-clients"]
```

(This requires `PORTUNUS_OPERATOR_TOKEN` to be in the container's
environment; set it via the orchestrator's secret system.)

## Networking notes

- The control-plane listener (`7443`) is the only port that **must**
  be exposed for cross-host clients.
- The operator HTTP listener (`7080`) and metrics listener (`7081`)
  should remain on loopback inside the container; expose them via
  reverse proxy + auth or SSH tunnel as needed.
